# V0.8.2 Spec — GitHub Copilot（Copilot SDK + 本地 Copilot CLI）

**状态：可施工**  
**目标版本：V0.8.2**  
**前置版本：V0.8.1 已落地（受控 Plan / Candidate、Claude Code、Codex App Server、真实 provider probe）**

---

## 0. 结论

V0.8.2 增加一个用户可见的 Agent：**GitHub Copilot**。它支持两种本地 runtime，但不是两个重复的产品入口：

| 用户可见名称 | Host 内部 runtime | 何时使用 |
| --- | --- | --- |
| GitHub Copilot | `local_cli` | 用户本机的 `copilot` CLI 能被官方 Copilot SDK 成功连接、认证并通过兼容检查时，优先使用。 |
| GitHub Copilot | `bundled_sdk_cli` | 本地 CLI 缺失或不兼容时，使用 `@github/copilot-sdk` 管理的兼容 CLI。 |

**不要实现裸 `copilot -p` executor。**它会让 HTML Genius 再维护一套进程、认证、stdout、权限与版本兼容逻辑，且无法达到 SDK 的 session / tool hook 可控性。这里的“支持 Copilot CLI”指 SDK 以 `cliPath` 连接用户本机 CLI 的官方模式；SDK 已明确支持该模式，但本地 CLI 与 SDK 的版本兼容需由我们验证。[Copilot SDK local CLI setup](https://docs.github.com/en/copilot/how-tos/copilot-sdk/setup/local-cli)

不做 ACP、VS Code Companion Extension、Copilot Chat 注入、GitHub Cloud Session、MCP、插件或自定义 Agent。本版本始终创建新的 ephemeral session，永不读取、列举、恢复或向用户已有的 Copilot 会话写消息。

### 1.1 必须先修复的 V0.8.1 缺陷

`extension/background.js` 的 `completePlan()` 广播 `bridge-plan-ready` 时，当前没有携带 `plan_sha256`；但 `extension/sidepanel.js:onPlanReady()` 会读取它，随后确认计划时 `PlanValidate.validatePlanConfirmation()` 要求该值是合法 SHA。这会让任一 provider 的“确认计划，生成新版本”失败。

先修复为：

```js
broadcast({
  type: 'bridge-plan-ready',
  tab_id,
  run_id: runId,
  plan_id: planId,
  plan_sha256: planRec.plan_sha256,
  plan: { ... }
});
```

并增加回归测试：从合法 `plan-ready` 事件进入 Side Panel 后，确认请求携带与存储记录相同的 `plan_sha256`，可通过 `validatePlanConfirmation()`。

这不是 Copilot 特有修复，但若不先做，Copilot 的 Plan-first 流程必然无法闭环。

## 3. Provider 身份与数据模型

### 3.1 单一 provider ID

新增 provider：

```js
const COPILOT_PROVIDER = 'github_copilot';
```

所有浏览器侧 task、run、plan 使用 `provider: 'github_copilot'`。不要把 `copilot_sdk` 与 `copilot_cli` 伪装成两个 provider ID。

新增 Host-only runtime：

```js
const COPILOT_RUNTIMES = {
  LOCAL_CLI: 'local_cli',
  BUNDLED_SDK_CLI: 'bundled_sdk_cli'
};
```

`bridge_runs` 与 `bridge_plans` 增加可选字段：

```js
provider_runtime: 'local_cli' | 'bundled_sdk_cli' | null,
provider_runtime_version: '...', // 只用于本地审计；UI 最多显示截断版本
```

### 3.2 runtime 选择规则

Probe 的优先顺序固定：

1. 发现本机 `copilot` CLI，使用 SDK `cliPath` 模式启动、读取 auth status、做 protocol health check。成功则 `local_cli`。
2. 本地 CLI 缺失 / 不兼容 / 无法由 SDK 启动时，使用 SDK 默认 managed runtime。成功则 `bundled_sdk_cli`。
3. 两种都不能启动时，GitHub Copilot 返回不可用状态。

**不得在一个已启动的 plan/candidate run 内 fallback。**一个 runtime 启动后失败，就以该 runtime 的明确错误结束。自动 fallback 只允许发生在执行前的 probe / runtime 选择阶段，否则用户无法判断计划与 candidate 是否由同一环境生成。

Plan → Candidate 有额外规则：

- plan-ready 将实际 `provider_runtime` 持久化。
- Plan Review 确认 Candidate 时，Host 必须使用相同 runtime。
- 该 runtime 此时不再可用，返回 `COPILOT_RUNTIME_CHANGED`；不得悄悄切换到另一种 runtime。

## 4. 依赖、安装与兼容性

### 4.1 Bridge 依赖

在 `bridge/package.json` 添加生产依赖并**精确锁定**：

```json
{
  "dependencies": {
    "@github/copilot-sdk": "1.0.7"
  },
  "engines": {
    "node": "^20.19.0 || >=22.12.0"
  }
}
```

`1.0.7` 是本 Spec 编写时的 latest stable。施工时执行 `npm install` 并提交生成的 `bridge/package-lock.json`；不要用 `^1.0.7`。SDK 与其默认 CLI 有版本耦合，锁文件是可复现性边界。

### 4.2 Node / 安装器修复

当前 `bridge/install-macos.mjs` 强制要求找到 Claude CLI。这是错误的安装前提：用户应当可以只使用 Copilot 或 Codex。

修改要求：

1. Native Host 安装只校验 Node、host 文件、extension ID、manifest 写入权限。
2. Node 最低版本改为 SDK 要求的 `20.19.0`（或 `>=22.12.0`），不能只检查 major `>=20`。
3. `--claude-path` 保留为可选兼容参数，但不再是 install 成功条件。
4. launcher PATH 仍可加入常见 CLI 目录和用户显式的 Claude 目录；不写死/不需要 Copilot CLI 路径。
5. 安装完成文案改为 provider-neutral：`HTML Genius Local Bridge installed`。

### 4.3 本地 CLI discovery

新增 `bridge/copilot-runtime.mjs`，不得通过 shell 拼接或 `which` 输出直接执行未验证路径。

规则：

- 从受控 PATH 分段及常见 macOS bin 路径扫描 `copilot`；使用 `lstat`，拒绝 symlink（或记录 realpath 后再验证输入不为 symlink）、目录、非 executable 文件。
- `spawn(path, ['version'])`，`shell: false`，超时 10 秒，stdout / stderr 都限长。
- 仅将版本字符串（最多 64 chars）作为 probe metadata；CLI 绝对路径永不返回 Extension。
- SDK 用 `new CopilotClient({ cliPath })` 启动该二进制并查询 auth / server status；成功才可宣称 `local_cli ready`。

SDK local CLI 模式要求调用方承担版本兼容性责任，因此“CLI 存在”不等于“可用”。只有 SDK health check 成功，才标为 ready。[Copilot SDK local CLI setup](https://docs.github.com/en/copilot/how-tos/copilot-sdk/setup/local-cli)

## 5. Copilot SDK Adapter

新增：

```text
bridge/copilot-runtime.mjs       # runtime discovery / SDK client factory / probe
bridge/copilot-adapter.mjs       # candidate + plan 编排
bridge/test/copilot-runtime.test.mjs
bridge/test/copilot-adapter.test.mjs
bridge/test/fake-copilot-sdk.mjs
```

### 5.1 严格允许的 SDK 调用

每个 run 创建独立 client 与新 session，完成或失败时 `disconnect()` / `stop()`：

```text
client.start()
client.getAuthStatus()
client.createSession(...)
session.sendAndWait(...)
session.disconnect()
client.stop()
```

允许为了只读 probe 使用 `client.ping()` / `client.getStatus()`。

明确禁止调用：

```text
listSessions, getLastSessionId, getForegroundSessionId,
resumeSession, send(mode: immediate|enqueue), getEvents,
session.rpc.plan.*, session.rpc.workspace.*, registerTools,
customAgents, mcpServers, cloud sessions, remote CLI URL,
telemetry, transcript export, attachments
```

理由：HTML Genius 只需要“建立一份新的、受控的任务并验证文件输出”；其它 API 会扩大到会话读取、历史、外部工具、远端服务或用户数据。

### 5.2 Session 共同配置

每个 `createSession()` 必须：

- cwd 为本 run 的 `plans/<run-id>` 或 `runs/<run-id>`，不是原 HTML 目录。
- 不传 `mcpServers`、`customAgents`、attachment、cloud 或 remote URL。
- 关闭 / 排除 `task`、shell/Bash、network/url/web、MCP、plugin、extension、Git/GitHub、memory、terminal、Edit 等全部非必需工具。
- 只提供满足任务的最小读写工具；实际工具名称在 implementation Preflight 中用 SDK server status / test fixture 确认，不能凭模型猜测。
- 注册 `onPreToolUse`：对每次工具调用做路径及工具名 allow-list 检查；任何不匹配返回 `deny`，并记录不含路径/正文的安全审计事件。
- 不使用 `approveAll`。非许可工具必须拒绝而不是静默批准。
- 设置合理 timeout：Plan 3 分钟，Candidate 8 分钟；超时后终止 session/client 并发 `bridge_failed`。

SDK 提供 `availableTools` / `excludedTools` 与 `onPreToolUse`，后者可针对每次调用做 allow/deny 和参数校验。[Copilot SDK pre-tool hook](https://docs.github.com/en/copilot/how-tos/copilot-sdk/hooks/pre-tool-use)

**不能只依赖 hook。**工具 allow-list、隔离 workspace、snapshot 的只读权限、Host hash 校验、固定输出 validator 共同构成安全边界。尤其必须排除 `task` / subagent 工具，确保不存在子 Agent 绕过父 session hook 的路径。

### 5.3 Candidate executor

`executeCopilotCandidateRun(msg, { emit, sdkFactory })` 与现有 Claude / Codex candidate executor 行为一致：

1. 校验 `run_id`、source、task、`session.mode === 'new'`、mode、可选 `approved_plan`。
2. 使用现有 `createWorkspace()`、`writeTaskBundle()`、`prepareCandidateRun()`。
3. 将 approved plan 写成 candidate run 内只读 `approved-plan.md`（沿用 V0.8.1 的统一契约）。
4. 建立 Copilot client / new session，cwd = `runs/<run-id>`，发送 `buildCandidatePrompt()`。
5. 不采信模型 response 文本；执行后检查真实 source hash、验证唯一 `candidate.html`、发布 sibling candidate、写现有 candidate manifest。
6. 发标准 `candidate-ready`，字段必须与 Claude / Codex 相同，额外可带：

```js
provider: 'github_copilot',
provider_runtime: 'local_cli' | 'bundled_sdk_cli'
```

7. 任何失败都不得创建 sibling candidate 或触发 artifact update。

Candidate 只允许：读取 `source.html` 与 task bundle；写 `candidate.html`。即使 Copilot 的模型网络请求本身需要访问 GitHub 服务，也不能给 Agent `url` / web 工具。

### 5.4 Plan executor

`executeCopilotPlanRun(msg, { emit, sdkFactory })`：

1. 使用现有 `preparePlanRun()`，cwd = `plans/<run-id>`。
2. 发送现有 `buildPlanPrompt()`；不得使用 Copilot SDK 的 `plan.md` API 或内建 `/plan`。
3. 只允许写 `output/plan.json`。
4. 运行后依次检查真实 source hash、task JSON hash、`validatePlanJson()`。
5. 写 `htmlgenius_plan_manifest`；失败 manifest 不含计划正文。
6. 发标准 `plan-ready`：

```js
{
  type: 'plan-ready',
  run_id,
  provider: 'github_copilot',
  provider_runtime,
  task_sha256,
  logical_document_id,
  source_uri,
  source_sha256_before,
  plan_sha256,
  plan,
  manifest_path
}
```

无论模型 response 里写了什么，只有 `output/plan.json` 通过 validator 后才可见。

### 5.5 错误码

新增并统一映射：

| 错误码 | UI 含义 |
| --- | --- |
| `COPILOT_SDK_NOT_INSTALLED` | GitHub Copilot runtime 未安装或 Bridge 依赖缺失。 |
| `COPILOT_CLI_NOT_FOUND` | 本地 CLI 模式不可用；若 SDK runtime 可用会在 probe 时自动转 SDK。 |
| `COPILOT_INCOMPATIBLE` | 本地 Copilot CLI 无法与锁定 SDK 建立兼容连接。 |
| `COPILOT_AUTH_REQUIRED` | 请先在本机完成 GitHub Copilot 登录。 |
| `COPILOT_RUNTIME_CHANGED` | 计划对应的 Copilot runtime 已不可用，请重新生成计划。 |
| `COPILOT_PERMISSION_DENIED` | Agent 尝试使用不在任务范围内的工具或路径。 |
| `COPILOT_PLAN_FAILED` / `COPILOT_PLAN_TIMEOUT` | 修改计划生成失败 / 超时。 |
| `COPILOT_RUN_FAILED` / `COPILOT_TIMEOUT` | 候选版本生成失败 / 超时。 |

错误消息对 UI 最长 400 字符；不得泄露路径、CLI 完整 stderr、token、账号或 session ID。

## 7. 安全与隐私验收边界

1. 原 HTML 永远不传真实路径给 Agent；只传受控 workspace snapshot。
2. 原 HTML、task JSON、task Markdown 只读；Host 前后 hash 校验。
3. Candidate 只认可唯一 `candidate.html`；Plan 只认可唯一 `output/plan.json`。
4. 不允许 shell、web/url、MCP、plugins、extensions、custom agents、subagents、workspace RPC、attachments、telemetry、session transcript。
5. 不读取或持久化 Copilot session ID；`bridge_sessions` 不为 Copilot 写记录。
6. 计划确认后新建 session，不 resume plan session；用户在 Copilot CLI、VS Code、GitHub Copilot App 中已有会话完全不接触。
7. 不发送用户 token；认证只使用本机 Copilot SDK / CLI 已登录态。SDK 文档说明本地模式使用系统 credential store 或用户自行配置的环境 token；HTML Genius 不读取或展示这些值。[Copilot SDK local CLI setup](https://docs.github.com/en/copilot/how-tos/copilot-sdk/setup/local-cli)

## 9. 自动化测试

### Runtime / probe

- SDK managed runtime ready / auth_required / missing 分类。
- 本地 CLI valid + SDK client health check → `local_cli`。
- 本地 CLI 存在但 SDK protocol incompatible → 尝试 bundled SDK；bundled 成功则 ready，失败才 incompatible。
- 仅 Copilot probe 失败不影响 Claude / Codex；反之亦然。
- probe 不调用 `createSession`、`send`、`listSessions`。
- probe 输出不含 CLI 路径、token、stderr。

### Copilot Plan

- fake SDK 写合法 `output/plan.json` → 标准 `plan-ready`，带 `provider_runtime`。
- 缺 plan、坏 JSON、未知字段、symlink、超长、source/task 运行时变更 → 普通失败且无 `plan-ready`。
- 计划 run 不创建 `candidate.html` 或 sibling candidate。
- 任何 shell/url/MCP/task/subagent / 越权 write tool call 被 pre-tool policy deny。

### Copilot Candidate

- fake SDK 写合法 `candidate.html` → 复用现有 `candidate-ready` / artifact consumer。
- source hash 改变、candidate 不存在 / symlink / 非 HTML / 超限 → 失败且无 sibling 输出。
- approved plan 写入只读文件；无 approved plan 的直接 candidate 回归通过。
- 对有 plan 的 candidate，Host runtime 与 `required_provider_runtime` 不同 → `COPILOT_RUNTIME_CHANGED`。
- 不调用 resume / list sessions / get history。

### Extension

- GitHub Copilot 可被 sanitize、显示、选择、派发。
- 未 ready 时不可 Plan / Candidate，但复制 Prompt 可用。
- `bridge-plan-ready` 带 `plan_sha256`，确认计划可以发出。
- Copilot plan → candidate 的 `provider_runtime` 传递与不匹配拒绝。
- 旧 Claude / Codex plan、candidate、probe 测试不回归。

## 10. 手动验收

在一台具有 GitHub Copilot 权益的 macOS 机器上：

1. 无本地 Copilot CLI：Bridge 使用 SDK managed runtime，菜单显示 GitHub Copilot ready。
2. 安装兼容的本地 CLI：重开 Side Panel / 30 秒后 probe，菜单改显示“本地 Copilot CLI”。
3. 未登录时菜单显示“需要在本机登录”，不允许 Plan / Candidate，复制 Prompt 仍可用。
4. 用 2 条根评论和 1 条回复请求 Plan：只出现 `plan.json` 与 plan manifest，不出现 candidate sibling。
5. 编辑计划后确认：创建新的 Copilot task、生成受控 candidate；原文件 hash 不变。
6. 在 Plan Review 后让本地 CLI 不可用：确认必须失败 `COPILOT_RUNTIME_CHANGED`，不能静默切 SDK runtime。
7. 切换 Claude、Codex、GitHub Copilot，确认原评论与所有 V0.8.1 UI 不变，无横向滚动。

## 11. 完成定义

V0.8.2 完成当且仅当：

- 用户可以在现有 Agent 菜单选择一个 GitHub Copilot，而非面对两个重复入口。
- 本地 Copilot CLI 与 SDK managed runtime 都能通过官方 SDK 接入，且 runtime 选择真实可见。
- Copilot 支持与现有 provider 等价的受控 Plan 与 Candidate；不依赖模型 stdout。
- 计划与 candidate runtime 一致，新建 session，不读取用户已有 session。
- 现有 Plan-first 的 `plan_sha256` 闭环缺陷已修复。
- 安全边界、原文件不覆盖、已有 Claude / Codex 行为和 V0.8.1 UI 均无回归。
