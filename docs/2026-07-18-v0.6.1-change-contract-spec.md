# v0.6.1 修改契约（Change Contract）实现 Spec

> 日期：2026-07-18  
> 前置版本：v0.6 已完成元素级直接编辑；v0.6.1 **不实现 bridge / Agent 通信**。  
> 目标：把当前「一键复制所有评论」升级为带明确修改范围、保护规则与验收条件的任务生成器，使同一任务可安全交给 glm5.2、Codex、Claude Code 或 Copilot 执行。

## 0. 给实现 Agent 的执行约束

这是一次 **局部功能新增**，只修改“批注 → 复制任务”的出口。不要重构存储、协同、元素编辑、锚点算法或现有批注 UI。

允许改动：

- `extension/sidepanel.html`
- `extension/sidepanel.css`
- `extension/sidepanel.js`
- `extension/i18n.js`
- 新增 `extension/change-contract.js`
- 新增 `extension/change-contract-test.html`
- `extension/content-script.js` 中 `get-export` 的响应元数据
- `extension/manifest.json`（仅在新增 script 后需要加载时）
- 本文档中列出的必要测试

禁止改动：

- `storage.js` 的 IndexedDB schema；v0.6.1 不保存 task、不保存 preset。
- 远端协同 API、登录、权限模型。
- 批注创建/回复/编辑/删除语义。
- v0.6 元素模式、直接编辑、撤销/重做、文件保存。
- 自动调用任何 Agent、API、MCP、Native Messaging 或本地 bridge。

## 1. 问题与成功定义

当前 `BuildPrompt.fromAnnotations(items)` 把全部批注拼成一段固定 prompt。它没有表达：

- 这次允许改多大范围；
- 未圈选内容是否受保护；
- 找不到定位目标时应如何处理；
- 用户是在要求局部修补、结构规划还是全篇重写。

因此 Agent 容易“顺手修改”未被要求的内容。v0.6.1 的成功不是 prompt 写得更长，而是用户在复制前明确授予修改权限。

### 成功标准

1. 用户可从一条顶层批注或全部未失效顶层批注发起任务。
2. 用户必须在四种修改模式中选择一种；默认 **精准修补**。
3. 复制的 prompt 明确写出允许范围、保护项、歧义处理和验收条件。
4. 同一任务可复制为人可读 prompt 或机器可读 JSON。
5. `结构重组` 与 `重新生成` 在缺少“补充说明”时不能复制。
6. 当前“一键复制所有评论”的旧固定 prompt 不再作为 UI 主出口，也不保留隐式全量修改语义。

## 2. 范围与非目标

### 本版本纳入

- 批注任务生成器（以下简称 Composer）。
- 四种修改契约。
- 单条顶层批注和全部未失效顶层批注两种任务来源。
- Prompt / JSON 双复制。
- 纯函数任务构建、验证和 prompt 渲染，以及浏览器测试页。
- 中 / 英 / 日 i18n。

### 明确不做

- 多选任意批注、任务历史、preset 保存、任务状态。
- Agent bridge、自动执行、API 调用、文件回写、diff 展示。
- 自动推断“最近 section”或生成 DOM 级可写范围；v0.6.1 只传精确锚点和语言约束。
- 让用户编辑原始 JSON。
- 改变现有 `BuildPrompt.fromAnnotations()` 的对外兼容行为；可保留供旧测试/内部兼容，但新 UI 必须使用 `ChangeContract`。

## 3. 用户入口与交互

### 3.1 入口

1. 每个**未失效的顶层批注卡片**动作区新增次级按钮：`生成任务`。
   - 点击后，Composer 只包含该顶层批注及其完整回复树。
   - 回复卡片不出现该按钮，避免用户只导出一段失去上下文的回复。
2. 批注列表底部现有 `#export-btn` 改为 `生成修改任务`。
   - 点击后，Composer 包含所有未失效顶层批注及各自回复树。
3. 没有可用顶层批注时，底部按钮 disabled；点击行为不得打开空 Composer。

### 3.2 Composer 形态

在 Side Panel 内显示一个覆盖式 sheet（不是浏览器 `prompt()` / `confirm()`）。打开时：

- 不离开批注 tab；背景内容仍可见但不可操作。
- 标题：`生成修改任务`。
- 显示来源摘要：`当前任务包含 1 条批注` 或 `当前任务包含 3 条批注`。
- 显示每条顶层批注的短引用（最多 48 字）和回复数量，仅供确认，不允许在此编辑批注。
- 顶部有关闭按钮；关闭只丢弃 Composer 临时状态，不修改批注。

### 3.3 修改模式

四个 radio card 必须互斥，默认选中 `精准修补`。每张 card 显示名称和一句约束说明：

| `mode` | 中文名称 | 描述 |
|---|---|---|
| `precise_patch` | 精准修补 | 只改被批注精确定位到的内容；其他内容完全不动。 |
| `local_optimize` | 局部优化 | 只优化这些批注所在的局部内容；页面其他部分不动。 |
| `restructure` | 结构重组 | 重新规划章节、顺序或局部结构；先产出计划，不直接改文件。 |
| `regenerate` | 重新生成 | 基于新的完整说明重写整份页面/报告；允许改全文。 |

### 3.4 附加字段与校验

所有模式都有：

- `补充说明` textarea，optional，最大 2,000 字。
- `必须保留` textarea，optional，最大 2,000 字；按换行拆成字符串数组，空行忽略。

`restructure` 与 `regenerate` 额外规则：

- `补充说明` required，trim 后至少 10 个字符。
- 不满足时，两个复制按钮 disabled，并在字段下展示错误提示。
- `restructure` 的按钮文案为 `复制规划任务`，提示中明确“先给计划，等待用户确认后再修改”。

`precise_patch` 与 `local_optimize`：

- `补充说明` optional；批注内容本身已经是修改要求。
- `必须保留` 会追加到默认保护规则，不能覆盖默认保护规则。

### 3.5 输出按钮

- Primary：`复制 Prompt`（`restructure` 显示 `复制规划任务`）。
- Secondary：`复制 JSON`。
- 成功后按钮文本短暂变为 `已复制 ✓`，沿用现有 toast 机制。
- Clipboard 拒绝时展示 `复制失败，请手动选择文本`；同时在 Composer 中展示只读 `<textarea>`，内含对应输出，供用户手动复制。
- Composer 不自动关闭，用户可以复制另一种格式或调整契约。

## 4. 数据模型

新增纯前端模块 `extension/change-contract.js`，暴露全局 `window.ChangeContract`。模块不得读取 DOM、chrome API、时间、随机数或 Clipboard，方便稳定测试。

### 4.1 输入 draft

```js
{
  mode: "precise_patch", // precise_patch | local_optimize | restructure | regenerate
  rootIds: ["ann_1"],    // 必须是顶层、非 stale 批注 id；由 UI 提供
  brief: "",             // 用户补充说明
  preserveText: "",      // textarea 原文本，按换行拆分
  artifact: {
    title: "页面标题",
    url: "file:///.../report.html", // 或 https URL
    isLocal: true
  }
}
```

### 4.2 输出 task（JSON schema v1）

```jsonc
{
  "schema_version": 1,
  "kind": "htmlgenius_change_contract",
  "mode": "precise_patch",
  "artifact": {
    "title": "页面标题",
    "url": "file:///.../report.html",
    "is_local": true
  },
  "source": {
    "root_annotation_ids": ["ann_1"],
    "root_annotation_count": 1
  },
  "annotations": [
    {
      "id": "ann_1",
      "selector": { "type": "TextQuoteSelector", "exact": "...", "prefix": "...", "suffix": "..." },
      "quote": "...",
      "comment": "...",
      "author": "张三",
      "replies": [
        { "id": "ann_2", "comment": "...", "author": "李四", "replies": [] }
      ]
    }
  ],
  "brief": "",
  "preserve": [],
  "contract": {
    "write_scope": "target_only",
    "locked_outside_scope": true,
    "on_ambiguous_target": "ask_or_stop",
    "verification": ["..."]
  }
}
```

### 4.3 mode → contract 映射（固定，不让 UI 改写）

| mode | `write_scope` | `locked_outside_scope` | `on_ambiguous_target` | 固定 verification |
|---|---|---:|---|---|
| `precise_patch` | `target_only` | `true` | `ask_or_stop` | 仅修改定位目标；报告前后片段或 diff；不改其他文件。 |
| `local_optimize` | `annotated_local_areas` | `true` | `ask_or_stop` | 仅改批注所在局部；报告每个局部的变更；不改其他页面区域。 |
| `restructure` | `document_structure` | `false` | `ask_or_stop` | 先输出结构计划、保留项和影响范围；**不得在本轮修改文件**。 |
| `regenerate` | `whole_document` | `false` | `ask_or_stop` | 列出采用的 brief、保留项、假设和待复核项；报告全文结构变化。 |

所有 mode 都必须传入用户 `preserve`，并以追加规则处理；不得用用户自由文本删除固定保护规则。

### 4.4 回复树规则

`annotations` 仅包含被选中的 root annotation 及其完整后代，顺序保持原始 items 顺序（DFS，父节点在子节点之前）。

- stale root 不得进入任务。
- 若 `rootIds` 中不存在、是 reply、或 stale，则 `validateDraft()` 返回错误。
- 遇到孤儿 reply 或循环引用时，不抛异常：保留可达节点，跳过重复节点。

## 5. 模块 API

```js
window.ChangeContract = {
  MODES,                                      // readonly array / metadata
  validateDraft(draft, allAnnotations),       // { ok: boolean, errors: { rootIds?, brief?, mode? } }
  buildTask(draft, allAnnotations),           // validation failed 时 throw Error("Invalid change contract")
  renderPrompt(task),                         // string
  serialize(task),                            // JSON.stringify(task, null, 2)
  getRoots(allAnnotations),                   // 非 stale 的顶层 annotation
  buildReplyTree(allAnnotations, rootIds)     // pure helper，供测试
};
```

`ChangeContract` 不替代 `BuildPrompt`。旧 `BuildPrompt.fromAnnotations()` 保持原样，以避免无关回归；新 UI 只能调用 `ChangeContract`。

## 6. Prompt 渲染规范

输出使用中文、模型中立的指令，不出现“你是 HTML Genius 自带 Agent”之类的角色绑定。固定结构如下：

```text
# HTML Genius 修改任务

## 任务模式
精准修补（只允许修改精确定位的目标）

## 文档
- 标题：...
- 地址：...
- 本地 HTML：是/否

## 修改契约
- 允许范围：...
- 受保护内容：...
- 目标不明确时：停止修改并说明需要确认的位置。

## 批注与讨论
### 批注 1
- 定位：前文「...」｜原文「...」｜后文「...」
- 评论：...
- 回复：...

## 补充说明
...

## 验收与输出
1. ...
```

额外强制语句：

- `precise_patch`：`禁止修改任何未被本任务定位的文本、DOM 结构、样式或其他文件。若目标出现多次且无法凭定位唯一判断，停止并请求确认。`
- `local_optimize`：`只能在每条批注定位到的局部内容中修改；不要扩展到页面其他区块。`
- `restructure`：`此轮仅输出可执行的重组计划、涉及范围和风险；不要修改文件，也不要输出重写后的完整 HTML。`
- `regenerate`：`允许重写全文，但必须保留“必须保留”列出的内容，并列出未能确认的假设。`

不要把完整 HTML 源码塞入 v0.6.1 prompt：现有复制流程没有可靠的文件/工作区上下文，且这会无谓增加 token。后续 v0.7 bridge 再负责将本地文件路径和工作区提供给 Agent。

## 7. 现有文件的具体改动

### 7.1 `sidepanel.html`

1. 在 `buildprompt.js` 后、`sidepanel.js` 前加载 `change-contract.js`。
2. 将 `#export-btn` 初始文本改为 `生成修改任务`（i18n）。
3. 新增 `#contract-sheet`，初始 `hidden`，包含：
   - 标题、关闭按钮、来源摘要 `#contract-source-summary`；
   - `#contract-source-list`；
   - 四个 `input[type=radio][name=contract-mode]`；
   - `#contract-brief` textarea + `#contract-brief-error`；
   - `#contract-preserve` textarea；
   - `#contract-copy-prompt`、`#contract-copy-json`、`#contract-output-fallback`。
4. sheet 必须有 `role="dialog"`、`aria-modal="true"`、可读 label；关闭按钮有 i18n `aria-label`。

### 7.2 `sidepanel.js`

新增临时状态（不持久化）：

```js
let _contractSourceRootIds = [];
let _contractArtifact = null;
let _contractOpen = false;
```

实现函数：

- `openContract(rootIds, artifact)`：从 `_lastItems` 取 roots，初始化 mode 为 `precise_patch`，清空 brief/preserve/error，渲染 Composer。
- `closeContract()`：只清临时状态、恢复焦点到触发按钮。
- `getContractDraft()`：收集 UI 值。
- `validateContractUI()`：调用 `ChangeContract.validateDraft()`，更新错误和复制按钮 disabled 状态。
- `copyContract(kind)`：`buildTask()` → `renderPrompt()` 或 `serialize()` → Clipboard；失败时显示 fallback textarea。
- `countReplies(root, allItems)`：仅用于来源摘要。

`renderCards()`：仅顶层非 stale card 新增 `生成任务` 按钮，点击时 `stopPropagation()`，然后使用当前页面 artifact 打开 Composer。

`#export-btn`：先通过现有 `get-export` 请求获取 `items + artifact`，过滤有效 roots；无 roots 时沿用空提示；否则 `openContract(rootIds, artifact)`。

不改变卡片点击滚动、回复、编辑、删除的行为。

### 7.3 `content-script.js`

仅扩展 `get-export` 的响应：

```js
{
  type: "export-data",
  items: window.__hgAnnotations || [],
  artifact: {
    title: document.title || "Untitled HTML",
    url: location.href,
    isLocal: isLocal
  }
}
```

不要在此消息中读取文件内容、绝对本地路径、storage schema 或任何用户敏感额外数据。

### 7.4 `sidepanel.css`

- Composer 覆盖式 sheet 固定在 Side Panel 内，内容过长可滚动；z-index 高于 account/lang sheet 与 toast。
- 四个模式是可点击 card；选中态使用现有 `--hg-brand` / `--brand-soft` token，不引入新主题体系。
- `restructure` / `regenerate` 的必填提示使用现有 danger/warning 色 token。
- primary/secondary/disabled 样式沿用现有按钮语言；移动窄侧栏不溢出。

### 7.5 `i18n.js`

所有新文字提供 `zh`、`en`、`ja` 三份翻译。不得只把中文硬编码进 `sidepanel.js` 或 HTML。

建议 key 前缀：`contract.*`，例如：

```text
contract.open, contract.title, contract.close, contract.source,
contract.mode.precise, contract.mode.local, contract.mode.restructure,
contract.mode.regenerate, contract.brief, contract.briefRequired,
contract.preserve, contract.copyPrompt, contract.copyPlan,
contract.copyJson, contract.copied, contract.copyFail, contract.empty
```

## 8. 测试与验收

### 8.1 新增纯函数浏览器测试

新增 `extension/change-contract-test.html`，打开后 title 为 `PASS` 或 `FAIL: <reason>`。至少覆盖：

1. `precise_patch` 任务包含一个 root、其多级 replies、精确 selector 和强制“目标外不修改”语句。
2. `local_optimize` prompt 只承诺局部范围。
3. `restructure` 与 `regenerate` 在 brief 少于 10 字符时验证失败；达到阈值后成功。
4. preserve textarea 被正确拆分、trim、去空行，并保留在 task 与 prompt 中。
5. stale root、reply id、不存在 id 不能作为 `rootIds`。
6. 两个 root 的输出顺序与输入 annotations 顺序一致；每个 root 包含完整 DFS 回复树。
7. 无效 mode 返回验证错误；`buildTask()` 不产生静默错误 task。
8. `serialize()` 可 `JSON.parse()`，并包含 `schema_version: 1`。

### 8.2 手动验收

1. 创建一条顶层批注和两层回复；卡片点击“生成任务”，默认精准修补，复制 prompt 后确认只包含该线程。
2. 点击底部“生成修改任务”；确认包含所有未失效顶层批注，且不包含 stale 区域。
3. `精准修补` prompt 明确禁止改未选中内容；复制 JSON 与 prompt 的 mode / root ids 一致。
4. 切到 `结构重组` 或 `重新生成`，不填 10 字以上补充说明时不可复制；填入后可复制。
5. 切换 zh/en/ja，所有 Composer 文案均切换，页面不出现 key 或中文硬编码。
6. 关闭 Composer、刷新批注、编辑/删除批注、切换编辑 tab，现有行为无回归。
7. 无批注时底部按钮 disabled，且不打开空任务窗口。

### 8.3 最低验证命令

```bash
node --check extension/change-contract.js
node --check extension/sidepanel.js
node --check extension/content-script.js
node --check extension/i18n.js
```

并在 Chrome 加载 `extension/` 后手测上述流程；如已有项目测试命令可运行，必须一并运行并报告结果。

## 10. 完成定义

v0.6.1 完成时，用户无需理解 JSON 或 prompt engineering，也能在复制前做出一个显式决定：

> 这次只允许 Agent 改被圈选的内容、一个局部区块、文档结构，还是整份文档？

HTML Genius 需要把这个决定不可歧义地传递给任何外部 Agent。它不负责执行 Agent，也不假装能保证模型一定遵守；它负责让用户的授权范围可见、可复制、可审计。
