# V0.8.1 Spec — 修改范围、可选评论范围与 Plan Review

**状态：可施工**  
**目标版本：V0.8.1**  
**前置版本：V0.8 已完成（Claude Code / Codex App Server 均可创建受控 candidate）**  
**施工方式：一个 Agent 可按本文连续实现；先完成 Preflight，再按里程碑提交。**

---

## 0. 结论与边界

V0.8.1 把“创建编辑任务”改成一个更短、更安全的主流程：

1. 用户从原有评论收件箱点击 **创建编辑任务**。
2. 默认带上全部非失效评论，直接选择 **精确 / 局部 / 全文** 的修改范围。
3. 用户可选地展开“高级选项”，补充目标、保留内容，或进入“选择评论范围”缩小本次输入。
4. 用户可复制 Prompt、直接让已连接 Agent 生成 `candidate.html`，或先让 Agent 输出可编辑的修改计划。
5. Plan Review 中用户确认计划后，**另起一个新的 Agent task** 生成 candidate；绝不续发计划 task，也不接管用户正在运行的任何外部会话。

这不是只改前端。`先给我看修改计划` 必须是一个独立的、受控的 Bridge `plan` run：Agent 只能写入受限的 `plan.json`，Host 严格校验后才把计划交给扩展。**不得从 Agent stdout 猜测计划文本，更不得把计划输出当成 candidate。**

### 本版本不做

- 不读取、列举、选择、注入用户已有的 Codex / Claude Code 会话。
- 不提供自有 AI 模型、云端对话或 Token 计费。
- 不自动区分评论是“修改建议”还是“澄清问题”。这是用户的选择，不是本版本的 AI 分类任务。
- 不做 candidate diff、采纳/覆盖原文件、回写后重锚定；沿用 V0.8 的 candidate 产出与打开流程。
- 不删除或重做评论卡既有交互（定位、回复、编辑、删除等）。
- 不暴露“继续上一个 task”“复制 JSON 导出”等 UI；本版本的每次派发均为新 task。

6. 不得把 Agent “已连接”画成静态绿点。状态必须来自真实的本机 probe；未就绪的 Agent 不可派发。

## 2. 用户可见流程

```text
评论收件箱（维持原功能）
  └─ 创建编辑任务
       └─ 选择修改范围（默认：精确；默认：全部评论已选择）
            ├─ 复制 Prompt
            ├─ 发送给 <已连接 Agent> → 新 candidate task
            ├─ 先给我看修改计划 → 新 plan task
            └─ 高级选项
                 └─ 选择评论范围（可选）
                      └─ 返回选择修改范围

Plan Review
  ├─ 编辑计划文本 / 返回调整任务条件
  └─ 确认计划，生成新版本 → 新 candidate task
```

### 2.1 状态机

`sidepanel` 内存态仅用于本轮草稿，关闭 Sheet 或 Esc 后清空，不能把评论正文、Prompt、计划草稿写入 `chrome.storage`。

```text
inbox
  -> compose                    点击创建编辑任务
compose
  -> comment-scope              高级选项 > 选择评论范围
comment-scope
  -> compose                    确认选择 / 上一步
compose
  -> plan-running -> plan-review 先给我看修改计划成功
compose | plan-review
  -> candidate-running -> candidate-result 生成新版本成功
any workflow state
  -> inbox                      关闭 / Esc / 上一步返回至收件箱
```

执行中的 run 不允许切换界面后并发启动第二个 run；仍使用当前 tab 的 `bridge_runs` 活跃锁。

## 3. UI 详细规格

### A. 评论收件箱（既有页）

- 评论列表在上，保持原来的上下级结构与卡片控制项。
- 底部固定动作区保留一个主入口：`创建编辑任务`。
- 删除旧的长注释文案（如“整理评论、创建编辑任务”的解释性副标题）。
- 点击入口时：向 content script 重新请求 `get-export`；获取最新评论与 artifact state 成功后才打开 B。没有有效评论时沿用现有 toast，不打开空任务。
- 默认选择：所有 **non-stale root + reply** 节点。根节点与其回复都必须进入选择集合；Change Contract 的 root IDs 仍由选中的 root 推导。

### B. 选择修改范围（主编辑任务页）

#### 顶栏

- 固定置顶：左侧 `← 上一步`，点击回到评论收件箱；右侧可保留关闭按钮 `×`。
- 标题：`选择修改范围`
- 不再显示“创建编辑任务”“本次将发送 N 条评论 / 更改”这类来源摘要模块。

#### 三个单选项

标题：`选择修改范围`

默认选中第一项；卡片整块可点击，键盘可操作。

| 内部 mode | 显示标题 | 给普通用户的一句话 |
| --- | --- | --- |
| `precise_patch` | `精确：只改圈选处` | 只修改评论指向的内容，其他地方保持不动。 |
| `local_optimize` | `局部：优化这一区块` | 可以连同附近内容一起调整，让这一块更完整。 |
| `regenerate` | `全文：重做整页` | 按这些要求重做整页，页面其他部分也可能改变。 |

不再在这里放“先给我改法 / 直接生成新版本”并列选项；计划是一个独立动作。

#### 高级选项（默认折叠、全部非必填）

`<details>` 或等价无障碍 disclosure，标题为 `高级选项（可选）`。展开后依序显示：

1. `你想让页面变成什么样？` textarea  
   帮助：`补充评论没有说清的目标、受众或整体风格。`
2. `有什么绝对不能动的内容？` textarea  
   帮助：`即使允许大改，也必须原样保留的内容。`
3. 选择评论范围模块：
   - 主行/按钮：`选择评论范围（已选 {selected} / 共 {total} 条）`
   - 注释：`有时不用把全部评论发给 Agent。`
   - 点击进入 C。

两个文本框均允许为空；其内容长度沿用 Change Contract 的 2,000 字符上限。不得再把 brief 设成必填，也不得因为 `全文` 模式强制 brief。

#### 固定底部动作区

由上到下 / 在窄宽中可换行，但不得横向溢出：

1. 次级按钮：`复制 Prompt`
2. split button：`发送给 {当前已选 Agent}` + 下拉箭头
3. 与以上并列的次级动作：`先给我看修改计划`

视觉以 Demo 为准。运行时两个 Agent 动作均显示 loading / disabled，复制按钮也禁用，直到 terminal event。

### C. 选择评论范围（仅高级选项入口）

- 固定顶栏左侧：`← 上一步`，回 B 并保留 B 的所有草稿值。
- 标题：`选择评论范围`。
- 展示计数：`共 {total} 条评论，已选择 {selected} 条`。
- 顶部有 `全选` / `取消全选` 切换；默认是全选。
- 总评论数大于 20 时展示非阻断提示：`评论超过 20 条时，不建议逐条手选；可按主题分批创建编辑任务。`
- 列表必须保留 root → reply 的层级：回复卡左侧缩进、连接线/边框，文本与原评论卡一致。不能把回复平铺成根节点。
- 勾选根节点时，默认连同所有子回复一起切换；取消根节点同样取消子回复。单独勾选回复允许，且必须计入“已选”；但任务的 `root_annotation_ids` 只包含实际被选中的根节点。
- 底部确认按钮文案实时变化：`确认选择（已选 {selected} / 共 {total} 条）`。选中为 0 时禁用。
- 确认后直接回 B；不新增“展示态”，不再有“继续 / 更改”的中间页面。

### D. Agent 下拉菜单（真实状态）

下拉只负责换 Agent，不负责选择 session，不提供 JSON 导出。

支持项：

- `Claude Code`
- `Codex`

状态由 `bridge-query-providers` 返回的真实 probe 结果驱动：

| probe 状态 | 菜单表现 | 可否选择 |
| --- | --- | --- |
| `ready` | `已连接`，可显示安全的版本摘要 | 是 |
| `checking` | `正在检查…` | 否 |
| `not_installed` / `not_found` | `未安装或未发现` | 否 |
| `auth_required` | `需要在本机登录` | 否 |
| `incompatible` / `untrusted` | `当前版本不兼容` | 否 |
| `error` | `暂不可用` | 否 |

规则：

- 打开 Sheet 时 probe；缓存最多 30 秒。用户点开下拉时若缓存过期再 probe。
- 恢复上次用户选择的 provider，仅当它仍为 `ready`；否则自动选第一个 `ready` 项。
- 一个都未就绪时，发送与计划按钮禁用，显示明确但简短的状态；`复制 Prompt` 仍可用。
- 不向 UI 返回 runtime 绝对路径、Team ID、schema 文件路径、完整 stderr 或认证信息。

### E. Plan Review

点击 `先给我看修改计划` 后：

1. 先显示与 candidate 相同等级的运行态：`正在让 {Agent} 制定修改计划…`。
2. 收到已校验的 `plan-ready`，进入 Plan Review，不生成或打开 HTML 文件。
3. 顶栏左侧 `← 上一步` 回 B；计划保留为未采纳草稿，B 中任务条件也保留。
4. 标题：`确认修改计划`
5. 显示一段说明：`计划只用于确认修改范围；确认后会新建一个任务生成候选版本。`
6. 提供一个可编辑的 `textarea`，标题 `修改计划`。内容是受控 `plan.json` 的 `plan_markdown`，不是 Agent 原始 stdout。最大 12,000 UTF-8 bytes。
7. 底部动作：
   - 次级：`返回调整任务条件`（回 B，不丢 B 草稿）
   - 次级：`重新生成计划`（新建新的 plan task；旧计划标记为 `rejected`）
   - 主按钮：`确认计划，生成新版本`
8. 主按钮需要非空计划文本；执行时以**当前编辑后的计划文本**和原 Change Contract 一起发送，创建新的 candidate task。绝不 resume plan task。
9. 用户从 B 改变修改范围、勾选评论、brief、preserve 或当前 artifact 后，已有计划标记 stale，不允许确认；要求重新生成。

Plan Review 不展示模型思维链、原始工具调用、stdout、token、外部 session ID 或本机文件路径。

## 4. 前端状态与 Change Contract

### 4.1 草稿模型

替换当前 `_contractStep = closed | select | compose` 的主语义，至少包含：

```js
{
  step: 'closed' | 'compose' | 'comment-scope' | 'plan-running' |
        'plan-review' | 'candidate-running' | 'candidate-result',
  items: [],                 // 每次打开 / 发送前从 get-export 刷新
  artifact: null,
  bridgeMeta: null,
  selectedNodeIds: new Set(),
  mode: 'precise_patch' | 'local_optimize' | 'regenerate',
  brief: '',
  preserveText: '',
  provider: null,
  providerStates: {},
  plan: null,                // 仅已校验、可展示的 plan record
  runningRunId: null
}
```

关闭任务 Sheet 时清空此对象。用户编辑 plan textarea 只更新内存中的 `plan.edited_plan_markdown`，直至点击确认时才通过消息发送给 background。

### 4.2 任务构建

继续使用 `ChangeContract.buildTask()`，但该 UI 路径只允许：

```js
'precise_patch' | 'local_optimize' | 'regenerate'
```

禁止通过执行选项把 mode 派生成 `restructure`。旧 `restructure` 数据可为了旧代码兼容而保留在底层，但不得出现在 V0.8.1 UI 及其新请求中。

每次复制、计划、candidate 派发前都必须重新 `get-export` 并验证：

- 当前 artifact URL 与草稿 artifact 一致；
- 所选节点仍存在、未 stale；
- 至少一个 root 被选中；
- 根 / 回复层级和稳定排序使用最新 item 数据。

若刷新后选中节点发生变化，更新计数并提示用户；若所有根失效，阻止发送。

### 4.3 Prompt 复制

`复制 Prompt` 不需要 provider ready。它复制 `ChangeContract.renderPrompt(task)`，其中必须反映当前的三档 mode、已选评论、brief、preserve。复制失败使用已有 toast；不得额外复制 JSON。

## 5. Background 消息与存储协议

### 5.1 Provider probe

新增 Extension → Background：

```js
{ type: 'bridge-query-providers' }
```

响应：

```js
{
  ok: true,
  providers: [
    { id: 'claude_code_cli', label: 'Claude Code', status: 'ready', version: '…', capabilities: ['candidate', 'plan'] },
    { id: 'codex_app_server', label: 'Codex', status: 'auth_required', capabilities: [] }
  ],
  checked_at: 'ISO-8601'
}
```

Background 通过 Native Host 请求 probe；缓存成功与失败结果 30 秒。provider probe 是只读检查，不能创建 session、thread 或 workspace candidate。

Native message 建议：

```js
{ type: 'provider_probe', providers: ['claude_code_cli', 'codex_app_server'] }
```

Host 返回一次 `provider_probe_result`，每个 provider 独立失败，不能因为 Codex 缺失而让 Claude probe 整体失败。

### 5.2 启动 run

扩展统一发送：

```js
{
  type: 'bridge-start',
  tab_id,
  provider: 'claude_code_cli' | 'codex_app_server',
  run_kind: 'plan' | 'candidate',
  session_mode: 'new',
  change_contract: task,
  plan: run_kind === 'candidate' ? {
    plan_id,
    plan_sha256,
    edited_plan_markdown
  } : undefined
}
```

`session_mode` 在本版本必须严格等于 `new`。Background 对 `continue` 直接返回 `SESSION_MODE_NOT_ALLOWED`。旧的 `bridge_sessions` 与 resume 代码不必删除，但 V0.8.1 UI 不得调用。

### 5.3 plan-ready

Host → Background：

```js
{
  type: 'plan-ready',
  run_id,
  task_sha256,
  logical_document_id,
  source_uri,
  source_sha256_before,
  plan_sha256,
  plan: {
    schema_version: 1,
    plan_markdown: '...',
    summary: '...',
    out_of_scope: []
  },
  manifest_path
}
```

Background 必须比对 `run_id`、provider、task SHA、logical document ID、source URI 与 run 记录的扩展侧 base artifact hash；任一不一致即失败，不入库、不广播计划。

**Hash 规则不可简化：**扩展的 `loadedArtifactHash` 来自 DOM 序列化，Host 的 `source_sha256_before` 来自文件原始字节，两者不能直接相等比较。Background 只校验自己保存的扩展侧 hash 是否仍对应当前 tab；Host 在 plan workspace 内校验自己运行前后的原始文件 hash。两种 hash 都可记录为证据，但绝不能相互比较，否则所有正常 plan 都会被误判失败。

校验成功后创建 `bridge_plans` 记录，广播：

```js
{ type: 'bridge-plan-ready', tab_id, run_id, plan_id, plan }
```

不要把 `manifest_path`、session ID 或任何 host 路径广播给 Side Panel。

### 5.4 计划确认与 candidate

candidate 若携带 `plan`，Background 在启动 Native Host 前必须校验：

1. `plan_id` 存在且 `status === 'draft'`；
2. `provider`、`logical_document_id`、`tab_id` 与本次一致；
3. 当前 artifact URL / loaded hash 与 plan 的 `source_artifact_uri` / `base_artifact_hash` 一致；
4. 当前 Change Contract SHA 与 plan 的 `task_sha256` 一致；
5. `edited_plan_markdown` 非空、UTF-8 byte 长度 `<= 12 KiB`；
6. `plan_sha256` 与存储的不可编辑原计划 SHA 一致。

失败码建议：

- `PLAN_NOT_FOUND`
- `PLAN_STALE_SOURCE`
- `PLAN_CONTRACT_CHANGED`
- `PLAN_EDIT_INVALID`
- `PLAN_ALREADY_USED`

通过后把 plan 标为 `approved` 并记录 `candidate_run_id`。candidate 仍是独立新 task；Host 收到的 `approved_plan` 是辅助执行约束，不能替代原 Change Contract。

### 5.5 IndexedDB 迁移

当前 `storage.js` 已使用 `DB_VERSION = 4`。本版本升为 **5**。迁移只能新增，不得删除或重建既有 `bridge_sessions`、`bridge_runs`。

新增 `bridge_plans`：

```js
keyPath: 'plan_id'
indexes: 'logical_document_id', 'tab_id', 'status', 'plan_run_id'
```

建议记录：

```js
{
  plan_id: 'hgp_...',
  plan_run_id: 'hgr_...',
  candidate_run_id: null,
  provider: 'claude_code_cli' | 'codex_app_server',
  logical_document_id,
  tab_id,
  source_artifact_uri,
  base_artifact_hash,           // extension DOM hash，仅与当前 tab 的 loaded hash 比较
  host_source_sha256_before,    // 原始文件 hash，仅由 Host 前后校验
  task_sha256,
  plan_sha256,
  mode,
  root_annotation_ids: [],
  selected_annotation_ids: [],
  plan_markdown: '...',
  summary: '...',
  out_of_scope: [],
  status: 'draft' | 'approved' | 'rejected' | 'stale',
  created_at,
  updated_at
}
```

限制：`plan_markdown` 最大 12 KiB，`summary` 最大 1 KiB，数组总序列化大小最大 4 KiB。禁止存 Agent 原始 stdout、完整 prompt、HTML 源码、认证信息和模型思维链。

在 `LocalStore` 与 `Storage` facade 补齐：`saveBridgePlan`、`getBridgePlan`、`updateBridgePlan`、`markDraftPlansStaleForDocument`（可选）。

## 6. 受控 Plan Bridge

### 6.1 为什么不能只让 Agent 在 stdout 回复计划

stdout 格式不稳定，可能含工具日志、解释、Markdown 或敏感内容；它不能作为可执行协议。计划的可信输入输出边界必须和 candidate 一样由 Host 文件协议控制。

### 6.2 Plan workspace

每个 plan run 创建独立目录，不与 candidate run 混用：

```text
<source-parent>/.htmlgenius-bridge/<provider>/<logical-document-id>/plans/<run-id>/
  source.html                 # source snapshot, 0400
  task-<run-id>.json          # 0400
  task-<run-id>.md            # 0400
  output/
    plan.json                 # Agent 唯一允许的输出
  plan-manifest.json          # Host 写入, 0600
```

- 目录 0700；`source.html` 是真实源的只读 snapshot，Agent 不获得真实源路径。
- 不复用 candidate 的 `runs/<run-id>`，避免 `candidate.html` 被错误接受。
- 创建前、运行后都重新计算真实 source hash。只要真实源在运行期间变化，计划废弃，返回 `SOURCE_MUTATED_DURING_PLAN`。
- `task` 文件运行前后 SHA 必须一致；变化则 `TASK_MUTATED_DURING_PLAN`。
- 运行完只读取唯一的 `output/plan.json`；拒绝 symlink、目录、空文件、超长文件和路径逃逸。

### 6.3 `plan.json` v1

唯一可接受结构：

```json
{
  "schema_version": 1,
  "kind": "htmlgenius_change_plan",
  "summary": "一句话说明计划目标。",
  "plan_markdown": "1. ...\n2. ...",
  "out_of_scope": ["不会修改的内容"]
}
```

严格要求：

- 仅允许以上四个字段；未知字段拒绝。
- `summary`：非空 string，UTF-8 最大 1 KiB。
- `plan_markdown`：非空 string，UTF-8 最大 12 KiB。
- `out_of_scope`：0–20 条 string，每条最大 512 bytes。
- JSON 整体最大 16 KiB，UTF-8，无 BOM 依赖。
- Host 计算原文件 SHA `plan_sha256`；Side Panel 编辑的是副本，不改写 Host 文件。

不得要求或保存“思考过程”。计划只写用户需要审核的操作与边界。

### 6.4 通用 Plan Prompt

在 `task-bundle.mjs` 增加 `buildPlanPrompt()`。固定前言必须表达：

```text
You are preparing an HTML Genius change plan in a controlled workspace.
- Read only source.html and the task bundle in the current directory.
- Do not modify source.html, task files, or any other input.
- Do not create candidate.html or edit the original document.
- Write exactly one UTF-8 JSON file to output/plan.json matching the required schema.
- State only the user-reviewable change plan and out-of-scope boundaries; do not include hidden reasoning.
- Do not use shell, network, browser, MCP, plugins, hooks, credentials, or other directories.
```

随后附上 `ChangeContract.renderPrompt(task)` 和 approved-plan schema。Prompt 是协作约束；真正的强约束仍是 workspace、hash、文件名、validator 和 Host completion 校验。

### 6.5 Claude Code adapter

为 `claude-cli.mjs` / `host-runner.mjs` 增加 `runKind: 'plan'`。

- 只能启动新 task；不走 `--resume`。
- cwd 为 plan run directory。
- 使用最小工具集 `Read,Glob,Grep,Write`；禁用 `Bash`、`Edit`、网络/Web、MCP、plugins、hooks。
- 由于 Claude CLI 的 `Write` 无法可靠地按文件路径授权，Host 必须在运行后检查 source/task hash，并且只接受固定 `output/plan.json`；其它写入不被采用。
- timeout 建议 3 分钟；超时失败 `CLAUDE_PLAN_TIMEOUT`。不得无限等待。
- 成功后不向 extension 传 Claude stdout，只发通过 validator 的 plan。

### 6.6 Codex App Server adapter

把 `CodexAppServerClient.runCandidate()` 的共用 JSON-RPC 序列抽为受限 `runTask()`，再提供 `runPlan()`；不得复制两套 handshake / notification 逻辑。

- `runPlan()` 永远调用 `thread/start`，不调用 `thread/resume`。
- `cwd` 为 plan run directory。
- `approvalPolicy: 'never'`；sandbox 为 workspace write，`writableRoots` 仅限 plan run directory，`networkAccess: false`。
- 仍只允许现有 App runtime discovery、schema compatibility 和 initialize handshake；不得尝试连接用户当前在 Codex macOS App 可见的 task。
- 运行成功后由 Host 校验 `output/plan.json`；不把 App Server notification 文本暴露给 extension。

### 6.7 Host 路由与 terminal event

`bridge/host.mjs` 在 `run_kind === 'plan'` 时路由到 provider 对应的 Plan executor；`candidate` 沿用现有 executor。`handoff` 可以保留给旧兼容路径，但 V0.8.1 UI 不得创建它。

新增 manifest：`htmlgenius_plan_manifest`，至少包含 run/provider/source hashes/task SHA/plan SHA/status/timestamp。失败 manifest 不含 plan 正文。

建议失败码：

- `PLAN_MISSING`
- `PLAN_INVALID`
- `PLAN_TOO_LARGE`
- `PLAN_SYMLINK`
- `PLAN_OUTPUT_PATH_INVALID`
- `SOURCE_MUTATED_DURING_PLAN`
- `TASK_MUTATED_DURING_PLAN`
- `CLAUDE_PLAN_FAILED` / `CLAUDE_PLAN_TIMEOUT`
- `CODEX_PLAN_FAILED` / `CODEX_PLAN_TIMEOUT`

任一失败：不建 candidate sibling，不写 `bridge_plans`，运行记录为 `failed`，发普通 `bridge-failed`。

### 6.8 将确认计划交给 candidate

candidate workspace 中不直接信任或读取旧 plan workspace。Background 将通过验证的用户编辑文本装入 Native message 的：

```js
approved_plan: {
  plan_id,
  plan_sha256,              // 原始受控 plan 的 SHA
  edited_plan_markdown      // 用户可见且已审核的文本
}
```

Host 限长后将其写入 candidate run 中只读 `approved-plan.md`，并将以下内容追加到 candidate prompt：

```text
## Approved implementation plan
Implement this reviewed plan only insofar as it is consistent with the Change Contract.
The Change Contract is the hard boundary. If the plan conflicts with it, preserve the Contract boundary.
```

Agent 仍只能输出 `candidate.html`。原计划不是授权扩大修改范围的通行证。

## 7. Provider probe 实现要求

Host 新增独立 probe helpers：

- Claude：检测 CLI 可执行、读取版本、调用现有的安全 auth/status 检查；不得启动实际 `-p` task。
- Codex：复用 V0.8 的 App discovery、签名校验、`--version`、schema generation / validation 与短初始化 handshake。不得创建 thread 或 turn。

probe 结果应是稳定分类错误，而不是把原始 stderr 透传 UI。超时建议 10 秒；单 provider failure 独立处理。

V0.8 的实际 Codex runtime 是已签名的 macOS App bundle runtime，而非 PATH 中同名 `codex` 包。该判断继续由现有 discovery 代码负责；不要重新引入 PATH `codex` 猜测。

## 9. 自动化测试（最低验收）

### Storage / Background

- DB v4 → v5 不删除已有 bridge session/run record。
- plan record CRUD、长度上限、状态转换。
- `bridge-start` 拒绝 `continue`、未知 provider、非本地 artifact、无 root、旧 `restructure`。
- plan-ready 的 run / task / hash / artifact 任一字段不匹配均拒绝。
- 确认计划时：source 变更、contract 变更、provider 变更、重复使用、空编辑文本均拒绝。
- provider probe 30 秒 cache，某一 provider probe 失败不污染另一 provider。

### Bridge

对 Claude 与 Codex fake adapter 各至少覆盖：

- 合法 `output/plan.json` → `plan-ready`。
- 缺失、无效 JSON、未知字段、超长、symlink、目录、路径逃逸 → 失败。
- 运行时 source 改动 / task 改动 → 失败且不返回计划。
- plan run 绝不创建 sibling candidate。
- candidate 携带 approved plan 时写入只读 plan 文件；无 plan 的直接 candidate 回归不受影响。
- Codex plan 永远 `thread/start`，不调用 resume；网络关闭，writable root 仅计划目录。

### Side Panel

- 点击 CTA 默认直接到 B；全部 non-stale root + reply 选中。
- 高级选项默认折叠；brief/preserve 可空。
- 计数、全选/取消全选、根子树联动、回复缩进、20+ 提示正常。
- 修改范围三项只产生三种合法 mode。
- provider 未 ready 时不可发送/计划；复制仍可用。
- agent menu 没有继续 task、外部会话、复制 JSON。
- plan 文本可编辑；修改 contract 或 artifact 后确认被阻止。
- 每个 workflow 状态无水平滚动；顶栏/底栏固定行为正确；深色主题与 Demo 一致。

## 11. 完成定义

以下全部满足才算 V0.8.1 完成：

- 主路径已经从“先挑评论”收敛为“默认全选 → 直接选择修改范围”。
- 评论范围、brief、preserve 都是可选的高级能力，且文案与 Demo 一致。
- 三档修改范围对新手可理解，且无 `restructure` / “先给我改法”混入范围选择。
- Plan-first 是一个真实、校验过、可编辑、可失效的后端闭环；不是静态 Mock 或 stdout 文本。
- 计划确认始终新建 candidate task，未读取或写入用户外部会话。
- Agent 选择状态为真实 probe，未就绪不能派发。
- V0.8 candidate 安全边界、原文件不覆盖和既有评论交互无回归。
- 自动测试通过，并在实现状态文档中如实注明真实 smoke 的完成情况。
