# v0.6.2 本地 HTML 文档版本一致性（Artifact Version Reconciliation）施工 Plan

> 日期：2026-07-18  
> 前置版本：v0.6.1 Change Contract 已完成。  
> 后续依赖：v0.7 Local Agent Bridge 必须建立在本版本之上。  
> 本版本不实现 Agent 调用、Native Messaging、Codex/Claude/Copilot adapter 或自动写文件。

## 0. 结论与边界

当前实现**不能直接作为 Agent 写回的基础**。

它已有一个避免旧编辑快照覆盖外部更新的补丁：`content-script.js` 在初始化时把当前 `body` 的规范化文本做 `djb2` hash，只在该 hash 与本地 IndexedDB 中最新编辑快照的 `base_hash` 一致时，才恢复快照。但这个机制只覆盖“同一路径、刷新后、正文文本发生变化”的部分场景。

存在四个确定缺口：

1. 浏览器已打开时，Agent/外部程序改了磁盘文件，当前标签页的 DOM 不会自动更新。
2. 只改变 `head`、CSS、class、attribute 或 DOM 结构而正文文本相同，会误判文件未变，并可能把旧的 `body.innerHTML` 快照恢复到新文件上。
3. 当前快照只保存 `body.innerHTML`，不是完整 HTML artifact，无法可靠表达视觉 HTML 的版本。
4. `document_id = location.origin + location.pathname`。若生成新文件路径，它被视为另一份文档，既有批注不会跟随。

v0.6.2 的目标是建立**可验证的 artifact 版本协议**，而不是“监听文件然后 reload”。核心模型必须区分：

- `logical_document_id`：用户认知上的同一份页面/报告，跨文件路径与版本保持不变；批注归属它。
- `artifact_uri`：当前版本实际加载的 URL/path，例如 `file:///.../report.html`。
- `artifact_hash`：完整 artifact 内容的 SHA-256；用于并发检测与快照恢复判断。
- `artifact_version`：一次已确认的内容版本；用 `base_hash -> result_hash` 表示父子关系。

本版本完成“扩展端能正确识别、接受和呈现一次已完成的本地 artifact 更新”的基础。未来 Bridge 只需在写文件成功后调用此协议，不能自行绕过它。

## 1. 非目标

本版本明确不做：

- 不实现或调用任何 LLM、Token、SaaS API。
- 不建立 Native Messaging host，也不访问本地任意文件系统。
- 不调用 Codex App Server、Claude Code 或 Copilot。
- 不实现文件 watcher。扩展本身没有也不应有对任意本地路径的静默监听权限。
- 不自动合并 Agent 结果与用户正在编辑的 DOM；发生基线不一致时必须停下并报告冲突。
- 不修改 v0.6.1 Change Contract 的 schema 与 UI 行为。
- 不修改远端协同服务/API；本版本仅支持本地 HTML artifact。
- 不尝试在 v0.6.2 内解决“语义上是否改对了”。这里只解决版本正确性和批注重定位；语义验收属于后续候选 diff/review。

## 2. 当前代码基线与必须替换的部分

| 现有位置 | 当前行为 | v0.6.2 处理 |
|---|---|---|
| `extension/storage.js` | IndexedDB `versions` 只存 `html_content`（body innerHTML）与 `base_hash` | 升级为完整 artifact 快照和 metadata；保留旧记录兼容读取 |
| `LocalStore.getDocumentId()` | 直接返回 `location.origin + location.pathname` | 对本地 artifact 引入稳定 `logical_document_id` 映射；物理 URI 不再直接充当批注 ID |
| `content-script.js: captureBodyForSave()` | 仅保存 body、去掉 overlay | 继续供 undo 使用；持久化 artifact 快照改为完整文档序列化 |
| `content-script.js: hashBodyText()` | 文本级 djb2 | 删除其作为 freshness 决策的用途；改为完整、确定的 SHA-256 指纹 |
| `content-script.js: restoreIfFresh()` | hash 相同则恢复旧 body 快照 | 只在完整 artifact hash 精确一致时恢复；不一致绝不写入 DOM |
| `content-script.js: get-export` | 返回标题/URL/isLocal | 返回 logical document 与 artifact version metadata，供后续 Bridge 任务携带 |
| `sidepanel.js` | 无版本状态、无手动受控刷新入口 | 显示本地 artifact 状态；提供“重新读取文件”作为 v0.6.2 的人工验收入口 |

注意：`captureBodyForSave()`、`applyRestoredBody()` 仍可被 undo/redo 使用。不要为了做 artifact 版本而重构 v0.6 编辑历史，否则会扩大风险。**本地编辑快照与完整 artifact 基线是两个不同用途的对象。**

## 3. 数据模型与 IndexedDB 迁移

### 3.1 DB version

将 `extension/storage.js` 的 `DB_VERSION` 从 `1` 升为 `2`。在 `onupgradeneeded` 中新增以下 object store；不得删除现有 `annotations`、`versions`，不得改变它们的 key。

#### `documents`

`keyPath: "logical_document_id"`。

```js
{
  logical_document_id: "hgd_...",       // 随机 UUID/足够随机 ID，仅首次绑定时产生
  canonical_uri: "file:///.../report.html",
  known_uris: ["file:///.../report.html", "file:///.../report-v2.html"],
  created_at: "ISO-8601",
  updated_at: "ISO-8601"
}
```

创建索引：`canonical_uri`（非 unique）。`known_uris` 不建 multiEntry 索引；通过逐条 document 扫描匹配即可，数据量很小。

#### `artifact_versions`

`keyPath: "id"`，`autoIncrement: true`；索引 `logical_document_id`（非 unique）、`artifact_hash`（非 unique）。

```js
{
  id: 1,
  logical_document_id: "hgd_...",
  artifact_uri: "file:///.../report.html",
  artifact_hash: "sha256:...",
  parent_hash: "sha256:..." | null,
  source: "local_edit" | "external_reconcile" | "bridge", // 本版只有前两个实际写入
  created_at: "ISO-8601",
  html_content: "<!doctype html>..." // 仅 local_edit 快照可存在；external_reconcile 不复制文件内容
}
```

`versions` 保留为 legacy read-only store：v0.6.2 不再向它写入。第一次打开存在旧版本记录的页面时，旧记录不能自动恢复，因为它没有完整文件 hash；保持当前文件 DOM，并在 console 记录一次迁移跳过。不得删除用户旧数据。

### 3.2 document 与 URI 的解析规则

在 `storage.js` 新增、并通过 `window.Storage` 暴露：

```js
getOrCreateLocalDocument(uri) -> Promise<{ logical_document_id, canonical_uri, known_uris }>
getDocumentByUri(uri) -> Promise<document | null>
linkArtifactUri(logicalDocumentId, uri) -> Promise<document>
getLatestArtifactVersion(logicalDocumentId) -> Promise<version | null>
saveArtifactVersion(record) -> Promise<record>
```

规则：

1. 首次打开 `file:` 或 localhost HTML：以去除 fragment 的完整 `location.href` 作为 `uri`，创建一条 document，并把原 `location.origin + location.pathname` 下的本地 annotations 迁移到新 `logical_document_id`。迁移必须幂等。
2. 已注册 URI：返回已有 `logical_document_id`。
3. 新 URI 不能靠“文件内容相似”自动认亲。只有未来 Bridge 的受控结果调用 `linkArtifactUri()`，才能把新路径显式归到已有逻辑文档；手动打开一个同内容的新文件时，应视作新文档，避免批注错误串联。
4. remote/synced 页面维持既有 `Storage.getDocumentId()` 语义，不纳入本版本。

### 3.3 批注 ID 迁移

旧本地 annotation 的 `document_id` 为 `legacy_uri_id = location.origin + location.pathname`。首次绑定本地 document 时：

- 列出旧 ID 下的 annotation；
- 只把其 `document_id` 改成新的 `logical_document_id` 后 `put` 回原 store；
- 不复制 annotation，不改变 id、回复关系、selector、author；
- 再次启动不得重复迁移；
- remote/synced mode 不迁移。

这一步是必要的。只新增逻辑 ID 而不迁移，页面加载、批注保存和 Change Contract 会分别读到不同集合。

## 4. 完整 artifact 指纹

### 4.1 指纹输入

新增纯函数模块 `extension/artifact-version.js`，暴露 `window.HgArtifactVersion`。必须不读取 `chrome` API，方便单测。

```js
normalizeArtifactHtml(html) -> string
sha256Hex(text) -> Promise<"sha256:...">
serializeCurrentArtifact(documentElement) -> string
stripExtensionNodes(root) -> void
```

`serializeCurrentArtifact` 的输出规则：

1. clone 整个 `document.documentElement`，而不是 body。
2. 移除扩展注入节点：`#hg-toolbar`、`.hg-hl`、`.hg-inspect`、`.hg-select`、`.hg-tip`、`.hg-drop`，以及扩展注入的 `<style>`。为注入 `<style>` 增加稳定标识 `data-hg-injected="ui"`，不能依赖 style 文本匹配。
3. 返回 `"<!doctype html>\n" + clone.outerHTML`；doctype 不存在时同样使用这个前缀，以保证同一页面的 hash 稳定。
4. 不以 `textContent`、`innerHTML` 或视觉截图作为 hash 输入。
5. `normalizeArtifactHtml` 只规范化 line ending (`\r\n`/`\r` → `\n`)；不压缩空白、不重排属性、不运行 sanitizer。目的是内容身份，不是语义等价判断。
6. 用 Web Crypto `crypto.subtle.digest("SHA-256", ...)` + `TextEncoder` 计算 hash。若浏览器不支持 Web Crypto，返回显式错误并禁用本地编辑快照恢复；不得降级回弱文本 hash。

### 4.2 时序

- content-script 注入 UI **前**，对原生 document 计算 `_loadedArtifactHash`。
- 页面初始化时先取得本地 `logical_document_id`，再决定是否恢复 local_edit 快照。
- 每次用户本地编辑防抖落盘时：保存完整 HTML snapshot 和 `base_artifact_hash = _loadedArtifactHash`；不要把编辑后 HTML 的 hash 误当原始基线。
- `restoreIfFresh()` 的条件改为：最新 `local_edit` snapshot 的 `base_artifact_hash === _loadedArtifactHash`。相同才恢复完整快照；否则不恢复任何 DOM。
- 从快照恢复后，页面当前呈现内容已不同于加载文件；响应 `get-artifact-state` 时必须同时返回 `loaded_artifact_hash` 与 `rendered_artifact_hash`，不要伪装它们相同。

完整快照必须通过一个新的 `applyRestoredArtifact(html)` 恢复：保留现有 toolbar 节点及其监听，重新挂回 body；`head` 则由解析后的 artifact 替换为 snapshot head，但要重新注入/保留扩展 UI style。实现时先单独构建 DOM helper 并测试，避免 `document.documentElement.innerHTML = ...` 直接破坏 content-script 的运行环境。

## 5. 版本状态机与消息协议

### 5.1 可见状态

content-script 维护以下只读状态：

```js
{
  logical_document_id,
  artifact_uri,
  loaded_artifact_hash,      // tab 最初加载的物理文件版本
  rendered_artifact_hash,    // 当前 DOM（可能是本地编辑快照）
  has_unsaved_local_snapshot,
  last_reconcile_status       // "clean" | "reloaded" | "conflict" | "error"
}
```

`get-annotations` 与 `get-export` 都添加此对象（字段命名统一为 `artifact_state`）。`get-export` 的 existing `artifact` 字段保持兼容，附加 `logicalDocumentId`、`loadedArtifactHash`，供后续 v0.7 task builder 使用。

### 5.2 v0.6.2 手动协调入口

Side Panel 的本地编辑视图增加一个小型次级按钮：`重新读取文件`。仅在 `isLocal === true` 时显示。

点击流程：

1. 向 content script 请求 `prepare-artifact-reload`。
2. 若 `_editing` 或 `has_unsaved_local_snapshot` 为 true，content script 返回 `needs_confirmation`，Side Panel 说明“刷新将丢弃仅存在于插件内、尚未导出的编辑快照”，用户二次确认后才能继续。
3. 确认后 Side Panel 调 `chrome.tabs.reload(tab.id, { bypassCache: true })`。
4. 新页面加载后，content script 执行完整 hash + `restoreIfFresh()` + `loadAnnotations()`；Side Panel 显示锚定结果摘要。

这里的 `bypassCache` 只解决浏览器 HTTP cache；对于 `file:` 页面它不是版本识别机制。真正的保护是 hash 和 restore gate。

### 5.3 预留给 Bridge 的受控完成协议

现在实现消息协议，但不实现 Native host。仅 extension 内部消息可以触发：

```js
{
  type: "artifact-update-ready",
  logical_document_id: "hgd_...",
  base_artifact_hash: "sha256:...",
  result_artifact_hash: "sha256:...",
  result_artifact_uri: "file:///.../report.html",
  result_kind: "overwrite" | "new_artifact",
  source: "bridge"
}
```

处理规则：

1. 拒绝非本地页、字段缺失、未知 `logical_document_id`、非 SHA-256 格式或 `source !== "bridge"` 的消息。
2. 当前 `loaded_artifact_hash` 必须等于 `base_artifact_hash`，且没有本地编辑快照；否则返回 `{ ok:false, code:"BASE_CONFLICT", current_hash }`，**不 reload、不写记录、不链接新 URI**。
3. `overwrite`：`result_artifact_uri` 必须等于当前 URI。保存一条 `artifact_versions` metadata 后，Side Panel/未来 background 执行受控 reload。
4. `new_artifact`：不得直接 reload 当前 tab；先调用 `linkArtifactUri(logical_document_id, result_artifact_uri)`，然后只允许未来 Bridge 显式导航到新 URI。导航完成后再由新 tab 的 content-script 校验完整 hash 必须等于 `result_artifact_hash`；不等则显示错误、不得加载/迁移批注状态。
5. 所有 reload 后都重新调用 `loadAnnotations()`，每条 annotation 只有 `open` 或 `stale` 两种结果。不要伪造“已迁移”状态：当前 TextQuote 算法无法区分原位置未变和通过 prefix/suffix 找到新位置。若要显示“已迁移”，必须先扩展 anchor 算法返回 relocation metadata，此版本不做。

消息必须只作为**完成事件**，不是“请 Agent 写入文件”的命令。写文件授权、Agent session、adapter 都属于 v0.7。

## 6. UI 与文案

仅增加最小可见 UI，不做任务中心：

- 编辑 tab、`isLocal` 时：显示 `重新读取文件`。
- 当出现 `BASE_CONFLICT`：显示“文件已在本地编辑后发生变化。为避免覆盖，请先导出或放弃插件内编辑，再重新读取文件。”
- reload 后在编辑 tab 顶部显示一次性状态（可复用现有 alert/toast）：
  - `已加载文件新版本；X 条批注已定位，Y 条需要处理。`
  - 不给出“已自动合并”“已安全同步”之类误导性措辞。
- Change Contract Composer 不改 UI；未来从 `get-export.artifact_state` 取得 version metadata 即可。

i18n 中/英/日三个语言都必须添加以上键。不要在 `sidepanel.js` 写中文硬编码。

## 7. 实施顺序

### Step 1：可测试的 artifact helper

新增 `extension/artifact-version.js` 与 `extension/artifact-version-test.html`。

先完成：完整序列化、扩展节点剥离、line ending 规范化、SHA-256。测试页面 title 用 `PASS` / `FAIL: reason` 表达结果，沿用仓库现有测试风格。

### Step 2：IndexedDB v2 与逻辑文档迁移

在 `storage.js` 实现 DB upgrade、document/URI 映射、legacy annotation 一次性迁移、artifact version CRUD。

新增 `extension/artifact-storage-test.html`：使用独立 test URI，验证首次绑定、重复绑定幂等、legacy annotation 迁移不复制、`linkArtifactUri` 后新 URI 返回同一 logical ID。

### Step 3：替换本地快照 freshness 判定

在 `content-script.js`：

- 用完整 SHA-256 取代 `hashBodyText()`；
- 不再写旧 `versions`；
- 新 artifact snapshot 写入 `artifact_versions`；
- 只在完整 `base_artifact_hash` 相等时恢复；
- 复核恢复 full document 时 extension 注入节点、toolbar 事件、批注高亮都仍正常。

保留 undo/redo 的 `body` 级实现，禁止把它改成全页面 undo。

### Step 4：状态查询与手动 reload

增加 `get-artifact-state`、`prepare-artifact-reload` 与 Side Panel 按钮/确认状态；reload 后恢复 Side Panel 激活，并刷新批注卡片。

### Step 5：Bridge completion protocol（仅消费者，不建 Bridge）

实现 `artifact-update-ready` 的严格 schema 检查、base conflict、同路径 overwrite、预注册链接与 new artifact 的跳转前检查函数。此步不需要真实 Native host；用 content-script test hook/浏览器手工消息验证即可。

建议对 `artifact-update-ready` 的真正分发位置预留在 `background.js`：未来 Native Messaging 只向 background 发送结果，background 再定位 target tab 并转发给 content script。v0.6.2 可以先不改 manifest permission，但应在 `background.js` 写一个**不启用的**纯转发 helper 或 TODO 注释，不得假装已接入 native host。

## 9. 验收标准

### 自动/独立测试

1. 相同完整 HTML 的 SHA-256 恒定；只改 class、style、`head`、属性、空白或 body 结构中的任一项，hash 变化（line ending 差异除外）。
2. 序列化 artifact 不含 htmlGenius toolbar、overlay、元素模式框或注入 style。
3. legacy annotation 首次迁移到 logical document；第二次调用不复制、不丢回复；remote document 不迁移。
4. 旧 `versions` 记录存在时不会触发 body 恢复，也不会抛异常。
5. current loaded hash 与 snapshot base hash 不同，`restoreIfFresh()` 不改变当前 DOM；相同则完整恢复 `head + body` 且 toolbar 仍是原节点。
6. `artifact-update-ready` 在 base hash 不同、存在 local snapshot、URI 不匹配等情况都返回 `BASE_CONFLICT`/validation error，且不 reload。
7. 受控 `new_artifact` 调用后新 URI 与旧 URI 返回同一 `logical_document_id`，未受控的新 URI 不会自动继承。

### 手工浏览器测试

用两个本地 HTML：`report.html` 与 `report-v2.html`。至少覆盖：

1. 打开 `report.html`，加批注；关闭/刷新后批注存在。
2. 在插件内修改 body，刷新，完整文件未变时快照恢复；toolbar、高亮、undo 仍工作。
3. 在外部编辑器只改 `head` CSS 或元素 class，点击“重新读取文件”；页面必须显示外部新视觉结果，不能恢复旧 body 快照。
4. 外部编辑器修改正文，reload 后仍能锚定的评论显示 `open`，找不到的显示 `stale`。
5. 有未导出的插件内编辑时点击“重新读取文件”，必须先出现明确确认；取消后不 reload。
6. 模拟 `artifact-update-ready` overwrite：base 一致才 reload；base 不一致时页面内容、URI、annotation 归属均不变。
7. 模拟 `new_artifact`：在明确 link 后打开 `report-v2.html`，批注随 logical document 读取并重新锚定；直接手动打开一个第三个同内容文件时，不出现这些批注。

## 10. 交付定义与下一步

v0.6.2 完成的标志不是“文件变化时能刷新”，而是以下不变量都成立：

1. 扩展永远不会因文本 hash 碰撞或 CSS/结构漏判，把旧插件快照恢复到外部更新后的文件上。
2. 本地批注的归属不再由文件路径隐式决定；新路径是否继承批注必须是受控、显式的决定。
3. 任何写回完成事件都必须携带并通过 `base_artifact_hash` 校验；冲突默认不动作。
4. reload 后批注重新锚定；不能定位的批注被保留并标为 stale，而不是静默丢弃。

达到这些条件后，v0.7 才开始做 Local Bridge：Bridge 负责创建/续发它拥有的 Agent session、写 candidate/target artifact、计算 SHA-256，并把 `artifact-update-ready` 交给本版本已实现的消费者。Bridge 不负责决定批注如何迁移，也不能绕过版本冲突校验。
