# v0.7.1 补充：评论到编辑任务的选择流 Spec

> 日期：2026-07-20  
> 前置：v0.6.1 Change Contract 与 v0.7.1 Claude Code handoff 已存在。  
> 实施性质：这是 **v0.7.1 的 Side Panel UI 补充**，不是新的 Agent、AI 分类或文件回写版本。  
> 对照稿：[change-contract-agent-flow.html](ui-mockups-v0.7.1/change-contract-agent-flow.html)

## 0. 先读这一段：这次要解决什么

当前入口是「评论列表 → 一键生成修改任务」。这个方向有问题：用户添加评论后，首先是在查看、补充、回复和澄清评论；并不意味着要立刻把全部评论交给 Agent。直接打开 Change Contract 会把“讨论”与“执行”混为一个动作。

新流程必须是：

```text
在网页上新增评论
  → Side Panel「评论」页：默认查看所有已添加评论
  → 点击「整理评论，创建编辑任务」
  → 选择本次要发送的顶层评论
  → 填写 Change Contract
  → 复制 Prompt 或发送给 Claude Code
```

**核心原则：** 评论默认只是讨论记录；只有被用户显式选中的评论才是本次编辑任务的输入。没有任何模型、规则或服务替用户判断一条评论是不是“修改建议”。

## 1. 范围、非目标与不可破坏项

### 本次必须做

1. 保留现有「评论」页作为默认落点；用户打开 Side Panel、点击页面高亮、或新建评论成功后，都先看到评论列表，而不是选择器或 Composer。
2. 将现有底部 `生成修改任务` 改为 `整理评论，创建编辑任务`。
3. 点击后在现有全屏 `#contract-sheet` 内先展示**评论选择步骤**；用户可多选任意未 stale 的顶层评论。
4. 用户点击「继续」后，进入已有 Change Contract 表单；表单和 Claude Code bridge 只接收该次选中的 root IDs。
5. 清楚显示总评论数与已选数；可处理评论超过 20 条时显示人工筛选提示。
6. 返回、关闭、Esc 都只丢弃临时选择，不修改原评论，也不启动 Agent。
7. 中、英、日 i18n 完整，不把新中文文案硬编码在 JS。

### 本次明确不做

- 不接入 AI 自动分类、Embedding、模型 API 或任何外部 token。
- 不新增、也不持久化 `澄清问题 / 修改建议` 的 annotation 字段、tag 或筛选。**本版勾选就是人工判断。**
- 不改评论卡片、评论创建/回复/编辑/删除/定位等既有交互，也不在卡片上新增操作按钮。此次只在既有列表底部增加进入任务选择的入口。
- 不改 IndexedDB schema、远端协同 payload、登录权限。
- 不改 `change-contract.js` 的 task schema；它已有 `rootIds: string[]`，足以承接多选。
- 不做历史任务、保存筛选、全选推荐算法、批量修改评论类别。
- 不改 v0.7.1 的 bridge ownership 边界：仍只允许新建 Claude task 或续发 HTML Genius 创建的 task，绝不读取、枚举或接管任意外部 Claude 会话。

### 允许改动文件

- `extension/sidepanel.html`
- `extension/sidepanel.css`
- `extension/sidepanel.js`
- `extension/i18n.js`
- 必要时新增一个纯 DOM 测试页，例如 `extension/comment-task-selection-test.html`

除非测试暴露真实 bug，不要修改 `storage.js`、协同 API、`content-script.js`、Native Host、bridge 协议或 `change-contract.js`。

## 2. 页面状态机

Side Panel 的“评论”tab 内有三个状态；不要把它做成三个浏览器页面，也不要新增一个 tab。

| 状态 | 何时出现 | 用户能做什么 | 离开方式 |
|---|---|---|---|
| A. 评论收件箱（默认） | 打开评论 tab、新建评论、点击页面批注 | 原样使用既有评论列表交互 | 点击底部 CTA 进入 B |
| B. 挑选本次评论 | 用户主动点击 CTA | 勾选/取消勾选顶层评论；确认数量 | 继续进入 C；返回 A；关闭回 A |
| C. Change Contract | B 至少选中 1 条后点击继续 | 填目标、范围、执行方式、保留项；复制或发 Claude | 返回 B 修改选择；关闭回 A |

`#contract-sheet` 可以继续作为全屏 overlay，但其内部需要有 `data-step="select" | "compose"` 两个显式子状态。禁止通过多套 `hidden` 布尔值拼状态，否则返回和 Esc 容易留下脏选择。

### 状态转换规则

```text
A --「整理评论，创建编辑任务」--> B
B --「继续」且 selectedRootIds.length > 0--> C
B --「返回页面评论」 / close / Esc--> A
C --「更改评论」 / back--> B（保留本轮已选项与已填 form）
C --close / Esc--> A（丢弃本轮 selectedRootIds 和 form draft）
```

没有可用 root comment 时，A 的 CTA disabled，并沿用现有空态。stale root 不进入 B，也不计入可处理总数。

## 3. 视觉与交互还原要求

施工前必须打开并以此为主视觉参考：

`docs/ui-mockups-v0.7.1/change-contract-agent-flow.html`

不要把它理解成“功能参考图”后自行换成普通表单。要还原以下结构、密度和层级；允许继续使用项目既有 design token，而不是复制 mockup 的具体 hex 色值。

### A. 评论收件箱（默认状态）

1. 维持项目现有卡片列表与回复树。**所有评论都在列表中自然连续呈现，超出首屏时由列表继续滚动，不能用“还有 N 条评论”代替真实卡片。**底部任务 CTA 固定在 Side Panel 底部，但不得阻断、替换或截断评论列表自身的滚动。
2. 页首只出现现有 tab 与正常评论内容，**不得**在用户尚未点击 CTA 前展示 checkbox、已选数、Prompt、Claude session 或“任务模式”。
3. 底部唯一任务 CTA：`整理评论，创建编辑任务`。视觉应为次级/outline CTA，而非压过“新增评论”的主操作。
4. CTA 附近有一个紧凑说明或 `?`：`先挑选这次要处理的评论；未选评论不会发送给 Agent。`
5. 不触碰评论卡片内部现有的定位、回复、编辑、删除等交互；本期既不重设计、也不在 mockup/代码中额外添加这些按钮。

### B. 评论选择步骤

1. 顶部：返回按钮 `← 返回页面评论`、标题 `挑选这次要处理的评论`、一句说明 `不是每条评论都要交给 Agent。先不确定的讨论可以不选。`，旁边 `?` tooltip。
2. 计数行：`可处理评论 N 条 · 本次已选 M 条`。这里的 N/M 只数**顶层且 non-stale** comment；回复会随其 root 一并进入任务，不单独占一个 checkbox。
3. 超过 20 条：显示 warning banner：`评论超过 20 条时，不建议逐条手选。可分批创建编辑任务。`。等于 20 不显示。
4. 每张 root card 的左侧 checkbox 是 16px 左右的真实 `<input type="checkbox">`，整张卡可点击切换；保证键盘可 focus/Space 切换。卡片继续显示 quote、评论正文和回复数。
5. 初次进入 B 时，默认选中全部可处理 root。原因是保持目前“全部导出”的连续性；用户的新增能力是**取消**本次不应进入 Agent 的评论。不要猜测、自动分类或自动排除任何评论。
6. 已选卡有清晰但克制的边框/背景高亮；取消后恢复普通卡。禁止只靠颜色表达选中状态。
7. 底部固定按钮文案动态为：`用选中的 M 条评论继续`；M=0 时 disabled，文案为 `请至少选择一条评论`。
8. 不做“修改建议 / 澄清问题”标签、筛选 pill 或“改成澄清问题”按钮。本期的勾选行为就是人工判断；若需要解释，放在 `?` 中，不能假装系统已识别评论类型。

### C. Change Contract 步骤

1. 复用现有 v0.6.1 Composer 表单与校验；不要再显示一份 checkbox 列表。
2. 表单标题前增加返回 B 的 `← 返回挑选评论`；来源区域首行显示 `本次将发送 M 条评论（可处理共 N 条）`，提供 `更改` 链接回 B。
3. 详情列表只列本轮 M 个 roots；每个 root 的回复仍由 `ChangeContract.buildTask()` 完整带出。
4. 模式文案改为用户可理解的四种目标：

| 内部 mode | 露出文案 | 一句话说明 |
|---|---|---|
| `precise_patch` | 只改圈选处 | 只改我评论对应的内容，页面其他地方完全不动。 |
| `local_optimize` | 优化这一区块 | 可以连同周围这一块调整，但不要影响页面其他部分。 |
| `restructure` | 先给我改法 | 先告诉我准备怎么改，不生成或修改文件。 |
| `regenerate` | 重做整页 | 按新的要求重做这份页面，原来的任何部分都可能改变。 |

`restructure` 仍然是 “plan only”；不能把“先给我改法”做成可同时叠加在其它 mode 上的开关，以免破坏既有 schema 和 bridge 逻辑。本次只更新呈现文案；若业务未来要把“范围”与“执行动作”真正拆成两个正交字段，必须另开 schema 版本，不允许在此偷改。

5. 将字段文案改成：
   - `补充说明` → `你想让页面变成什么样？`
   - placeholder/帮助 → `批注里没说清的目标、受众或风格。`
   - `必须保留` → `有绝对不能动的内容吗？（可选）`
   - 帮助 → `只写即使允许大改，也必须原样保留的内容。`
6. 长解释使用标题旁 16px 左右的 `?` icon + hover/focus tooltip，不能常驻占用 Side Panel 高度。tooltip 必须支持 keyboard focus；触屏/点击以 click toggle 也可，但不能只支持 hover。
7. Copy Prompt 与 Claude Code handoff 的现有逻辑保持可用。发送区仍明确只提供：`创建新的 Claude task`、`继续 HTML Genius 上次创建的 Claude task`；补充文案：`不会显示或接管你正在运行的其他 Claude 会话。`

## 4. 数据与实现设计

### 4.1 临时 UI 状态

状态只存在 `sidepanel.js` 内存中，关闭 sheet 时清空：

```js
let _contractStep = "closed"; // closed | select | compose
let _selectedContractRootIds = new Set();
let _contractItems = [];
let _contractArtifact = null;
let _contractBridgeMeta = null;
```

不要把选择写入 annotation、`chrome.storage` 或 IndexedDB。重新打开任务选择时重新默认全选当前 non-stale roots；页面评论实时更新后也必须按最新 `get-export` 数据重新计算。

### 4.2 入口与数据获取

现有 `#export-btn` listener 不再直接调用 `openContract(roots.map(...))`。改为：

1. 调用 `get-export`；失败按现有 toast 处理。
2. `roots = ChangeContract.getRoots(resp.items)`。
3. 无 roots：toast `contract.empty` 并 return。
4. 保存本轮 export snapshot；`_selectedContractRootIds = new Set(roots.map(root => root.id))`。
5. 进入 `openContractSelector(roots, resp.items, resp.artifact, bridgeMeta(resp))`。

单卡片的 `生成任务`（如仍保留）只能作为快捷入口：直接以该 root 建 Set 后进入 **B**，不能绕过选择步骤直接打开 C。更推荐本期删除单卡片“生成任务”按钮，避免两个心智模型；保留回复、编辑、删除、定位。

### 4.3 从 B 进入 C

点击继续时使用 Array.from Set 生成 `rootIds`。`openContract(rootIds, _contractItems, _contractArtifact, _contractBridgeMeta)` 的现有初始化逻辑可复用，但需要增加一个参数或拆出初始化，避免它每次都重置选择。

`getContractDraft()` 必须从 `_selectedContractRootIds` 读取 rootIds，不能再依赖旧的固定 `_contractRootIds`。提交前 `ChangeContract.validateDraft()` 是最终防线；若某条评论在选择后已 stale/删除，显示可理解错误并回到 B 刷新，而不是发送旧 task。

### 4.4 DOM 安全

- 评论正文、作者、quote 一律经现有 `esc()` / `linkify()` 处理；禁止为了快速渲染 checkbox 卡片拼入未转义 HTML。
- checkbox 的 `change` listener 用事件委托；卡片 `scroll-to` click 必须忽略 checkbox、label、button 子元素，避免勾选时页面跳转。
- 多选顺序必须保持 `getRoots(items)` 的原始顺序，不按勾选时间排序，保证 Prompt/JSON 稳定。

## 6. 验收清单（实现 Agent 必须逐条验证）

### 手工流程

1. 新安装/打开 Side Panel 的评论 tab，看到的是正常评论列表；不会直接看到 checkbox 或 Composer。
2. 添加一条评论后，列表实时更新；点击底部 `整理评论，创建编辑任务` 才进入选择步骤。
3. 选择步骤初始全选所有 non-stale roots；取消任意两条，计数、按钮文案、卡片状态同步更新。
4. 选择 0 条时不能继续；重新选择 1 条后恢复。
5. 有 root 回复时，只显示一个 root checkbox；生成的 Change Contract 包含完整 reply tree。
6. 选择步骤超过 20 个可处理 roots 显示 warning；20 个及以下不显示。
7. B → C 后，C 的来源摘要和 Prompt/JSON 都只含选中 roots；未选 root 的 quote、comment 和 replies 都不存在于输出。
8. C → `更改评论` → B 后仍保持选择和已填的 brief/preserve/mode；C 关闭或 Esc 后再次进入，则恢复默认全选且表单为空。
9. stale comment 永不出现 checkbox、永不计入 N、永不进入 task。
10. 复制 Prompt、复制 JSON、发送 Claude Code 的成功/失败行为与 v0.7.1 现有测试一致。
11. 三个状态（A 评论收件箱、B 评论选择、C Change Contract）都必须 `overflow-x: hidden`，**绝不出现横向滚动条**。Side Panel 宽度固定，长文本必须换行；tooltip、Agent 菜单等浮层必须在面板可视宽度内向内展开，不能以横向滚动作为溢出处理。
12. 窄 Side Panel（最小宽度）中底部 CTA 不被挤出，列表和 sheet 仅可纵向滚动，tooltip 不被横向裁切。
13. 深色、浅色主题与 zh/en/ja 均检查一次；无英文 key 泄漏。

### 自动化/测试页最低要求

新增或扩展浏览器测试，至少断言：

- `openContractSelector()` 初始选中所有 `ChangeContract.getRoots()`；
- 取消选择后 task 的 `source.root_annotation_ids` 精确相等；
- reply 不会作为 root checkbox 和计数项；
- stale root 被过滤；
- M=0 禁用继续；
- 关闭会清空临时 Set；
- `ChangeContract.buildTask()` 和既有 v0.6.1 tests 保持通过。
