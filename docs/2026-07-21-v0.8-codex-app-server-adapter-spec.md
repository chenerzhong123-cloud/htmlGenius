## 0. 给 GLM 5.2 施工 Agent 的结论

实现 `codex_app_server` provider：用户在 HTML Genius Side Panel 选择 Codex 后，一键把完整 Change Contract 交给**本机已安装、已登录的 Codex Mac App 所捆绑的 App Server runtime**，生成新的 candidate HTML。它复用既有 Native Host、candidate workspace、hash、manifest、`new_artifact` 和重锚路径；**不得重建第二套文件协议**。

```text
Side Panel
  → Background（校验当前 artifact 和 run）
  → 既有 Native Host
  → codex app-server（stdio JSON-RPC）
  → run workspace/candidate.html
  → Host hash + manifest
  → 既有 new_artifact + candidate 打开 + 评论重锚
```

这不是 Codex 对话 UI、也不是读取用户 Codex 历史的集成。用户得到的是“新建一个 HTML Genius 拥有的 Codex task，或续发此前同一页面由 HTML Genius 创建且已结束的 task”。它**不是**向当前桌面 App 正在运行或已打开的 task 写消息；App 没有公开该能力，且这也违反本产品的 ownership 边界。

## 1. 产品契约与硬边界

### 必须做到

1. 只使用用户机器上已有的 Codex Mac App 登录态及其随 App 安装的 runtime；HTML Genius 不收集、不保存 API key、ChatGPT 凭据、Cookie 或 Codex response。
2. 点击后创建一个新的 bridge-owned Codex thread；只有该逻辑文档记录过、provider 为 `codex_app_server`、且 `last_status` 为 terminal 的 thread 才允许续发。
3. 所有可写模式只生成 candidate artifact；source HTML 从不被 Codex、Host 或扩展覆盖。
4. candidate 成功后必须走 v0.6.2 已有的受控 `new_artifact` 路径，打开新版本并重新锚定评论。
5. Side Panel 显示受控、可行动状态：未安装、未登录/未就绪、协议不兼容、正在生成、source 冲突、candidate 无效、成功及重锚统计。
6. 新 provider 必须与 Claude Code 并存；选择一个 provider 不得改变另一个 provider 的 session、run 或 UI 状态。

### 绝对禁止

- `thread/list`、`thread/read`、搜索、picker 或导入任何用户既有 Codex thread。
- `turn/steer`、`thread/inject_items`，或向正在运行的会话注入内容。
- 续发未由 HTML Genius 创建的 thread，或任何 running / interrupted thread。
- 以 prompt 代替安全边界；直接给 Codex 真实 source path 或 source parent 的写权限。
- 覆盖 source、自动 promote、diff/review UI、队列/后台常驻任务、远程 Codex 服务、MCP task pull。
- 通过 Native Host 接收任意 command、任意 cwd、任意环境变量或用户可控 JSON-RPC method。

### `restructure` 特例

`restructure` 仍是 plan-only。它只能复制规划 Prompt；不能出现 `发送给 Codex`，不能创建 thread，不能创建 candidate。

## 2. 官方协议依据、术语与版本兼容门槛

Codex App Server 是 Codex 用于深度产品集成的 JSON-RPC interface。官方要求每条连接先 `initialize` 再发送 `initialized`；之后可 `thread/start` 新建、`thread/resume` 续发、以 `turn/start` 执行工作，并从 stdout 读取流式事件。`turn/start` 可携带 thread、用户输入、cwd 与 sandbox 等运行覆盖。官方还提供 `generate-json-schema` / `generate-ts`，供集成方按实际 runtime 生成类型。 [Codex App Server 官方文档](https://developers.openai.com/codex/app-server/)

因此 Host 必须使用本机的真实 schema，而不是相信本文某个 JSON 示例永远不变。

### 2.1 必须区分的三个概念

| 名称 | 本版含义 | 是否可用 |
|---|---|---|
| Codex Mac App | 用户此刻正在使用的桌面产品。它随安装包携带一个官方 Codex runtime，并管理其登录态。 | **本版的前提** |
| App bundle runtime | `ChatGPT.app/Contents/Resources/codex`；由 HTML Genius Native Host 启动 `app-server`，作为 bridge-owned task 的执行进程。 | **本版唯一允许的执行入口** |
| PATH 中的 `codex` | 可能不存在，也可能是同名第三方软件；不能作为本版发现或执行入口。 | **禁止使用** |

不要把“使用 App runtime 新建一个 bridge-owned task”误写成“连接/注入当前 App 窗口里的 task”。当前 App 自己运行的 App Server 是私有 stdio 管道；没有公开、稳定的 API 让第三方附着其已有 thread。HTML Genius 也不应尝试读取 App 的 session 文件、SQLite 或 IPC socket。

### 2.2 App runtime 发现与信任（强制）

仅支持 macOS。Native Host 必须发现 `CFBundleIdentifier = com.openai.codex` 的已安装 App bundle，并推导可执行文件：

```text
<Codex.app>/Contents/Resources/codex
```

首发可检查 `/Applications/ChatGPT.app` 和 `~/Applications/ChatGPT.app`；若需要支持改名或其它安装路径，使用 LaunchServices / `mdfind` 按 bundle identifier 查找，**不能**搜索 `PATH`。对每个候选必须同时验证：

1. `Contents/Info.plist` 的 `CFBundleIdentifier` 精确为 `com.openai.codex`；
2. `Contents/Resources/codex` 是 regular executable file；
3. `codesign -dv --verbose=2 <runtime>` 成功，且 TeamIdentifier 为 `2DC432GLL2`；
4. `<runtime> --version` 成功，并呈现 Codex runtime 版本；
5. 不把版本号、App 安装路径、codesign 原文或用户主目录展示给 Side Panel，仅保存在 Host 的诊断 metadata 中。

任何一步失败均为 `CODEX_APP_NOT_FOUND` 或 `CODEX_APP_UNTRUSTED`。UI 文案：`未找到可用的 Codex Mac App。请安装或打开 Codex App 后重试。` 不得改用 npm 包、Homebrew 包或用户 PATH 中名字恰好为 `codex` 的程序。

### 启动前兼容检查（强制）

每次 Native Host 冷启动，或检测到已发现 App runtime 的版本 / code signature / App path 改变时：

1. `spawn(resolvedAppRuntime, ["app-server", "generate-json-schema", "--out", tempDir], { shell: false })`；
2. 解析生成的 schema，确认至少存在 `initialize`、`thread/start`、`thread/resume`、`turn/start` 和 turn terminal event；
3. 启动一次只读 `resolvedAppRuntime app-server --stdio`，完成 `initialize` → `initialized` handshake 后退出；
4. 确认 schema 可表达：每 turn 的 `cwd`、受限 sandbox / workspace-write、非交互 approval policy（若 schema 采用不同字段名，做有类型的 mapping，不可猜字符串）；
5. 将 runtime version、App bundle version、signature identity 和 schema hash 缓存为非敏感 metadata。

任一步失败，provider 状态为 `incompatible`。UI 提示“Codex App 的本地运行环境不可用或版本不兼容，请更新或重新登录后重试”；不得 fallback 到 npm 同名 CLI、外部 HTTP、App 私有 IPC / SQLite，或复制一段看似成功的文本。

## 4. 架构：provider adapter，而不是第二套 Bridge

### 4.1 组件职责

| 组件 | v0.8 新职责 | 不得承担 |
|---|---|---|
| Side Panel | 选择 Codex、新建/续发 bridge-owned Codex task、显示摘要状态 | 直接启动进程、读文件、显示完整 Agent 对话 |
| Background | 校验 tab artifact、provider-scoped run/session、路由 Native Host completion | 信任 Side Panel 带来的 path/hash/thread ID |
| Native Host | 发现并验证 App bundle runtime；启动 App Server、JSON-RPC、schema compatibility、Codex adapter、复用 candidate protocol | 接收任意方法 / 访问外部 thread / 接触 App 私有 IPC 或 SQLite |
| `codex-app-server-client` | request ID、initialize、事件解析、超时和正常退出 | 业务文件移动或 Chrome 通信 |
| `codex-adapter` | 只允许的 RPC 序列和 prompt 组装 | source/candidate 完整性裁决 |
| 既有 candidate manager | snapshot、hash、manifest、sibling candidate、`candidate-ready` | provider 特有 RPC |

### 4.2 预期新增文件

以 Preflight 找到的真实结构为准，预期为：

```text
bridge/
  codex-app-server-client.mjs     # stdio JSON-RPC + schema compatibility
  codex-adapter.mjs               # provider-specific allowed workflow
  test/codex-app-server-client.test.mjs
  test/codex-adapter.test.mjs
  test/fake-codex-app-server.mjs
```

不得复制 `host.mjs`、Native Messaging framing、candidate manifest 或 artifact completion。能复用的接口必须抽到 provider-neutral 模块；不可为了“抽象漂亮”重构无关 Claude 代码。

## 5. Provider-scoped 数据与 ownership

沿用已有 bridge storage；若已存在通用 `bridge_sessions` / `bridge_runs`，只增加 provider 可区分字段，不升级到新 store。若尚未存在，按 Night Pack A 的最小 metadata 原则新增。

### Session record

```js
{
  logical_document_id: "hgd_...",
  provider: "codex_app_server",
  ownership: "htmlgenius",
  thread_id: "thr_...",
  cwd_root: "/.../.htmlgenius-bridge/codex/<logicalId>",
  last_status: "completed" | "failed" | "cancelled",
  created_at: "ISO-8601",
  updated_at: "ISO-8601"
}
```

只保存 Host 从成功 `thread/start` response 获得的 `thread_id`。没有“输入 thread ID”UI，也没有根据 ID 查询任意会话的 API。

### Run record

```js
{
  run_id: "hgr_...",
  provider: "codex_app_server",
  logical_document_id: "hgd_...",
  tab_id: 123,
  base_artifact_hash: "sha256:...",
  task_sha256: "sha256:...",
  session_mode: "new" | "continue",
  thread_id: "thr_..." | null,
  turn_id: "turn_..." | null,
  status: "starting" | "running" | "completed" | "failed" | "conflict",
  error_code: null,
  candidate_uri: null,
  created_at: "ISO-8601",
  completed_at: null
}
```

不存完整 Change Contract、Prompt、candidate HTML、stdout、Agent message、思维链或任意 credential。完整任务只继续存在受权限保护的本地 task bundle 中。

同一个 tab 同时最多一个 `starting/running` run，**不论 provider**；但每个 provider 对同一 logical document 各自有独立 session record。

## 6. Codex App Server 的受控运行序列

### 6.1 Host → App Server 固定序列

允许的 RPC 顺序仅为：

```text
spawn resolvedAppRuntime app-server --stdio (shell:false)
  → initialize (request)
  ← result
  → initialized (notification)
  → thread/start OR thread/resume(stored bridge-owned id)
  ← thread
  → turn/start(threadId, candidate workspace cwd, controlled input/config)
  ← turn-start result + events
  ← terminal turn event
  → stdin close / graceful child exit
```

禁止 client 发 `thread/list`、`thread/read`、`thread/fork`、`turn/steer`、`thread/inject_items` 或 schema 中未来可能出现的任意其它方法。client 的 public API 应只有：

```js
checkCompatibility()
runCandidate({ sessionMode, storedThreadId, workspace, task })
```

### 6.2 新建与续发

- `new`：只调用 `thread/start`，成功拿到 thread ID 后立即上报 Background 但标为 `starting`；只有 turn terminal 后更新为 terminal。
- `continue`：Background 必须先查当前 `logical_document_id + provider` 的 record，验证 ownership、thread ID、last status terminal、cwd root 匹配；Host 才能调用 `thread/resume`。
- resume 失败（thread 不存在、cwd 无法恢复、App Server 拒绝）时，返回 `CODEX_SESSION_UNAVAILABLE`；不得静默退化到 `thread/start`，因为用户选择的是“继续”。
- 不支持从用户已有 Codex UI 里选择历史 session；这是明确的产品边界，不是首发缺失功能。

### 6.3 turn 运行配置

App Server 的 JSON schema 是唯一字段来源。通过 schema mapper 创建 `turn/start` params，必须满足：

1. `cwd` 为本 run 的候选 workspace，例如 `.htmlgenius-bridge/codex/<logicalId>/runs/<runId>`，绝不是真实 source path 或 source parent。
2. sandbox 只允许该 workspace 的读写；禁网络；不允许扩大到 Home、source parent 或其它工作目录。
3. approval policy 为非交互、fail-closed：不能由 Agent 在 Side Panel 外弹出未被 HTML Genius 处理的确认。
4. 不挂接用户 MCP、插件、hook、Skill 或额外目录；schema 若不支持明确关闭这些能力，标记 incompatible。
5. 只传 Host 固定生成的任务 prompt 和当前 Change Contract；用户评论不得成为 JSON-RPC method、cwd、sandbox 参数或其它控制字段。
6. Host 设 180 秒 hard timeout（可配置常量）；超时杀掉 child，标记 `CODEX_TIMED_OUT`，不采用 candidate。

不要使用 `dangerously-bypass-approvals-and-sandbox` 或等价绕过选项。

### 6.4 Prompt 前言（固定、不可由用户覆盖）

```text
你正在 HTML Genius 的受控候选工作区中执行任务。

- 只读取当前目录中的 source.html 以及 task bundle。
- 只将最终、完整、可直接打开的 HTML 写入当前目录的 candidate.html。
- 不要修改 source.html、task bundle、其它文件；不要使用网络、MCP、外部插件或其它工作目录。
- 严格遵守 Change Contract。无法唯一定位目标时，保留该内容并在最终消息中说明。
- 不要以 Markdown、diff、解释文本或多个文件替代 candidate.html。
```

固定前言在 `ChangeContract.renderPrompt(task)` 之前拼接；task 本身不能删除或重写前言。Prompt 仅是协作说明，可信边界仍是 sandbox、workspace 隔离、host hash 和 candidate validator。

## 7. Candidate 与 artifact 复用规则

v0.8 不定义新 candidate 格式。它使用 Night Pack A 的：

- `source.html` snapshot；
- `candidate.html`；
- source SHA-256 before / after；
- candidate HTML validator；
- `candidate-manifest.json`；
- source 同级 `...candidate.html` 命名；
- `candidate-ready` → Background double-check → content-script `artifact-update-ready` (`new_artifact`)；
- logical document link、打开 candidate、annotation re-anchor。

唯一 provider 差异是 manifest 的：

```json
{
  "provider": "codex_app_server",
  "session": { "id": "thr_...", "ownership": "htmlgenius" }
}
```

Host 必须在 terminal turn 后验证：真实 source hash 未变化、candidate 存在且有效、candidate 不是 symlink、同名 result 不会覆盖，才发 `candidate-ready`。任一失败都不得向 content script 发 `new_artifact`。

## 9. 失败码（稳定、可测试）

| Code | 条件 | 用户动作 |
|---|---|---|
| `CODEX_APP_NOT_FOUND` | 找不到受支持的 Codex Mac App bundle/runtime | 安装或打开 Codex Mac App 后重试 |
| `CODEX_APP_UNTRUSTED` | bundle identifier、runtime 路径或 OpenAI code signature 校验失败 | 从官方渠道重新安装/更新 Codex Mac App |
| `CODEX_INCOMPATIBLE` | App runtime 的 schema/handshake/required sandbox 能力缺失 | 更新 Codex Mac App |
| `CODEX_AUTH_REQUIRED` | App runtime 返回认证未就绪 | 在 Codex Mac App 登录后重试 |
| `CODEX_SESSION_UNAVAILABLE` | bridge-owned thread 无法 resume | 创建新的 Codex task |
| `CODEX_TURN_FAILED` | terminal event 为失败/中断 | 查看简短错误，重试新 task |
| `CODEX_TIMED_OUT` | 超过 180 秒 | 重新发起，或改用 Claude Code |
| `SOURCE_CHANGED_BEFORE_START` | source base hash 已不同 | 重新加载文件后重试 |
| `SOURCE_MUTATED_DURING_CANDIDATE` | run 后 source hash 改变 | 保留 source，重新核对后重试 |
| `CANDIDATE_INVALID` | candidate 缺失/非 HTML/symlink/超限 | 重试或复制 Prompt |
| `COMPLETION_REJECTED` | Background 的 run/task/hash/URI 比对失败 | 不导航，记录诊断 metadata |

日志只写 stderr；不得把完整 prompt、HTML、评论、Codex agent message 或 credential 放入 Native Messaging stdout、Chrome console、IndexedDB、manifest 或 error message。

## 10. 测试与验收

### 自动测试（必须）

新建 fake App Server，严格模拟 stdio JSON-RPC，并断言：

1. 任何 request 前必须先 `initialize` → `initialized`；client 未完成 handshake 不会发 thread/turn。
2. `new` 只发 `thread/start`；成功获得的 thread ID 才能保存为 `ownership: htmlgenius`。
3. `continue` 只对已记录的 provider-scoped ID 发 `thread/resume`；不存在/非 terminal/错误 provider 直接拒绝，绝不调用 list/read。
4. client 永远不发 forbidden method（list/read/fork/steer/inject）。
5. `turn/start` 的 cwd 是 candidate workspace；真实 source path 不出现于 RPC params 或 prompt 路径指令。
6. schema 缺 cwd、受限 sandbox、非交互 approval 任一能力时返回 `CODEX_INCOMPATIBLE`。
7. timeout、非 JSON、未知 response ID、stdout 噪声、child exit、terminal failure 都清理 run、不保存成功 session。
8. source hash 变化、candidate 缺失/无效时不产生 `candidate-ready` / `new_artifact`。
9. Background 拒绝 tab、logical document、task hash、base hash、candidate URI 任一不匹配的 completion。
10. Claude adapter 的现有 test 完整通过；provider 间 session 不能互相显示为“继续”。

### 手工 smoke（必须记录结果）

1. 在本机确认解析出的 `<Codex.app>/Contents/Resources/codex app-server generate-json-schema --out <temp>` 成功，且 Codex Mac App 已登录。
2. 打开 managed local HTML，添加至少 3 条评论，进入任务选择，仅选 2 条。
3. 选 `只改圈选处` → Codex → 新建 task；确认 source 文件 byte hash 不变、同级生成 candidate、未选评论不在 task bundle。
4. candidate 打开后，检查评论 open/stale 统计和相对资源。
5. 同一逻辑文档再次选择 Codex 的“继续上次 task”；只允许已记录 Codex thread，不展示任何其它 Codex 对话。
6. 模拟外部修改 source、杀掉 App Server、删除 candidate、退出 Codex 登录，各自得到正确失败状态且不导航。

真实 smoke 会使用用户自身 Codex 额度；自动测试不得调用真实 Codex。

## 11. 交付与停止条件

### 允许修改

```text
bridge/（仅 provider adapter、JSON-RPC client、测试、依赖声明）
extension/sidepanel.{html,css,js}
extension/background.js
extension/i18n.js
必要的 provider/session metadata 访问层
docs/CURRENT_IMPLEMENTATION_STATE.md
docs/implementation-notes/2026-07-21-v0.8-codex-preflight.md
```

禁止无关重构 `storage.js`、远端协同、账号/登录、评论语义或 artifact schema。若现有 metadata 无法表达 provider-scoped session，先报告最小 migration；不能在夜间自行扩大数据库重构。

### 完成后必须交付

1. 全部测试命令及结果；`git diff --check`。
2. Preflight/reconciliation 报告，标出每处 Spec 与真实代码的等价适配。
3. 更新 `CURRENT_IMPLEMENTATION_STATE.md`：将 v0.8 状态写成真实已完成/未完成，不追加第二份结论。
4. 不超过 30 行交接：完成项、未完成项、真实 smoke 结果、风险与下一步。

### 必须停止而非猜测的情况

- App Server schema / handshake 无法满足受控 cwd、sandbox、approval 的组合；
- Night Pack A candidate 约束尚不存在；
- 实现需要读取任意 Codex thread 才能实现“继续”；
- 实现需要让 Codex 访问真实 source directory 或使用 bypass sandbox；
- 任何 source hash 校验、candidate validator 或 `new_artifact` completion 失败。

下一版本才讨论 review / promote 或 Qoder / CodeBuddy 等 provider。v0.8 的价值是验证第二个本地 Agent adapter 能复用同一安全闭环，而不是扩大成通用 Agent 平台。
