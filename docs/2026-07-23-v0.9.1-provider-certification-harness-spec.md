# HTML Genius V0.9.1：Provider Certification Harness & Automated Verification

> **施工目标：** 将 V0.9 的 Bridge 接入验证收敛为可重复执行的自动化验证体系，并建立新增 Agent/provider 必须遵循的认证契约。以后接入第 4、第 5 个 Agent 时，施工 Agent 只要实现 adapter 还不够；必须提交 provider descriptor、fake runtime、能力矩阵和通过同一套 harness 的证据。  
> **版本基线：** V0.9 已具备 `bridge-health.mjs`、`bridge-install.mjs`、`bin/htmlgenius-bridge.mjs`、`bridge_health` Native Host 消息及对应 Node 测试；当前 provider 为 Claude Code、Codex App Server、GitHub Copilot。  
> **目标平台：** macOS + Chrome。Windows/Linux、DMG/PKG、真实 Agent 安装/登录不属于 V0.9.1。

---

## 0. 结论与范围

V0.9.1 不是再写一份人工验收清单，也不是只给当前三家 Agent 增加 mock。它要建立三道可持续复用的门：

```text
Provider descriptor（声明能力与前提）
        ↓
Fake runtime contract（可控、无账号、可制造失败）
        ↓
Certification Harness（统一断言 Bridge / 安全 / UI 状态 / artifact 闭环）
        ↓
Opt-in real smoke（真实账号，仅发布前或专用测试机运行）
```

目标是让“新增 provider”成为一项有明确交付物、稳定失败码和自动回归门禁的工程工作，而不是在多个 if/else 中补分支、靠人工试一次。

### 0.1 本版本做什么

- 将 V0.9 已有 CLI/health/install/Native Host 测试组织为明确命令和可机读报告。
- 建立唯一的 provider descriptor registry，减少 `background.js`、`host.mjs`、`provider-probe.mjs`、Side panel 各自硬编码 provider ID 的漂移。
- 为每个 provider 定义 fake runtime fixture 和必须覆盖的行为场景。
- 提供 provider-neutral certification suite：probe、health 映射、candidate、plan（如声明支持）、安全隔离、失败不发布 artifact、无 session 泄露。
- 提供 opt-in 的真实 Bridge/Chrome/provider smoke 框架；默认 CI 与 `npm test` 不需要账号、不访问网络。
- 产出脱敏 `verification-report.json` 和可读摘要，供发布判断与后续新增 provider 比较。

### 0.2 本版本明确不做

- 不把真实 Claude/Codex/Copilot 登录、订阅、OAuth 纳入 CI；不存 token、Cookie、session ID。
- 不承诺 GitHub-hosted CI 可验证 macOS Chrome Native Messaging；真实 Chrome smoke 应在受控的 macOS 测试机/专用本地用户运行。
- 不自动下载或执行 DMG/PKG，不安装 Node/Agent，不打开 Terminal。
- 不重做 Change Contract、评论、范围选择、candidate UI 或 Diff/Promote。
- 不接入新的 provider；只为未来接入建立框架，并对现有三家补齐认证。

## 1. 当前代码基线与不可破坏项

施工前必须确认以下事实；当前仓库已经有 V0.9 实现，不能倒退到“从零写验证脚本”的错误方向。

| 已有能力 | 入口 | V0.9.1 的用法 |
|---|---|---|
| CLI | `bridge/bin/htmlgenius-bridge.mjs` | 作为 bootstrap/doctor/setup/repair/uninstall 的真实可执行被测对象。 |
| 安装核心 | `bridge/bridge-install.mjs` | 继续以临时 HOME/hosts-dir 隔离测试；不改真实用户目录。 |
| health 契约 | `bridge/bridge-health.mjs` | 作为 provider status → reason code/remediation 的唯一映射层。 |
| Native Host | `bridge/host.mjs` + `native-protocol.mjs` | 使用 framed stdin/stdout 子进程做协议集成测试。 |
| provider probe | `bridge/provider-probe.mjs` | 保持每 provider 独立失败域；harness 从此处或其 descriptor 适配层调用。 |
| 现有 fakes | `bridge/test/fake-claude.mjs`、`fake-claude-bin/`、`fake-codex-app-server.mjs`、`fake-copilot-sdk.mjs` | 迁入/包装为 provider fixture，不删除已有深度测试。 |
| 当前 Node test | `bridge/package.json` 的 `node --test test/*.test.mjs` | 保持无账号、无网络、全绿；新增 tests 必须能并行运行。 |

不可破坏：

- 现有 `com.htmlgenius.local_bridge` host 名、单 origin、受管目录、source 永不覆盖、candidate/plan 验证与 Copilot runtime 锁定。
- 现有三个 provider ID：`claude_code_cli`、`codex_app_server`、`github_copilot`。
- provider probe 不创建 Agent session/thread/candidate。
- Native Host stdout 只输出长度前缀帧；日志只能到 stderr。
- 所有自动化测试不得访问真实用户 HOME、真实 Chrome host 目录、真实 Agent 配置、网络或认证信息。

## 2. 验证金字塔与命令

### 2.1 四层验证

| 层级 | 名称 | 默认运行 | 是否真实账号/网络 | 证明什么 |
|---|---|---:|---:|---|
| L0 | Unit & contract | 是 | 否 | 纯函数、数据形状、脱敏、状态映射、参数校验。 |
| L1 | Isolated Bridge integration | 是 | 否 | CLI 安装/修复、Native framing、host health、task/artifact 安全闭环。 |
| L2 | Provider certification (fake runtime) | 是 | 否 | 每个 provider 的 probe/run/失败/安全语义满足共同契约。 |
| L3 | Real smoke | 否，显式 opt-in | 是 | 真实 macOS + Chrome + 已登录 provider 能完成最小闭环。 |

### 2.2 必需命令

在 `bridge/package.json` 增加以下 scripts（名称可微调，但语义和分层不可省略）：

```json
{
  "scripts": {
    "test": "node --test test/*.test.mjs",
    "verify:bootstrap": "node verify/bootstrap-verify.mjs --report verification-report.json",
    "verify:providers": "node verify/provider-certify.mjs --all --report verification-report.json",
    "verify": "npm run test && npm run verify:bootstrap && npm run verify:providers",
    "smoke:local": "node verify/real-smoke.mjs",
    "smoke:provider": "node verify/real-smoke.mjs --provider"
  }
}
```

约束：

- `npm run verify` 是施工完成的最低自动化门，不依赖 Chrome、账号和网络；失败退出非 0。
- `verify:bootstrap` 只使用 `mkdtemp` 创建的 HOME/host 目录，以及 CLI 内部测试注入环境变量；绝不碰 `~/Library/...`。
- `verify:providers --all` 只使用 fakes；可用 `--provider <id>` 认证某一 provider。
- `smoke:*` 默认必须拒绝运行并提示设置显式确认环境变量，例如 `HTMLGENIUS_ALLOW_REAL_SMOKE=1`；避免误在开发者日常账户上跑真实 Agent。
- 任何命令可写 `--report <path>`，但 report 必须通过 §7 脱敏器；默认输出目录应在临时目录或 gitignore 的 `artifacts/verification/`。

不要引入一套第二测试框架。优先 Node 内建 `node:test`、现有 fake 与 helper；只有在确实实现 Extension DOM 自动化时才引入 Playwright，且其安装/浏览器下载必须是可选 dev dependency，不能成为基本 `npm test` 的前置条件。

## 3. Provider Descriptor Registry（未来扩展的核心）

### 3.1 为什么必须有 registry

当前 provider ID 和能力散落在 background、host、probe、adapter、sidepanel/i18n 中。继续这样加第 4 家 Agent，最常见结果是：能发任务但 health 状态缺失，或菜单显示 ready 但 adapter 不支持 plan。V0.9.1 必须建立单一的生产 descriptor registry。

新增 `bridge/provider-registry.mjs`（生产代码，不能放在 `test/`），其输出是不可变 allow-list。建议形状：

```js
export const PROVIDER_REGISTRY = Object.freeze({
  claude_code_cli: {
    id: "claude_code_cli",
    label_key: "provider.claude",
    capabilities: ["candidate", "plan"],
    dispatch_type: "claude_handoff_start",
    probe: "claude",
    runtime_policy: "provider_default",
    supports_real_smoke: true
  },
  codex_app_server: {
    id: "codex_app_server",
    label_key: "provider.codex",
    capabilities: ["candidate", "plan"],
    dispatch_type: "codex_handoff_start",
    probe: "codex",
    runtime_policy: "signed_app_only",
    supports_real_smoke: true
  },
  github_copilot: {
    id: "github_copilot",
    label_key: "provider.copilot",
    capabilities: ["candidate", "plan"],
    dispatch_type: "copilot_handoff_start",
    probe: "copilot",
    runtime_policy: "runtime_locked",
    supports_real_smoke: true
  }
});
```

必须提供：

```js
listProviderIds();
getProviderDescriptor(id);          // 不在 allow-list → null
assertProviderDescriptor(value);    // 启动/测试期严格校验
providerSupports(id, capability);  // candidate / plan
```

规则：

- descriptor 不含路径、命令、token、登录态、用户数据、动态版本或任意可执行函数。
- 生产运行时只相信 registry 的静态能力；不能让 provider probe 自报能力后扩大权限。
- 将 `background.js` 的 `SUPPORTED_PROVIDERS`、handoff type 映射以及 `host.mjs` 分发 allow-list 收敛至 registry 或由 registry 派生。若 adapter 的复杂执行函数不能直接放入 registry，可保留显式 adapter table，但必须有 registry 对齐测试。
- extension UI 不直接 import Node 模块；由 background 将脱敏后的 provider descriptor 子集传给 Side panel，或在 extension 内拥有同源的只读 provider metadata。两者必须有测试确保 ID、label key、capability 一致。

### 3.2 新 provider 的强制交付物

未来任何 provider PR/施工包都必须包含：

1. 生产 descriptor；
2. probe adapter 及 reason-code 映射；
3. candidate adapter；若声明 `plan`，还必须有 plan adapter；
4. provider fake runtime fixture（§4）；
5. capability/health/安全认证测试（§5）；
6. i18n 三语言 label 与 remediation 文案；
7. `docs/providers/<id>.md`，说明官方前提、用户手动登录方式、支持能力、已知限制；
8. 可选但强烈建议：真实 smoke 执行说明。未做真实 smoke 时必须在实现状态中标“mock 认证已过，真实 smoke 未验证”。

缺任何一项，该 provider 只能停留在实验分支，不能出现在正式发送菜单。

## 4. Fake Runtime Contract

### 4.1 Fixture 目录与接口

新增目录：

```text
bridge/test/providers/
  provider-fixture-contract.mjs
  claude-code.fixture.mjs
  codex-app-server.fixture.mjs
  github-copilot.fixture.mjs
```

可复用现有 fake 文件，不要求重写其协议模拟；fixture 是给通用 harness 的标准包装层。

每个 fixture 至少导出：

```js
export const fixture = {
  provider: "<registry id>",
  capabilities: ["candidate", "plan"],
  scenarios: [
    "ready",
    "not_installed",
    "auth_required",
    "incompatible",
    "probe_error",
    "candidate_success",
    "candidate_missing",
    "candidate_out_of_scope",
    "source_mutated",
    "plan_success",
    "plan_invalid",
    "runtime_changed" // 仅 runtime_policy 要求锁定的 provider
  ],
  makeProbeScenario(name),
  makeRunScenario(name, context),
  cleanup(context),
  expected: {
    no_session_persisted: true,
    candidate_filename: "candidate.html",
    plan_filename: "output/plan.json"
  }
};
```

接口可按现有 adapter 注入点做小幅调整，但以下语义不可变：

- fixture 不得调用真实 executable、App Server、SDK 网络或读取 `$HOME`。
- fixture 能让通用 harness 精确制造 probe 状态与 run 结果；不允许通过“跳过此场景”来逃避认证。
- `candidate_success` 必须产生可验证 HTML；`plan_success` 必须产生符合当前 plan schema 的 JSON。
- `candidate_missing`、`candidate_out_of_scope`、`source_mutated`、`plan_invalid` 都必须失败，且不产生可发布 sibling artifact。
- adapter 若不支持 plan，则 descriptor 不声明 `plan`，通用 suite 断言它不会暴露 Plan UI/dispatch；不能写一个空实现假装支持。

### 4.2 通用 fixture contract test

新增 `bridge/test/provider-fixture-contract.test.mjs`，对 registry 中每个 provider fixture 循环断言：

- fixture provider 与 registry ID 一致，能力集合完全一致；
- 所有必需 probe 场景存在；
- 声明 `candidate` 必有四种 candidate 成功/失败场景；
- 声明 `plan` 必有成功和 invalid 场景；未声明则不得有可执行 plan success；
- 需要 runtime lock 的 provider 必有 `runtime_changed`；
- fake 返回和 report 不含敏感 key（§7）。

这条测试是未来“忘记给新 Agent 写 fake”的第一道硬门。

## 5. Certification Harness 的统一断言

新增 `bridge/verify/provider-certify.mjs`，不需要 Chrome，不需要真实账号。它读取 registry，再依次载入 fixture 执行以下矩阵；每项产出 pass/fail/skipped 与 machine reason，不因一个 provider 崩溃而中断其他 provider。

### 5.1 Probe 与 health 矩阵

对每个 provider：

| 场景 | 必须断言 |
|---|---|
| `ready` | `status=ready`；capabilities 等于 registry；health 无 reason code；可在发送菜单选择。 |
| `not_installed` | 非 ready；provider 专属稳定 `reason_code`；有安全的人工 remediation；不显示原始错误。 |
| `auth_required` | 非 ready；不触发登录；显示“在本机完成登录”而非自动登录。 |
| `incompatible` | 非 ready；不得 fallback 到未声明 runtime 或更高权限模式。 |
| `probe_error` | 只使该 provider 为 error；其他 provider probe 仍完整；无 stderr/path 泄露。 |
| `blocked`（如适用） | 显示 policy/环境阻断，不建议绕过企业策略。 |

### 5.2 Candidate 矩阵

每个支持 `candidate` 的 provider 必须通过：

| 场景 | 必须断言 |
|---|---|
| `candidate_success` | 仅 workspace `candidate.html` 可写；validator 通过；source hash 未变；只发布版本化 sibling；ready 事件 provider/task hash/run ID 对齐。 |
| `candidate_missing` | `CANDIDATE_MISSING` 或对应稳定失败码；不发布 sibling；不标 completed。 |
| `candidate_out_of_scope` | validator/围栏拒绝；不写 source；不发布 sibling。 |
| `source_mutated` | hash 不匹配；candidate 废弃；不发布 sibling。 |
| 安全注入 | prompt/文件名中的 shell 特殊字符不改变 argv 或写出允许路径。 |

### 5.3 Plan 矩阵

仅对 `capabilities` 声明 `plan` 的 provider 执行：

| 场景 | 必须断言 |
|---|---|
| `plan_success` | 仅写 `output/plan.json`；schema/hash/manifest 通过；不产生 candidate 或 sibling；plan-ready 带 provider 和必要 runtime metadata。 |
| `plan_invalid` | 拒绝；不产生 plan-ready/candidate/sibling。 |
| Plan → Candidate | 使用相同 provider；有 runtime lock 策略时锁定同一 runtime，不可静默 fallback。 |
| runtime changed（如适用） | 返回稳定 `RUNTIME_CHANGED` 类错误；要求重建 Plan。 |

### 5.4 不变量（所有 provider）

每一次 certification 后统一扫描：

- source 字节 hash、HTML 内容、mtime（在允许的文件系统误差下）没有被 adapter 改写；
- 没有 source 之外的新增 HTML 或任意项目文件；
- provider 无权读取/写入另一个 provider workspace；
- 不持久化 session/thread ID；Copilot 继续保持不读 session 的 V0.8.2 边界；
- 事件、health、report、IndexedDB mock 不含绝对路径、token、cookie、stderr、完整 prompt/comment、session/thread；
- fixture cleanup 完成后临时目录删除；并行运行不共享 mode/log 文件。

任何一条不变量失败，即使 candidate 看起来生成成功，也必须使 `verify:providers` 失败。

## 6. Bootstrap / Native Host 自动验证

新增 `bridge/verify/bootstrap-verify.mjs`。它应优先复用现有 CLI tests，形成一个完整、可独立运行的产品级验收序列，而不是复制测试实现。

在临时 HOME 与临时 hosts dir 下自动执行：

1. `doctor --json --extension-id <valid>` → `BRIDGE_NOT_INSTALLED`，且不创建任何目录；
2. `setup --json --scope user --extension-id <A>` → `changed:true`；
3. 重复 setup → `changed:false`；manifest 仅一个 origin，launcher 为 0700，且不指向 npx 临时 cache；
4. `doctor` → bridge ready / origin OK；
5. 换 extension ID `<B>` → `EXTENSION_ORIGIN_MISMATCH`，原 manifest 原封不动；
6. 破坏/删除受管 bridge 文件 → `doctor` 标为 repair required；
7. 执行 `repair` → 仅恢复 HTML Genius 受管文件；
8. 启动 `host.mjs` 子进程，以真实 Native Messaging frame 发送 `bridge_health`、允许的 `bridge_repair`、拒绝的 repair；验证 stdout 只含合法帧；
9. `uninstall` → 仅删除带 HTML Genius marker 的 manifest/launcher/managed install，保留 workspace 审计证据和假第三方 host；
10. 逐步扫描 report 与 stdout/stderr，确保没有 temp path、真实 HOME、敏感键。

这不是“真实 Chrome E2E”，但它自动验证了 Chrome 真正依赖的 manifest/launcher/Native framing 边界。不要谎称它覆盖了 macOS Gatekeeper 或 Chrome 实际 profile 发现。

## 7. 脱敏报告与可观测性

### 7.1 Report schema

所有 `verify:*` 共用 `verification-report.schema_version = 1`：

```json
{
  "schema_version": 1,
  "kind": "bootstrap | provider_certification | real_smoke",
  "started_at": "2026-07-23T00:00:00.000Z",
  "finished_at": "2026-07-23T00:00:00.000Z",
  "result": "passed | failed | blocked | skipped",
  "bridge": { "version": "0.9.1", "protocol_version": 1 },
  "providers": [
    {
      "id": "github_copilot",
      "result": "passed",
      "capabilities": ["candidate", "plan"],
      "checks": [
        { "id": "probe.ready", "result": "passed", "reason_code": null }
      ]
    }
  ],
  "summary": { "passed": 24, "failed": 0, "skipped": 0 }
}
```

### 7.2 Report 安全

- 不得含 `path`、`command`、`argv`、`stdout`、`stderr`、`token`、`cookie`、`session`、`thread`、`prompt`、`comment`、`html`、`stack`、`env`、真实用户名或机器名。
- 共用/提炼 V0.9 `sanitizeHealth` 的策略，但不能直接假定 health sanitizer 足够；新增 `sanitizeVerificationReport`，对嵌套键递归剥离并有单测。
- report 只记录稳定 check ID、provider ID、result、reason code、版本、耗时聚合；可用于用户复制诊断和后续 CI 归档。
- `smoke:real` 的报告同样脱敏，真实 Agent 的原始输出只能写入临时、默认删除的本机调试文件，且不提交/不上传。

## 8. Real Smoke（非默认，必须诚实标注）

### 8.1 目的

fake certification 证明我们没有破坏自己的协议；它不能证明真实 Agent 某个新版本的认证、CLI/App Server 协议、订阅权益或企业策略仍可用。因此需要一个可重复但显式 opt-in 的 L3 smoke。

### 8.2 运行保护

`npm run smoke:local` / `npm run smoke:provider -- --provider <id>` 必须先检查：

```text
HTMLGENIUS_ALLOW_REAL_SMOKE=1
HTMLGENIUS_SMOKE_WORKSPACE=<绝对临时测试目录>
```

任一缺失则退出非 0，且不连接 provider。workspace 必须是新建目录，拒绝项目根、HOME、Desktop、Documents 及 source repo 内路径。`--provider` 只接受 registry ID。

### 8.3 分两种 smoke，不能混称

1. **Bridge/provider smoke（本版本必须实现）**：在指定临时工作区创建固定无敏感 fixture HTML 与 Change Contract，经真实已登录 provider 生成 candidate；验证 source 未变、candidate 成功、无越界 artifact、再清理。它不依赖真实 Chrome UI。
2. **Chrome Native Messaging smoke（本版本提供 runner 骨架 + 明确 manual gate）**：使用专用 macOS 测试用户、已加载固定 extension ID 的 Chrome，验证 `connectNative → bridge_health → provider_probe`。因为 Chrome 的 Native Host 查找路径和浏览器 profile/系统权限不可在通用 CI 中可靠伪造，runner 应输出结构化“准备完成/需人工点击确认”步骤，不得声称无人值守 E2E 已覆盖。

真实 smoke 失败必须报告 `blocked` / `failed` 与稳定 reason code，不得把 provider stderr 放进报告。真实 smoke 成功也不能自动提升 provider 为正式支持；发布前仍需人工确认版本与账号环境。

## 9. Extension UI 验证

V0.9.1 不要求重写 Side panel，但必须补齐连接中心的可自动验证部分。

### 9.1 最低要求（不新增浏览器依赖）

- 将 Connection Center 的状态选择、bootstrap 文本生成、report 脱敏、按钮可用性抽为纯函数模块（建议 `extension/connection-center-state.js`）。
- 用 `node:test` 运行这些纯函数，覆盖 §5.2 状态矩阵：未安装、需修复、OS 不支持、Bridge ready 但无 provider、provider ready。
- 断言未连接态仍有 Copy Prompt；`repair` 不会在 `BRIDGE_NOT_INSTALLED` 态出现；发送菜单只能选择 ready provider；bootstrap 变量不含 Change Contract/页面数据。
- 为中文、英文、日文的新增 key 建一致性测试；没有翻译 key 直接失败，禁止 fallback 空字符串。
- CSS/DOM 源码级测试或轻量渲染测试断言 Connection Center 不设置 `overflow-x: auto/scroll`；不把评论区结构删除或隐藏。

### 9.2 可选 Playwright 增强

只有当施工 Agent 能在不影响默认测试的前提下增加 Playwright 时，实现 `npm run verify:sidepanel`：加载 unpacked extension，以 mock background message 覆盖 health 结果，截图/断言窄宽度下的 Connection Center。

它是增强项，不是本版本 completion 的硬前提。不得为了截图测试引入需要下载浏览器、导致普通 `npm test` 不可用的强依赖。

## 10. 新增 Agent 的标准施工模板

未来每个 Agent Spec 必须直接引用本节，并按下列顺序施工：

1. **声明**：在 registry 加 descriptor，明确 `candidate`/`plan` 能力、runtime policy、官方前提；不要先改 UI。
2. **安全 adapter**：先实现 probe 和受限 candidate workspace；未验证的 API 不可用“默认 shell”绕过。
3. **fixture**：实现 §4 的 fake runtime 和所有必需失败场景。
4. **认证**：运行 `npm run verify:providers -- --provider <id>`；通过才允许连接 Side panel 菜单。
5. **文案与文档**：三语言、remediation、`docs/providers/<id>.md`、已验证/未验证状态。
6. **真实 smoke**：有授权测试环境时运行，若没有，明确标记未验证；不得伪称已 E2E。
7. **回归**：`npm run verify` 全绿；报告以脱敏 JSON 存档或附在 PR/施工回报中。

这套模板的目的不是增加形式主义，而是避免最危险的回归：新 provider 使 Bridge 自动执行未受限 shell、泄露 Agent 登录态、覆盖 source，或因 probe/UI/adapter 能力不一致产生假“已连接”。

## 12. 完成定义与施工回报

### 12.1 Completion Definition

V0.9.1 完成需同时满足：

- `npm run verify` 在无账号、无网络、无真实 Chrome/Agent 的环境全绿；
- 当前三 provider 都有 registry descriptor、fixture，并通过统一 certification；
- bootstrap verifier 覆盖 install → idempotent → origin mismatch → corruption → repair → Native Frame → uninstall；
- report 具有稳定 schema，且有自动测试证明不泄露敏感字段或临时路径；
- real smoke 默认不会误运行；有明确 opt-in、隔离 workspace 与清理策略；
- Side panel Connection Center 的关键状态和三语言 key 有自动化断言；
- 所有既有 V0.8/V0.9 test 仍全绿，source/candidate/plan 的安全边界无回归。

### 12.2 施工 Agent 最终回报格式

必须报告：

1. 新增/修改的 registry、fixture、harness、scripts、文档文件；
2. `npm run test`、`npm run verify:bootstrap`、`npm run verify:providers`、`npm run verify` 的真实结果；
3. 三家 provider 的认证矩阵摘要（fake）及真实 smoke 是否运行；
4. report 的实际样例（脱敏）；
5. 未做的 Chrome 真实手测、真实账号 smoke、发布/CI 环境限制；
6. 任何与本 Spec 不一致的现有代码事实，先说明再请求决策，不能静默偏离。
