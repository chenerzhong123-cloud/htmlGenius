# HTML Genius V0.9：Agent-assisted Local Bridge & Connection Center

> **施工目标：** 把当前「进仓库、`npm install`、复制 Extension ID、执行 macOS 脚本」的开发者安装流程，收敛成普通用户可通过 **已有 Agent** 或 **一条 Terminal 命令** 完成的用户级初始化；Side panel 能解释、复查和引导修复连接。  
> **版本基线：** V0.8.2（当前仓库）；Chrome MV3、macOS、Claude Code / Codex App Server / GitHub Copilot 三个 provider 已存在。  
> **不做：** 本版本不制作 DMG/PKG、不做 Windows/Linux Native Host、不发布 npm 包、不代替用户登录任一 Agent、不读取/接管既有 Agent 会话。

---

## 0. 先读：本版本要解决什么，不要解决什么

### 0.1 当前真实问题

当前 [LOCAL_BRIDGE.md](LOCAL_BRIDGE.md) 要求最终用户进入源码仓库，运行 `npm install`、从 `chrome://extensions` 复制 ID、再运行 `node install-macos.mjs --extension-id …`。这不是面向产品用户的接入路径；而且一旦 Native Host 根本不存在，扩展无法自己在本机执行修复命令。

Chrome Native Messaging 的安全模型决定：扩展只能连接**已注册**的本地 host；它不能自行启动 Terminal、下载后运行程序或注册 host。因此 V0.9 不能承诺「Side panel 一键完成首次安装」。它应把用户明确授权的一次初始化，降到以下两条路径：

1. **首选：Agent-assisted Setup**。用户在 Side panel 点击「让 Agent 帮我连接」，复制一段严格限定的 Setup Prompt，自己粘贴给正在使用的 Claude Code / Codex / Copilot。Agent 只运行 HTML Genius 官方 CLI，不得修改项目文件。
2. **兜底：Terminal Setup**。用户复制一条固定版本的命令到 Terminal。

完成一次初始化后，Side panel 才能调用已注册的 Bridge 执行「检查连接」「安全修复」「重新检测 Agent」。

### 0.2 产品边界（必须保持）

- 本地优先；不新增托管 AI、统一 Token、云端转发或账号系统。
- 每次编辑仍只创建新 task；不得列举、读取、恢复、注入或续发用户已有的 Claude/Codex/Copilot 会话。
- Change Contract、候选文件、Plan/candidate 安全边界均沿用 V0.8.2；本版本不得放宽 Agent 文件权限。
- source HTML 永不覆盖；Bridge 的安装/诊断不能读取用户 HTML、评论全文、prompt、Agent token、Cookie、历史会话、完整命令行、绝对路径或 stderr 并回传扩展。
- 未安装/未连接时始终保留「复制 Prompt」编辑降级路径；不能把用户锁死在安装 Bridge 的流程里。

### 0.3 非目标

- 不实现 DMG/PKG、自动下载安装/静默执行 shell、管理员权限、Homebrew 安装、`curl | sh`。
- 不自动安装 Claude Code、Codex、Copilot 或 Node；不自动跑 OAuth 登录、输入 API key、接受订阅或绕过企业策略。
- 不支持任意 Agent 的“万能连接”。本版本仅接入已存在的三个 provider。
- 不做跨平台安装器；Windows/Linux 仅在界面中准确标为“尚不支持”，不得给出失效命令。
- 不改变评论列表、评论树、编辑契约、范围选择、candidate 回写、Plan review 的产品交互。

## 1. 施工前验收基线

施工 Agent 必须先核对，不得凭旧 Spec 假定文件结构：

| 项目 | 当前事实 | 施工约束 |
|---|---|---|
| Native Host | `com.htmlgenius.local_bridge` | 不改 host 名；现有用户安装不应失效。 |
| Extension | `extension/manifest.json`，v0.8.2，含固定 `key` 与 `nativeMessaging` | 不新增广泛 host permission；不新增 `management`、文件系统等权限。 |
| 后台入口 | `extension/background.js` | Side panel/content script 均不得直接 `connectNative()`。 |
| macOS 安装器 | `bridge/install-macos.mjs` | 保留为开发兼容入口；其纯函数应被新 CLI 复用或迁移，不要双写两套 manifest 规则。 |
| Host | `bridge/host.mjs` | stdout 只允许 Native Messaging 帧，日志只入 stderr。 |
| provider 探测 | `bridge/provider-probe.mjs` | 继续独立失败域；不得创建 session/thread/candidate。 |
| 支持 provider | `claude_code_cli`、`codex_app_server`、`github_copilot` | 不扩范围。 |
| Node | `^20.19.0 || >=22.12.0` | 延续 Copilot SDK engine 约束，不能简化为“Node 20+”。 |

运行基线：

```bash
cd bridge && npm test
node --check extension/background.js
node --check extension/sidepanel.js
```

若当前分支与上表不一致，先在 [CURRENT_IMPLEMENTATION_STATE.md](CURRENT_IMPLEMENTATION_STATE.md) 重写事实层，再调整本 Spec 的实现映射；不要“兼容猜测”。

## 2. 用户可见流程

### 2.1 未检测到 Bridge

用户完成 Change Contract、来到发送区域时，若 native host 连接失败，不能只显示“未检测到本地 Bridge”。显示紧凑的连接提示卡，不遮挡「复制 Prompt」：

```text
尚未连接本地 Agent
安装本地连接组件后，可将这次编辑任务直接交给你的 Agent。

[ 让 Agent 帮我连接 ]     主按钮
[ 复制 Terminal 命令 ]     次按钮
继续不连接也可使用 [复制 Prompt]
```

点击「让 Agent 帮我连接」：

- 只把一个 Setup Prompt 写进剪贴板，并显示「连接说明已复制。请粘贴到你的 Agent 中执行，完成后回到这里检查连接。」
- **不**试图打开/控制 Claude、Codex、Copilot，不扫描正在运行的应用，也不假装命令已经执行。
- 卡内提供「重新检查」；用户粘贴并执行后可点击它，或重新打开 Side panel。

点击「复制 Terminal 命令」：

- 将一条**无 HTML/comment/prompt/token**的固定版本命令写入剪贴板。
- 明示“该命令会在当前用户目录安装 HTML Genius Bridge，并注册当前 Chrome 扩展；不会安装或登录 Agent。”

点击「复制 Prompt」保持现有行为，不要求建立连接。

### 2.2 已连接但没有可用 provider

Bridge 已健康而三个 provider 均不可用时，显示 Connection Center（见 §5）及逐项真实状态。不要把“Bridge 已连通”误写成“Agent 已连接”。

例：

```text
本地连接组件已就绪
Claude Code：需要在本机登录      [查看如何完成]
Codex：未检测到可用 App
GitHub Copilot：运行时不兼容

[ 检查连接 ]  [ 复制诊断 ]
```

### 2.3 已连接且至少一个 provider 就绪

- 原有发送下拉菜单继续只允许选择真实 `ready` provider。
- Connection Center 默认折叠，不抢占契约页的主编辑动作。
- 「检查连接」只重新探测，不修改本机。
- 「安全修复」仅在 Bridge 已连通且 doctor 给出可安全修复项时显示；必须二次明确写出会修改的项目，例如“重写 HTML Genius 的 Chrome Host 注册文件”。

### 2.4 用户绝不应看到的内容

- `~/.htmlgenius`、`/Applications/...`、CLI 绝对路径、TeamID、schema 路径、Node 路径、stderr、token、session ID。
- “万能修复”“自动安装所有 Agent”“已获得系统权限”等误导承诺。
- 任何 Agent 的历史会话标题、列表或状态。

## 3. 受控 CLI：`htmlgenius-bridge`

### 3.1 分发定位

本仓库本版本只实现并测试 CLI/安装逻辑；**不执行 npm publish**。发布流程是后续外部授权事项。

未来公开发行名定为 `@htmlgenius/bridge`，要求：固定版本、npm provenance/完整性校验、不可让 Side panel 默认使用 `latest`。在未发布阶段，README 只能标为“发布后命令”，不能声称用户可以执行。

正式用户命令的目标形式：

```bash
npx --yes @htmlgenius/bridge@0.9.0 setup --scope user --extension-id <extension-id>
```

开发态可继续使用仓库里的 `node bridge/install-macos.mjs`，但文档必须标为开发者路径，不再作为产品主路径。

### 3.2 新命令与退出码

在 `bridge/` 建立可直接执行的 CLI 入口（建议 `bin/htmlgenius-bridge.mjs`），并在 `bridge/package.json` 暴露：

```json
{
  "bin": {
    "htmlgenius-bridge": "bin/htmlgenius-bridge.mjs"
  }
}
```

必需子命令：

| 命令 | 副作用 | 用途 |
|---|---|---|
| `doctor --json --extension-id <id>` | 无 | 检查 OS、Node、Bridge 文件、Native Host manifest、origin、provider 状态。 |
| `setup --json --scope user --extension-id <id>` | 有 | 幂等地建立/修复当前用户的 Bridge 与 Native Host 注册。 |
| `repair --json --extension-id <id>` | 有 | 仅修复 doctor 判定为 Bridge 自身可安全修复的项目。 |
| `uninstall --json --scope user` | 有 | 删除本工具拥有的 host manifest/launcher/受管 Bridge 文件；不删除工作区审计证据或任一 Agent。 |
| `version --json` | 无 | 输出 Bridge 版本和 protocol version。 |

规则：

- 没有 `--json` 时输出简短可读文本；有 `--json` 时 stdout 必须是**唯一一个 JSON object**，不得混入进度/日志。错误和调试只到 stderr。
- `doctor` 永不写文件、永不触发 Agent 登录、永不创建 Agent session。
- `setup` 与 `repair` 只接受 `--scope user`；拒绝 root/system-wide scope。
- `setup` 必须幂等：同版本、同 extension ID、文件内容正确时返回 `changed:false`。
- extension ID 必须继续用当前 `[a-p]{32}` 严格校验。
- 不支持 OS 或 Node 版本不足时返回稳定 machine code，退出码非 0；不得猜测安装 Node 的命令。
- CLI 的成功 JSON 不能包含绝对路径。开发调试若需路径，使用 `--verbose` 且只写 stderr；Side panel 永不调用 `--verbose`。

### 3.3 安装布局与迁移

不要让 Native Host launcher 指向临时的 `npx` cache；它随后可能被删除。`setup` 应将当前版本的 Bridge 及所需依赖 materialize 到当前用户私有、版本化的受管目录（例如 `~/.htmlgenius/bridge/versions/<version>/`），launcher 只引用该受管目录的绝对 node/host 路径。

要求：

- 目录权限 `0700`，可执行 launcher `0700`，含敏感任务证据的既有文件权限沿用 V0.8.2。
- manifest 仍只写 Chrome 用户级目录：`~/Library/Application Support/Google/Chrome/NativeMessagingHosts/`。
- manifest 的 `allowed_origins` 必须只包含这一次传入的 extension ID。
- `setup` 先将新版本写到临时目录、验证 `host.mjs`/依赖存在，再原子替换/切换 launcher；任何失败不得留下半写 manifest。
- 从 V0.8.2 旧 `install-macos.mjs` 安装迁移时：如旧 host manifest 的 host 名和 extension ID 匹配，直接以受控 V0.9 launcher 替换；如 ID 不同，拒绝覆盖并返回 `EXTENSION_ORIGIN_MISMATCH`。
- `uninstall` 只删除 manifest 中 `name === com.htmlgenius.local_bridge` 且 launcher 带有 HTML Genius 受控标记的文件。不得根据文件名盲删第三方 host。

`install-macos.mjs` 可变成兼容薄包装，调用共享安装模块；其 `--uninstall` 语义保持。安装规则只能有一个实现源。

### 3.4 Doctor 输出契约

新增 `bridge/bridge-health.mjs`（名称可调整，但职责不可混入 UI），输出/Native Host 共用的 protocol：

```json
{
  "schema_version": 1,
  "overall": "ready",
  "bridge": {
    "status": "ready",
    "version": "0.9.0",
    "protocol_version": 1,
    "managed_install": true
  },
  "platform": { "os": "macos", "arch": "arm64", "supported": true },
  "browser": { "status": "origin_ok" },
  "providers": [
    {
      "id": "claude_code_cli",
      "status": "ready",
      "capabilities": ["candidate", "plan"],
      "reason_code": null,
      "remediation": null
    }
  ],
  "actions": ["check"]
}
```

字段限制：

- `overall`：`ready | action_required | unsupported | error`。
- `bridge.status`：`ready | install_required | repair_required | protocol_incompatible | error`。
- `browser.status`：`origin_ok | origin_mismatch | manifest_missing | host_unreachable | unknown`。
- `providers[].status`：沿用并扩展为 `ready | not_installed | auth_required | incompatible | blocked | error | unsupported`。
- `reason_code` 是唯一可依赖的机器诊断字段；UI 不得从英文错误文本做逻辑判断。
- `remediation` 只可为 `{ "kind": "agent_login" | "manual_update" | "terminal_setup" | "none", "label_key": "..." }`。不能含命令、URL、路径、token 或原始错误。
- `actions` 只允许 `check | repair | copy_setup_prompt | copy_terminal_command | copy_diagnostics`。
- 绝不在 JSON 输出 `path`、`command`、`stderr`、`token`、`cookie`、`session`、`thread`、`schema`、完整版本原文。

必需 `reason_code`：

```text
OS_UNSUPPORTED
NODE_UNSUPPORTED
BRIDGE_NOT_INSTALLED
BRIDGE_FILES_CORRUPT
NATIVE_HOST_MANIFEST_MISSING
EXTENSION_ORIGIN_MISMATCH
BRIDGE_PROTOCOL_TOO_OLD
BRIDGE_PROTOCOL_TOO_NEW
CLAUDE_NOT_INSTALLED
CLAUDE_AUTH_REQUIRED
CODEX_APP_NOT_FOUND
CODEX_APP_UNTRUSTED
CODEX_APP_INCOMPATIBLE
COPILOT_RUNTIME_NOT_FOUND
COPILOT_AUTH_REQUIRED
COPILOT_RUNTIME_INCOMPATIBLE
PROVIDER_POLICY_BLOCKED
PROVIDER_PROBE_FAILED
```

必要时可加新 code，但不允许把底层 stderr 透传给扩展。

## 4. Extension ↔ Bridge 协议与后台接线

### 4.1 新 Native Host 消息

在已连通状态，`extension/background.js` 向 `com.htmlgenius.local_bridge` 发送：

```json
{
  "type": "bridge_health",
  "protocol_version": 1,
  "extension": { "id": "<chrome.runtime.id>", "version": "0.9.0" }
}
```

Host 回：

```json
{ "type": "bridge_health_result", "health": { "...": "见 §3.4" } }
```

新增 `bridge_repair`：

```json
{
  "type": "bridge_repair",
  "protocol_version": 1,
  "extension": { "id": "<chrome.runtime.id>" },
  "confirmed_actions": ["repair_native_host"]
}
```

Host 只能执行明确 allow-list 的修复：检查/重写它自身的 launcher 和 manifest、重新 probe provider。不能安装 Node/Agent、运行包管理器、打开 shell、写项目目录。结果仍回 `bridge_health_result`。

### 4.2 最重要的失败分支

Native Host 尚未注册时，`connectNative()` 根本无法让 host 响应。此时 background 根据 `chrome.runtime.lastError` 映射为：

```text
BRIDGE_NOT_INSTALLED
```

不要把 Chrome 原始错误展示给用户；也不要假装能区分所有系统原因。只有 host 成功启动后，才可依赖 `bridge_health` 细分 `origin_mismatch`、版本、Node、provider 等原因。

新增 extension message：

```text
bridge-query-health     # 无副作用，background → native host
bridge-repair           # 仅用户点击且确认后；background → native host
bridge-get-bootstrap    # 纯本地生成 Setup Prompt / Terminal 命令
```

所有 Native Messaging 仍仅由 `background.js` 发起；Side panel 不得直接通信。

### 4.3 兼容策略

- V0.8.2 host 不认识 `bridge_health` 时会返回未知消息或断开；extension 显示“本地连接组件需要更新”，提供 Setup Prompt/Terminal 命令，不自动覆盖。
- host protocol 大于 extension 支持范围时显示“扩展需要更新”；不派发编辑任务。
- 保持现有 `provider_probe` 与 handoff 消息兼容，避免 V0.9 更新使旧已安装 bridge 的发送功能失效。
- `BRIDGE_VERSION` 目前在 background 中为 `0.8.1`，施工时必须统一为 V0.9 的单一版本来源，避免 manifest、package、background、health 四处漂移。

## 5. Side panel：Connection Center

### 5.1 位置与原则

Connection Center 只新增在既有契约发送区附近，以可折叠状态承载；**不得**替换评论 UI、改动评论条目样式、移除既有发送菜单或改变修改范围流程。它是连接故障和设置入口，不是第四个产品页面。

窄 Side panel 约束：无横向滚动、按钮可触达、长文本换行；沿用当前 mint 浅/深色 token，不新增独立视觉体系。

### 5.2 状态矩阵

| Health 状态 | 主文案 | 主操作 | 次操作 |
|---|---|---|---|
| `BRIDGE_NOT_INSTALLED` | 尚未连接本地 Agent | 让 Agent 帮我连接 | 复制 Terminal 命令、复制 Prompt |
| `BRIDGE_PROTOCOL_TOO_OLD` / `BRIDGE_FILES_CORRUPT` | 本地连接组件需要修复 | 让 Agent 帮我修复 | 复制 Terminal 命令、复制诊断 |
| `OS_UNSUPPORTED` | 当前系统暂不支持本地一键发送 | 复制 Prompt | 复制诊断 |
| Bridge ready、无 provider ready | 本地连接组件已就绪 | 检查连接 | 各 provider 的官方登录/更新说明、复制诊断 |
| provider ready | 已连接 {n} 个 Agent | 发送给 Agent（沿用原按钮） | 检查连接、折叠详情 |

「安全修复」仅用于已运行 host 证明可处理的 `repair_required`；首次未安装态绝不出现该按钮。

### 5.3 Setup Prompt 内容与安全

`bridge-get-bootstrap` 必须在 extension 本地用固定模板生成；只动态填入：

- `chrome.runtime.id`；
- extension version；
- 已发布/配置的固定 Bridge version；
- 当前平台标签（本版本只 `macOS`）。

不得将页面 HTML、评论、Change Contract、文件路径、用户名称、provider 登录态填入 Setup Prompt。

中文模板（发布前 `<BRIDGE_VERSION>` 用配置常量，不得出现 `latest`）：

```text
请只帮我初始化 HTML Genius 的本地连接，不要修改任何项目文件、HTML 文件或 Agent 配置。

用户已明确授权你在“当前用户目录”安装/修复 HTML Genius Local Bridge；不要请求管理员权限，不要安装或登录任何 Agent，不要读取历史会话、密钥、Cookie 或项目文件。

Chrome Extension ID：<EXTENSION_ID>
需要的 Bridge 版本：<BRIDGE_VERSION>

请按顺序执行：
1. 先运行只读检查：
   npx --yes @htmlgenius/bridge@<BRIDGE_VERSION> doctor --json --extension-id <EXTENSION_ID>
2. 若检查显示 Bridge 未安装、损坏或需要修复，运行：
   npx --yes @htmlgenius/bridge@<BRIDGE_VERSION> setup --json --scope user --extension-id <EXTENSION_ID>
3. 再运行一次 doctor。

最后只用简短中文汇报：Bridge 是否已就绪；哪些已支持 Agent 可用；哪些仍需要我自行登录或更新。不要输出绝对路径、token、会话信息或原始日志。
```

Terminal 命令模板：

```bash
npx --yes @htmlgenius/bridge@<BRIDGE_VERSION> setup --json --scope user --extension-id <EXTENSION_ID>
```

注意：未发布 npm 包时 UI 不得把这两条命令暴露给真实用户。开发态由 feature flag `bootstrap_distribution = development` 生成仓库内命令，且明显标注“仅开发环境”；正式渠道没有已发布版本时只保留「复制 Prompt」并显示“本地连接组件即将提供”。禁止输出一个必然失败的假命令。

### 5.4 诊断复制

「复制诊断」只复制 §3.4 的脱敏 health JSON，外层可加版本/时间；禁止复制错误堆栈。若 Native Host 不存在，只复制：

```json
{
  "schema_version": 1,
  "overall": "action_required",
  "bridge": { "status": "install_required" },
  "reason_code": "BRIDGE_NOT_INSTALLED",
  "extension_version": "0.9.0"
}
```

## 6. 安全要求

1. **不可把 Setup Prompt 变成任意命令执行通道。** 模板固定，变量严格转义；不拼接用户评论、URL、文件名或页面内容。
2. **CLI 只写用户级 HTML Genius 目录和 Chrome host manifest。** 执行前 canonicalize 路径；拒绝 symlink 逃逸；不接受任意 `--output-dir`、`--hosts-dir` 作为正式发行接口（测试可由内部注入）。
3. **Native Host origin 仍是硬边界。** 单 extension ID，无 wildcard；迁移遇到不匹配 origin 必须停下，不得覆盖。
4. **不把 provider probe 变成登录器。** 探测只返回分类状态；用户登录只能跳转/提示官方流程。
5. **不泄露。** JSON/Native 帧/UI/IndexedDB 均不得持久化或传递路径、原始 stderr、token、cookie、会话 ID、任务正文或 HTML。
6. **不可静默升级。** extension 发现 host 过旧时给出明确修复操作；用户复制/执行 Setup 或确认 repair 后才替换受管文件。
7. **不会扩大 Agent 权限。** Claude/Codex/Copilot 的现有 workspace 围栏、候选写入限制、Plan runtime 锁定均不允许回归。

## 7. 实现拆分与推荐顺序

### A. 抽取共享安装核心

## 8. 测试要求

### 8.1 自动测试（必须）

- CLI：参数、JSON stdout 唯一性、退出码、无副作用 `doctor`、Node 不兼容、OS 不支持、错误脱敏。
- 安装：合法/非法 extension ID、单 origin、权限、原子失败回滚、幂等 setup、旧 V0.8.2 migration、origin mismatch 拒绝、uninstall 不误删非 HTML Genius host。
- 安装布局：launcher 不指向 npx 临时目录；受管目录被删除/损坏后 `doctor` 为 `repair_required`。
- provider health：三 provider 独立失败，一个 probe 崩溃不污染另两个；输出不含路径/stderr/session/token。
- Native protocol：`bridge_health`、旧 host unknown、`bridge_repair` 的 allow-list 与未经确认拒绝。
- background：Native Host 缺失映射 `BRIDGE_NOT_INSTALLED`；protocol mismatch；Side panel 发出的 bootstrap payload 不含 Change Contract/页面内容。
- Side panel：状态矩阵 DOM/按钮、复制内容、未连接仍能 Copy Prompt、无横向 overflow、三语言 key 完整。
- 回归：`cd bridge && npm test` 保持全绿；candidate/plan/provider probe 的既有测试不删除、不降级。

### 8.2 手动验收（macOS + Chrome）

1. 无 host manifest：打开 Side panel，显示「让 Agent 帮我连接」和 Terminal 兜底，Copy Prompt 仍可用。
2. 把生成的 Setup Prompt 发给一个可执行 shell 的本地 Agent；它只执行 doctor/setup/doctor，未修改项目文件；回到 Side panel 点击「重新检查」后 Bridge ready。
3. 同一 extension ID 再 setup：`changed:false`，不产生重复 launcher/manifest。
4. 更换/伪造 extension ID：setup 拒绝覆盖现有 origin，Side panel 不显示底层路径。
5. Bridge 已连接但 Claude 未登录、Codex 未装、Copilot 不兼容：三项各自正确显示，仍可复制 Prompt。
6. 已有 ready provider：发送 candidate 与 plan（若 UI 已公开）均保持 V0.8.2 的安全闭环。
7. `repair` 仅修复受管 host 文件；它不安装 Agent、不打开登录、不修改 source/candidate 工作区。
8. 浅/深色、中文/英文/日文检查 Connection Center；窄 Side panel 无横向滚动。

真实 Copilot 仍需有订阅权益的设备单独 smoke；模拟测试通过不等于真实认证/协议通过。

## 9. 完成定义

V0.9 只有同时满足以下条件才可标记完成：

- 从一个没有 Native Host 的 macOS Chrome 用户态开始，用户不进入源码仓库、不手抄 Extension ID，也能通过复制给本机 Agent 的受控 Prompt 完成用户级 Bridge 初始化；或者通过一条 Terminal 命令完成同样操作。
- Side panel 能区分“Bridge 未装”“Bridge 需更新/修复”“Bridge 已好但 Agent 未就绪”，且不泄露系统细节。
- `doctor/setup/repair/uninstall` 具有稳定 JSON 契约、幂等性、迁移策略和自动测试。
- 任何失败都保留 Copy Prompt；没有静默 shell 执行、自动登录、管理员权限、任意路径写入或既有会话访问。
- V0.8.2 的三 provider、candidate 安全闭环、Plan runtime 锁定和评论 UI 无回归。
