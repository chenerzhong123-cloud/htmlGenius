# htmlGenius v0.5 · 飞书 OAuth 登录 — 设计文档

> 状态：Draft · 日期：2026-07-06 · 基线：v0.4.1（main）· 分支：`v0.5-lark-oauth`
> 配套实现计划：`docs/2026-07-06-v0.5-lark-oauth-plan.md`（后续 writing-plans 产出）

---

## 1. 背景与动机

v0.4 已实现多人协同批注（自托管 FastAPI + SSE + team-token 鉴权 + presence + 仅作者删除级联），代码就绪但**协同入口在 v0.4.1 被隐藏**，原因有二：

1. **身份是软约束**。`X-User-Id` / `X-User-Name` 是客户端自报的请求头（`server/app.py` create_annotation / delete_annotation），后端无法核实。"仅作者删除"靠这个头 → 拿到 team_token 的人可伪造任意作者名强行删别人的批注。这是 v0.4 已知边界，明确留给 v0.5。
2. **team-token 是共享密钥**。一个团队一把 token，无法区分具体用户、无法单独踢人/注销。

v0.5 接入**飞书 OAuth**：用户用飞书账号登录，后端拿到**飞书校验过的 `open_id`** 作为硬身份，自签 session token 下发给扩展。"仅作者删除"从此变成**不可伪造的硬约束**；协同入口随之开放。

## 2. 目标与非目标

### 目标
- **G1 飞书 OAuth 登录**：扩展点「飞书登录」→ `chrome.identity.launchWebAuthFlow` → 飞书授权 → 后端用 code 换用户信息 → 建 session。
- **G2 session token 鉴权**：后端发随机不透明 token，扩展用 `Authorization: Bearer` 携带；替代 team-token。方案一（见 §3）。
- **G3 硬身份**：批注 author 的 `id` = 飞书 `open_id`、`name` = 飞书用户名，**由后端从 session 注入，不再来自请求头**。`delete_annotation` 的作者校验变成硬约束。
- **G4 team = 飞书租户**：`team_id = tenant_key`（飞书自建应用单组织 → 实际为单团队；多组织自动隔离）。
- **G5 复用不动**：SSE rooms、presence、批注模型与表结构、team 隔离语义**全部不改**。只换"进门鉴权 + 身份来源"。
- **G6 协同入口开放**：登录成功后侧边栏切到协同模式（`mode:"synced"`）。
- **G7 开发可用**：提供 `dev-login` 旁路（env 开关），不依赖飞书即可本地开发/测试。

### 非目标（YAGNI）
- **不做 JWT**（方案二已被否决，见 §3）。
- **不做 refresh token / 滑动续期**：session 固定过期，到期重登录即可。
- **不做群组/成员管理 UI**（路线图后续）。
- **不做 PKCE**：飞书授权码流程用 `state` 防 CSRF 已足够。
- **不改 SSE/presence/批注 schema**。
- **不保留 `HG_TEAMS` 兼容**：team-token 鉴权整体下线，由 session 取代（迁移见 §10）。

## 3. 关键决策（已确认）

| 维度 | 结论 | 理由 |
|---|---|---|
| 凭据 | 接真飞书 OAuth（用户已有自建应用 App ID/Secret） | 本期端到端可测 |
| Session 方案 | **方案一：服务端 session token** | 单机 SQLite 查询零成本；注销=删一行；无密钥依赖；和现有 team-token 心智一致。JWT 的无状态优势只在多实例时发挥，用不上 |
| 团队范围 | **`team_id = 飞书 tenant_key`** | 自建应用单组织 → 单团队；多组织自动隔离。若所用飞书 API 未返回可用 tenant 字段，回退 `HG_DEFAULT_TEAM`（默认 `"default"`） |
| 身份来源 | author `id`=`open_id`、`name`=飞书名，**后端 session 注入** | 杜绝 `X-User-Id` 伪造 |

## 4. 数据流：OAuth 登录全流程

```
扩展                      后端                         飞书 OpenAPI
 │                         │                              │
 │ ① getRedirectURL()      │                              │
 │ ──GET /auth/lark/login?redirect=…──────────────────────►│(无)
 │                         │ ② 生成 state(5min TTL)        │
 │                         │   拼 authorize URL            │
 │ ◄──── {auth_url, state} │                              │
 │ ③ launchWebAuthFlow(auth_url, interactive:true)         │
 │ ────────────────────────────────────────────────────────►│ 授权页
 │                         │                   用户登录授权 │
 │ ◄────── 重定向 https://<ext-id>.chromiumapp.org/?code=…&state=…
 │ ④ 从 redirect URL 解析 code (+state)                     │
 │ ──POST /auth/lark/callback {code, redirect_uri, state}─►│
 │                         │ ⑤ 校验 state                  │
 │                         │   code→user_access_token ─────►│ /authen/v1/access_token
 │                         │   user_info ───────────────────►│ /authen/v1/user_info
 │                         │   ← {open_id, name, tenant_key}│
 │                         │ ⑥ sessions.create_session()    │
 │                         │   → 随机 token                 │
 │ ◄──── {token, user:{id,name}, team_id} ─────────────────│
 │ ⑦ chrome.storage.sync 保存 {mode:"synced", backend, session_token, user}
 │ ⑧ 后续请求 Authorization: Bearer <session_token>（无 X-User-* 头）
```

> 重定向 URI：`chrome.identity.getRedirectURL()` 返回 `https://<扩展ID>.chromiumapp.org/`。用户须在飞书应用后台把该串配进"重定向 URL"白名单。扩展把该值传给后端，后端拼进飞书 authorize URL 与 token 交换请求（两处必须一致）。

## 5. 后端改动

### 5.1 新增 `server/sessions.py`（session 存储）
SQLite 新表（`storage.init_db` 加迁移）：
```sql
CREATE TABLE IF NOT EXISTS sessions (
  token TEXT PRIMARY KEY,
  open_id TEXT,
  name TEXT,
  team_id TEXT,
  created_at TEXT,
  expires_at TEXT
);
CREATE INDEX IF NOT EXISTS idx_sessions_open_id ON sessions(open_id);
```
接口：
- `create_session(open_id, name, team_id, ttl_seconds=7*86400) -> str`：生成 `"sess_" + secrets.token_hex(24)`，写入。
- `get_session(token) -> dict | None`：返回 `{open_id, name, team_id}`，过期/不存在返回 None。
- `delete_session(token) -> bool`：注销。
- `prune_expired()`：可选清理。

### 5.2 新增 `server/lark.py`（飞书 OpenAPI 客户端）
env：`HG_LARK_APP_ID`、`HG_LARK_APP_SECRET`、`HG_DEFAULT_TEAM`（缺省 `"default"`）。
- `authorize_url(redirect_uri, state) -> str`：拼 `https://open.feishu.cn/open-apis/authen/v1/authorize?app_id=…&redirect_uri=…&state=…`。
- `app_access_token() -> str`：POST `/open-apis/auth/v3/app_access_token/internal`，带短期缓存（默认 2h，到期重取）。
- `user_access_token(code, redirect_uri) -> dict`：POST `/open-apis/authen/v1/access_token`（带 app_access_token + code）→ 返回含 `user_access_token`。
- `user_info(user_access_token) -> dict`：GET `/open-apis/authen/v1/user_info` → `{open_id, name, tenant_key?, ...}`。
- `team_id` 选取：优先 `tenant_key`；缺失则 `os.environ["HG_DEFAULT_TEAM"]`。

> 飞书 authen 有 v1 / v2 两套端点，**确切端点在实现时对照飞书当前文档确认**（spec 锁定流程与字段，不锁死端点版本）。所有外部 HTTP 走 `httpx.AsyncClient`，失败抛统一异常 → 401/502。

### 5.3 改写 `server/auth.py`
- 删除 `require_team` / `require_team_query` / `_teams()` / `HG_TEAMS` 读取。
- 新增 `require_session(authorization: Optional[str] = Header(None)) -> Session`：Bearer → `sessions.get_session` → 401 缺失/无效/过期。返回 `Session(open_id, name, team_id)`（pydantic 或 dataclass）。
- 新增 `require_session_query(token: Optional[str] = Query(None)) -> Session`：同上，给 SSE `EventSource` 用（`?token=`）。
- `Session` 携带 `team_id`（=tenant_key），后续依赖直接读 `session.team_id`。

### 5.4 改写 `server/app.py` 端点
新增鉴权端点：
- `GET /auth/lark/login?redirect=<enc>` → 生成 state（内存/DB 带 5min TTL）→ 返回 `{auth_url, state}`。
- `POST /auth/lark/callback` body `{code, redirect_uri, state}` → 校验 state → 换 token → 取 user_info → `create_session` → 返回 `{token, user:{id:open_id, name}, team_id}`。
- `GET /auth/me`（Bearer）→ 返回 `{id, name, team_id}`，扩展启动时校验 session 是否仍有效。
- `POST /auth/logout`（Bearer）→ `delete_session`。
- `POST /auth/dev-login`（仅当 env `HG_AUTH_ALLOW_DEV=1`）→ 用配置的假用户直接建 session，返回同 callback 结构。生产默认关闭。

批注/协同端点（**只换依赖与身份来源**）：
- `POST /api/annotations`：`Depends(require_session)`；**删除 `X-User-Id`/`X-User-Name` 参数**；`author = {"id": session.open_id, "name": session.name}`；`team_id = session.team_id`。其余（broadcast）不变。
- `GET /api/annotations`：`require_session`，`team_id = session.team_id`。
- `DELETE /api/annotations/{aid}`：`require_session`；`storage.delete_annotation(aid, session.team_id, session.open_id)`（actor_id 现为 open_id，**硬校验**）。403 语义不变。
- `POST /api/presence`：`PresenceIn` 改为 `{doc, op}`（去掉 `user`）；服务端用 `{"id": session.open_id, "name": session.name}` 组装 user。
- `GET /api/stream`：`require_session_query`；`team_id = session.team_id`。

> `storage.delete_annotation(aid, team_id, actor_id)`、`rooms`、`presence.update`、`models.py` **签名与实现均不改**——actor_id 由"header 字符串"变成"session.open_id"，对它们是透明替换。

## 6. 扩展改动

### 6.1 `manifest.json`
- `permissions` 增加 **`"identity"`**（`launchWebAuthFlow` 所需）。
- 恢复 **`host_permissions`** 指向后端域名（v0.4.1 为上架商店移除了；协同需要它跨域 fetch/EventSource）。本地开发用 `http://localhost:8000/*`，生产 `https://<后端域名>/*`。

### 6.2 `sidepanel.html` + `sidepanel.js`
- 协同配置区从 v0.4.1 的"隐藏"改为开放，但内容换成：**后端地址输入框 + 「飞书登录」按钮**（移除团队 token / 用户名输入）。
- 登录逻辑：
  ```js
  const redirect = chrome.identity.getRedirectURL();
  const { auth_url } = await fetch(`${backend}/auth/lark/login?redirect=${encodeURIComponent(redirect)}`).then(r=>r.json());
  const respUrl = await chrome.identity.launchWebAuthFlow({ url: auth_url, interactive: true });
  const code = new URL(respUrl).searchParams.get("code");
  const state = new URL(respUrl).searchParams.get("state");
  const { token, user } = await fetch(`${backend}/auth/lark/callback`, {
    method:"POST", headers:{"Content-Type":"application/json"},
    body: JSON.stringify({ code, redirect_uri: redirect, state })
  }).then(r=>r.json());
  chrome.storage.sync.set({ mode:"synced", backend, session_token: token, user });
  ```
- 启动时若已有 `session_token`，`GET /auth/me` 校验；失效则提示重新登录。
- 显示当前登录用户名 + 「退出登录」（调 `/auth/logout` 并清 storage）。

### 6.3 `remote-store.js`
- `authHeaders(cfg)`：`"Authorization": "Bearer " + cfg.session_token`；**删除 `X-User-Id` / `X-User-Name`**；保留 `Content-Type`。
- 其余（buildCreateRequest / make / save / list / delete）不变。

### 6.4 `sync.js`
- `opts.team_token` → `opts.session_token`；EventSource URL 与 presence 请求的 `Authorization: Bearer` 改用 session_token；**删除 presence 请求里的 `X-User-Id`/`X-User-Name` 头**；`user` 字段保留（仅展示用，服务端以 session 为准）。

### 6.5 `content-script.js` / `storage.js`
- `cfg` 字段 `team_token` → `session_token`；`Storage.configure` 与 `Sync.start` 透传 session_token；协同入口恢复显示。

## 7. 配置与环境变量

| env | 用途 | 默认 |
|---|---|---|
| `HG_LARK_APP_ID` | 飞书自建应用 App ID | 必填（协同） |
| `HG_LARK_APP_SECRET` | 飞书自建应用 App Secret | 必填（协同） |
| `HG_DEFAULT_TEAM` | tenant_key 缺失时的 team_id 回退 | `"default"` |
| `HG_AUTH_ALLOW_DEV` | 是否开放 `/auth/dev-login` | `"0"`（生产关） |
| `HG_SESSION_TTL` | session 有效期（秒） | `604800`（7 天） |

> 移除 `HG_TEAMS`。

## 8. 安全考量
- **硬身份**：author.id 来自 session.open_id（飞书校验），客户端无法伪造 → 仅作者删除为硬约束。
- **state 防 CSRF**：login 生成随机 state 带 5min TTL，callback 校验；redirect_uri 由扩展传入且必须与飞书后台白名单一致。
- **App Secret 不落扩展**：code→token 交换只在后端进行，扩展只拿到 session token。
- **session 过期**：`expires_at` 校验；过期需重登录。
- **dev-login 必须关闭生产**：`HG_AUTH_ALLOW_DEV=1` 才挂载该路由。
- **注销即失效**：`delete_session` 删行，token 立刻无效（team-token 做不到）。

## 9. 测试策略（TDD）
- **sessions**：create → get → 过期 → delete；TTL 边界。
- **auth 依赖**：`require_session` 缺失/无效/过期 → 401；有效 → 返回正确 team_id。
- **lark 客户端**：monkeypatch `httpx`，断言 authorize_url 参数、code→token、user_info 解析、team_id 选取（含回退）。
- **端点**：`/auth/lark/login` 返回 auth_url+state；`/auth/lark/callback`（mock lark.exchange）建 session 返回 token；`/auth/me`、`/auth/logout`、`/auth/dev-login` 受控。
- **批注端点**：create 的 author.id == session.open_id（非请求头）；delete 非作者硬 403；跨 team 隔离。
- **扩展**：`remote-store.authHeaders` 用 session_token 且无 X-User 头；`sync.js` URL/Authorization 用 session_token；新增 login-flow 测试页（mock launchWebAuthFlow + callback）。

## 10. 迁移与兼容
- `annotations` 表无 schema 变化（`team_id` 含义从"token 映射的团队"变为"飞书租户"，旧数据 `team_id="default"` 在 `HG_DEFAULT_TEAM=default` 下仍可读）。
- `sessions` 表新增，`init_db` 幂等迁移。
- v0.4 的 `HG_TEAMS` 配置失效；已部署实例升级 v0.5 后，老 team-token 全部 401，需改用飞书登录。
- 本地（`mode:"local"`）模式**完全不受影响**（LocalStore / IndexedDB / 不连后端）。

## 11. 部署要点
- 后端：配置 `HG_LARK_APP_ID/SECRET`；生产关闭 `HG_AUTH_ALLOW_DEV`；Nginx SSE 配置沿用 v0.4 deploy 文档。
- 扩展：`manifest.json` 加 `identity` + 后端 `host_permissions`；上架商店时在审核说明里解释 `identity`（OAuth 登录）与 host_permissions（连自托管后端）用途。
- 飞书后台：把 `chrome.identity.getRedirectURL()` 返回值加入"重定向 URL"白名单（开发机与正式扩展 ID 不同，分别配置）。
