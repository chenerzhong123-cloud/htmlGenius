## 1. 背景与动机

v0.3 插件实现了"任意网页划词批注 + 回灌"，但是**纯客户端、单用户**（IndexedDB）。真实诉求是：生成的 HTML 链接发到团队后，多人能像飞书文档那样**同时**划词评论、互相回复，评论聚合后一键回灌给 Agent 改。

硬约束（本轮澄清后确定）：
1. **数据必须自托管**——批注/正文片段不得经过任何第三方服务器。排除 hypothes.is 公共服务、Liveblocks 托管服务等一切 SaaS 协同后端。
2. **协同走 Chrome 插件形态**——基于 v0.3 插件演进，不是新建 web app。
3. 阶段化：先做**多人批注协同 + 回灌（B1）**；多人直接改正文（B2，需 CRDT）后做。

**核心洞察**：v0.3 插件 + v0.2 FastAPI 已经具备 B1 的 90%。缺的只是一条**插件 ↔ 自有后端的实时推送通道**，把"插件的 IndexedDB"与"后端的 SQLite"连起来，让多用户共享同一份批注。

## 2. 目标与非目标

### 目标
- **G1 多人实时批注**：多用户插件打开同一 URL，A 划词 → B、C 实时看到高亮 + 侧栏卡片。
- **G1.1 多级回复**：任意用户可对任一已发布批注回复，回复可再被回复（任意深度线索，`parent_id` 串联）。
- **G1.2 删除（仅作者）**：发布者本人可删自己的批注；他人不可删（hover 出按钮，见 G1.3）。
- **G1.3 卡片 hover 交互**：每张已发布批注卡片在 hover 时浮现「回复」「删除」按钮（仅作者显「删除」）。
- **G2 数据自托管**：所有批注存在你阿里云的 SQLite/Postgres，零第三方。
- **G3 团队隔离**：按 `(team, document_id)` 隔离；不同团队互不可见。
- **G4 在线状态（presence）**：侧栏显示"谁正在看这页"。
- **G5 一键复制所有评论**：原「回灌」按钮改文案为「一键复制所有评论」；复制时**遍历全部多级回复**（不只顶层），组装成结构化 prompt 复制剪贴板。组装逻辑零改动，仅加回复树遍历。
- **G6 跨版本重定位沿用**：document_id(=URL) 稳定即可，AI 重写后批注自动 re-anchor（v0.3 已有能力）。

### 非目标（YAGNI，明确排除）
- **不做多人直接改正文**（B2，需 CRDT/Yjs）。
- **不做版本同步**：版本/编辑历史仍留本地 IndexedDB。
- **不做飞书 OAuth**：MVP 用团队共享 token；OAuth 列后续。
- **不做 CRDT 合并**：批注是"追加为主、低冲突"，last-write-wins 足够。
- **不重构 v0.3 单机模式**：本地/远程单机批注（IndexedDB）原样保留，同步是叠加的新模式。

## 3. 架构

```
用户A 插件(content-script 持 EventSource) ──┐
                                            ├── https://你的阿里云  ── FastAPI
用户B 插件(content-script 持 EventSource) ──┤     │                  ├─ SSE 流 /api/stream(按 team+doc 分房间)
                                            │     │                  ├─ REST /api/annotations(已有,加鉴权)
用户C 插件(content-script 持 EventSource) ──┘     │                  ├─ REST /api/presence(join/心跳/bye)
                                                   │                  └─ SQLite(team_id, document_id)
                                                   │
   写:插件 ──REST POST/DELETE──► 后端(token→team_id) ──存 SQLite──► SSE 广播该变更到房间
   推:后端 ──SSE event──► 房间内所有插件 ── delta 应用到 window.__hgAnnotations + overlay
   presence:插件 ──REST POST join/心跳/bye──► 后端 ──SSE 广播 users[]
```

关键边界：
- **SSE 长连接住在 content-script**（每标签页一条 `EventSource`，随标签生命周期）。**不住 background service worker**——MV3 SW 30s 回收会掐断长连接。
- **写走 REST**（复用已有 `/api/annotations`，加鉴权），**SSE 只做 server→client 单向推送**。写路径简单（每次调用独立鉴权），推路径单向（纯 HTTP，无需 Nginx Upgrade，自动重连）。
- **后端是批注的唯一真源（source of truth）**；插件 IndexedDB 退化为离线缓存。
- **选 SSE 而非 WS 的理由**：B1 流量是"推送为主、写走 REST、人速低频"（回复 = 一次写+一次推，不构成双向需求）；唯一需 WS 高频上行的"多人共打正文"是已推迟的 B2。SSE 在本场景运维更省（无 Upgrade 配置、`EventSource` 自带重连），且 B2 真上时加一路 WS 即可，不动 SSE 这套（payload 都是 JSON）。

## 5. 关键设计决策

### 5.1 写走 REST、推走 SSE（单向）
写用 REST：每次请求独立带 `Authorization: Bearer <token>`，无状态、易测、可穿越任何代理。SSE 只推送变更（单向 HTTP 流）。presence 走 REST POST（join/心跳/bye）。回复（B 对 A 再评论）= 一条带 `parent_id` 的新批注写 + 一次推送，**不构成双向需求**。

### 5.2 EventSource 住在 content-script，不住 background
MV3 service worker 30s 空闲回收，长连接必断。content-script 随标签存活，是 MV3 下唯一稳定的长连接宿主。代价：多标签 = 多 SSE 流（可接受，后端按房间聚合）。

### 5.3 后端是唯一真源，IndexedDB 退化为缓存
synced 模式下，所有读写以后端为准。断网时：写仍尝试 REST（失败则本地队列待重连后重放）；重连后用一次 `GET /api/annotations` 全量对账（按 id 集合 diff）。SSE `EventSource` 自带断线重连。**不做操作级 CRDT**——批注低冲突，last-write-wins + 全量对账足够。

### 5.4 team_id 由 token 决定，不由请求体决定
防越权：服务端从 token 解析 team_id 注入批注，**不信任客户端 body 里的 team_id**。GET 查询也按 token 的 team_id 过滤。

### 5.5 跨版本重定位 = 稳定 URL（沿用 v0.3 能力）
document_id = `origin + pathname`（v0.3 §5.4）。**只要 HTML 挂在稳定 URL**，AI 重写新版本挂同一 URL，后端批注按 document_id 存着，插件刷新自动 re-anchor 到新 DOM；AI 改掉的文字 → 既有 stale/归档 兜底。**无需新代码**，靠部署约束（URL 不带版本号）。

### 5.6 host_permissions 增加自有后端域名
content-script 连后端需跨域权限。manifest 增 `host_permissions: ["https://<你的后端域名>/*"]`（**scoped 到自有域名，非 `<all_urls>`**），过审可说明用途。`activeTab` 不够（不覆盖跨域 fetch/EventSource）。

### 5.7 鉴权 MVP：团队共享 token（生命周期）
- **是什么**：整个团队共用的一串口令（类比 Wi-Fi 密码 / Slack 工作区邀请码）。token→team_id。同一团队所有人用同一 token；其批注只对持同一 token 者可见。**团队级身份，非个人级**（插件里"用户名"只是自报展示标签，服务端不审计谁干的）。
- **生成**：管理员造一随机串（如 `tok_alpha_a3f9b2e7`），配进服务端 env `HG_TEAMS='{"tok_alpha_a3f9b2e7":"team_alpha"}'`。
- **分发**：通过飞书/邮件私发团队成员（out-of-band）。
- **用户获取**：成员在**插件设置侧栏**粘进"团队 Token"框，存 `chrome.storage.sync`（随其 Chrome 账号同步多设备）。插件每次请求带上：REST 走 `Authorization: Bearer`；SSE 走 `?token=` 查询（EventSource 无法设 header）。
- **轮换**：有人离职 → 换新 token、改 env、通知留下的人。**无单人撤销**（局限）。
- **以后换飞书 OAuth**：只替换"token→team_id"这一层（鉴权依赖），SSE/REST/房间/批注模型全不动（§11）。

### 5.8 presence
连接建立后插件 `POST /api/presence {doc, user}` → 后端加入房间用户集 → SSE 广播 `{event:"presence", users:[...]}`。心跳 25s 一次 POST；最后心跳超过 60s 视为掉线 GC。卸载用 `navigator.sendBeacon` 发 bye。多标签同用户按 (user, 标签) 去重展示。

### 5.9 卡片交互：hover 出「回复/删除」
每张已发布批注卡片默认只显正文；hover 时浮现「回复」「删除」。「删除」仅在 `annotation.author.id === 当前 user.id` 时显示（客户端隐藏）。点「回复」展开输入框 → 提交即 `POST /api/annotations` 带 `parent_id`（§7.3）。

### 5.10 删除：仅作者 + 级联
- **鉴权**：服务端校验 `annotation.author.id === 请求声明的 user.id`，不符 → 403。**注意（团队 token 下为"软"约束）**：token 是团队级共享密钥，声明的 user.id 未经密码学验证，理论上队友可伪造他人 id 绕过。小团队可接受；**硬授权需 OAuth**（§11）。
- **级联**：删一条带回复的批注 → 整棵子树一并删（前端二次确认"这条有 N 条回复，将一并删除"）。MVP 选级联最简；备选（软删留占位 / 有回复时阻止删除）列后续。

### 5.11 底栏：「一键复制所有评论」+「切换编辑/查看模式」
- 评论栏底部两按钮：
  - **「一键复制所有评论」**（原「回灌」改名）：调 `buildPrompt`，**DFS 遍历回复树**把所有层级评论纳入结构化 prompt → 复制剪贴板（§7.6）。
  - **「切换编辑模式 ↔ 切换查看模式」**（原「编辑」改名，toggle）：进编辑 = 开 contenteditable（v0.3 既有本地编辑）；已在编辑 = 按钮文案变「切换查看模式」，点回只读。**B1 范围内此编辑为本地单用户**（content 改动不协同给他人，那是 B2）；编辑改 DOM 后批注按 v0.2 既有 debounce re-anchor。

### 5.12 扩容演进（不在 B1 实现，仅留路径）
单进程内存房间在"几百连接"量级零压力。当连接数上千或需多 uvicorn worker 时，**把房间广播从"进程内 Queue"换成 Redis pub/sub backplane**——只在 `sse.py`/`presence.py` 改广播那几行，不动架构、不动协议、不动插件。

## 6. 数据模型

### 6.1 后端批注表扩展（SQLite 迁移）
```sql
ALTER TABLE annotations ADD COLUMN team_id   TEXT DEFAULT 'default';
ALTER TABLE annotations ADD COLUMN parent_id TEXT;   -- 回复指向父批注 id;NULL=顶层
```
`Annotation` 模型加 `team_id: str = "default"`、`parent_id: Optional[str] = None`；`AnnotationCreate` 加可选 `author: dict`、`parent_id: Optional[str]`。`team_id` 服务端从 token 注入，**不接受 body**。索引：`(team_id, document_id)`。**多级回复**：`parent_id` 指向直接父级，前端按它重建任意深度树（MVP 不限层级）；**删除 = 行移除 + 级联子树**（§5.10）。

### 6.2 协议

**SSE 事件（server → client，`text/event-stream`）**
```
event: hello
data: {"room":"<team_id>:<document_id>"}

event: annotation:created
data: { /* 完整 Annotation */ }

event: annotation:deleted
data: {"id":"ann_..."}

event: presence
data: {"users":[{"id":"u_a","name":"A"},...]}
```
（`event: annotation:updated` 协议预留，MVP 可不接线。）

**REST 端点（client → server）**
- `POST /api/annotations`（Bearer；body: selector/quote/body/author/parent_id）→ 创建，team_id 注入
- `DELETE /api/annotations/{id}`（Bearer + 声明 user.id）→ **作者校验通过则删（级联子树）+ 广播**；非作者 403
- `GET  /api/annotations?document_id=...`（Bearer）→ 拉全量初值（按 token 的 team 过滤）
- `POST /api/presence`（Bearer；body: `{doc, user, op:"join"|"heartbeat"|"bye"}`）
- `GET  /api/stream?doc=...&token=...` → SSE 流（EventSource）

> **MVP 实现范围**：create + delete + presence + 回复（create 带 parent_id）；广播与接收俱全。`annotation:updated`（编辑评论/action）协议预留，卡片编辑列为收尾项。

### 6.3 插件配置（chrome.storage.sync）
```jsonc
{ "mode": "synced" | "local", "backend": "https://your-host",
  "team_token": "tok_alpha_a3f9b2e7", "user": { "id": "u_a", "name": "阿甲" } }
```
`mode=local` 时行为完全等同 v0.3（零回归）。

## 7. 数据流

### 7.1 连接建立
插件加载 → 读配置 → 若 `mode=synced`：`sync.js` 先 `GET /api/annotations` 拉全量初值渲染 → 开 `EventSource('/api/stream?doc=<document_id>&token=<token>')` → 后端验 token 得 team_id → 加入房间 `(team_id, document_id)` → 推 `hello` + `presence` → 插件 `POST /api/presence{op:join}`。

### 7.2 创建批注（自己）
选区 → `describe` 得 selector → `remote-store.saveAnnotation` → `POST /api/annotations`（Bearer，body 带 selector/quote/body/author）→ 后端存(team_id 注入)+**SSE 广播 `annotation:created` 到房间**→ 本机 `applyRemoteChange` 渲染 + 其他用户收到渲染。

### 7.3 回复（B 对 A）
在 A 的卡片点回复 → 输入 → `POST /api/annotations`（body 多带 `parent_id=A.id`）→ 后端存 + 广播 `annotation:created`（带 parent_id）→ 各端在 A 卡片下挂子卡。

### 7.4 接收批注（他人）
SSE `onmessage` 收 `annotation:created` → `applyRemoteChange({op:"create", annotation})` → push 进 `window.__hgAnnotations`（有 parent_id 则挂到父）→ 单条 `anchor`+overlay（不全量重渲染）→ 通知 sidepanel 刷卡片。

### 7.5 删除批注（仅作者）
作者 hover 自己卡片 →「删除」→ 二次确认（有回复提示 N 条）→ `DELETE /api/annotations/{id}`（带声明 user.id）→ 后端校验作者 + 级联删子树 + 对被删的每条各广播 `annotation:deleted` → 各端移除对应 overlay + 卡片。非作者请求 → 403。

### 7.6 一键复制所有评论（原「回灌」）
底栏「一键复制所有评论」→ `chrome.tabs.sendMessage(get-export)` → content-script 返回 `window.__hgAnnotations`（**已含他人与回复**）→ `buildPrompt` **按 parent_id 重建树并 DFS 遍历**，把**所有层级**评论（顶层 + 嵌套回复）按线索缩进纳入结构化 prompt → 复制剪贴板。组装逻辑零改动，仅新增回复树遍历与缩进格式。

### 7.7 跨版本重定位
AI 重写 → 新版本 HTML 挂同一 URL → 团队刷新 → `loadAnnotations` 重 anchor 到新 DOM（v0.3 既有）；失效项 stale。无新代码。

## 8. 部署

- **后端**：阿里云，uvicorn + Nginx 反代（已有 Certbot SSL）。
- **Nginx SSE 段**（关键：关缓冲，否则事件被攒着不推）：
  ```nginx
  location /api/stream {
    proxy_pass http://127.0.0.1:8000;
    proxy_buffering off;
    proxy_cache off;
    proxy_read_timeout 3600s;
    proxy_set_header Connection '';
    proxy_http_version 1.1;
  }
  ```
  并确保对外走 **HTTP/2**（Certbot + 现代 Nginx 默认），消除 HTTP/1.1 下每域名 6 条 SSE 的多标签限制。
- **团队 token 配置**：后端 env `HG_TEAMS='{"tok_alpha_a3f9b2e7":"team_alpha"}'`。
- **HTML 托管**：生成 HTML 挂稳定 URL（`/d/<document_id>`），**URL 不含版本号**；版本切换由后端取 current，URL 不变。
- **分享**：发 URL + 团队 token（token 进插件设置）。

## 9. 测试策略

### 后端（pytest，复用现有 tests/）
- **SSE 房间管理**：两连接同 `(team,doc)` 互收事件；不同 team/doc 隔离不收。
- **鉴权**：无 token / 错 token → 401；token→team_id 注入正确。
- **越权**：body 带 team_id 被忽略，以 token 为准；team A 查不到 team B 批注。
- **写后推送**：POST/DELETE 后房间内收到对应 SSE event。
- **presence**：join 后广播 users；超时未心跳被 GC 移除。
- **回复与多级**：带 parent_id 的批注按线索聚合；任意深度树重建正确。
- **删除鉴权**：作者 DELETE 成功并级联子树；非作者 403。
- **一键复制多级**：含嵌套回复的文档，复制产物含所有层级（DFS 不漏）。
- **迁移**：`team_id`/`parent_id` 列存在；旧行 default；现有测试全绿（回归）。

### 插件（手动矩阵 + 可选 Playwright）
- **双机实时**：两标签同 URL，A 批注 → B 秒级出高亮+卡片。
- **回复串联**：B 在 A 卡片 hover→回复 → A 卡片下出现子卡；多级回复树正确。
- **作者删除**：A 能删自己批注（hover 出按钮、二次确认）；B 看不到删除按钮、API 403；级联删提示 N 条。
- **一键复制多级**：含嵌套回复时，复制产物含全部层级。
- **编辑/查看 toggle**：「切换编辑模式」开 contenteditable；再点「切换查看模式」回只读。
- **presence**：A 进入 → B 侧栏出现 A；A 关标签 → B 移除 A。
- **断线重连**：`EventSource` 自动重连；重连后 `GET` 全量对账一致。
- **mode 回归**：`mode=local` 行为与 v0.3 完全一致。
- **跨版本**：换版本 HTML（同 URL）→ 批注 re-anchor，失效项 stale。

## 10. 已知边界

- **严格 CSP 站点**：少数第三方页 `connect-src` 极严，可能阻断 content-script 到后端的 SSE/REST。**MVP 缓解**：自有托管 HTML（主场景，CSP 可控）必过；第三方严 CSP 站点回退 REST 轮询（`GET /api/annotations` 每 N 秒，无实时但可用）。
- **token 进 SSE query string**：会进后端 access log（低敏）。可接受；后续可换 cookie（`cookies` 权限）。
- **多标签多 SSE 流**：HTTP/2 下无 6 条限制；HTTP/1.1 极端代理下可能撞限（缓解：确保 Nginx HTTP/2）。
- **写冲突**：并发改同一条批注 last-write-wins，无提示（批注低冲突，YAGNI）。
- **token 无单人撤销**：离职需整体轮换（§5.7）；要单人撤销则后续上 OAuth。
- **「仅作者删除」在 token 下为软约束**：声明的 user.id 未经密码学验证，可被伪造绕过（§5.10）；硬授权需 OAuth。
- **远程 SPA 页**：URL 不变 DOM 变 → TextQuote 找不到 → stale（v0.3 既有，UI 明示）。
- **MV3 host_permissions**：增加自有后端域名会让商店审核多一项说明（可控）。

## 11. 与 v0.3 的关系 & 后续

- v0.3 单机模式（local/remote + IndexedDB）**保留**，`mode=local` 零回归。
- v0.4 同步模式是叠加：`mode=synced` 时启用后端 + SSE。
- **B2（多人改正文）**：待 CRDT 取向定（约束 schema vs 保任意 HTML）。届时若需高频上行（光标/共打），**加一路 WS**（y-websocket 等），与本 SSE 推送并存或替换；payload 都是 JSON，迁移成本低。房间与鉴权骨架复用本版。
- **飞书 OAuth**：token 方案稳定后再接，**仅替换 token→team_id 鉴权层**，SSE/REST/房间/批注模型全不动。

## 12. 退出标准（B1）

- 两台机器、两个真实浏览器，同 URL 实时互见批注 + 多级回复 + presence。
- 卡片 hover 出「回复」（所有人）/「删除」（仅作者）；作者删除级联子树，非作者 403。
- 「一键复制所有评论」产物含**全部多级回复**，粘给 Agent 能正确改 HTML。
- 「切换编辑/查看模式」toggle 正常（B1 内编辑仍本地单用户）。
- AI 重写新版本（同 URL）后，批注自动 re-anchor。
- `mode=local` 与 v0.3 行为一致；后端现有测试全绿。
- 数据全程只在自有阿里云，不触达任何第三方。
