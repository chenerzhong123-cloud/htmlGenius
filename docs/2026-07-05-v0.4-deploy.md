# v0.4 部署清单(多人协同批注)

> 配套 spec §8:`docs/2026-07-05-v0.4-plugin-collab-design.md`。
> 后端跑在阿里云、Nginx 反代 + Certbot SSL 的现成机器上。

## 1. 后端起服

```bash
HG_TEAMS='{"tok_alpha_a3f9b2e7":"team_alpha","tok_beta_c8d1e2f3":"team_beta"}' \
  uv run uvicorn server.app:app --port 8000 --reload
```

- `HG_TEAMS`:JSON map,**键 = 团队 token(随机串),值 = team_id**。多团队用逗号分隔。
- token 由管理员自造随机串,不要用人类可猜的词。team_id 是批注在 DB 里的隔离键。
- 不配 `HG_TEAMS` 或 JSON 非法 → 所有需要鉴权的接口 401。

## 2. Nginx:`/api/stream` 必须关缓冲(SSE 关键)

SSE 靠流式推送;Nginx 默认会把响应攒在缓冲区里,客户端永远收不到事件。必须单独给 `/api/stream` 关掉 buffering:

```nginx
location /api/stream {
    proxy_pass http://127.0.0.1:8000;
    proxy_buffering off;
    proxy_cache off;
    proxy_read_timeout 3600s;     # SSE 长连接,给够超时
    proxy_http_version 1.1;       # 上游到 uvicorn 用 HTTP/1.1
    proxy_set_header Connection '';
}
```

并确保**对外走 HTTP/2**(Certbot + 现代 Nginx 默认就开):

```nginx
listen 443 ssl http2;
```

> 为什么强调 HTTP/2:HTTP/1.1 下浏览器对同一域名最多 6 条并发连接,开 6 个标签贴着同一 SSE 域名就把额度耗光、第 7 个标签连不上。HTTP/2 多路复用没有这个限制。

## 3. `HG_TEAMS` 环境变量配置

- 生产别明文写在 `systemd` unit 里也行——用 `EnvironmentFile=/etc/htmlgenius/env` 指向一个 `chmod 600` 的文件,内容形如 `HG_TEAMS={"tok_alpha_a3f9b2e7":"team_alpha"}`。
- 增减团队 = 改 env + 重启 uvicorn;**老 token 一旦改掉,持有它的插件会全部 401**(需重新填 token)。
- team_id 变更要谨慎:DB 里已有批注按旧 team_id 落库,改 token 的值会让这批数据「失联」(仍存在但新 team 下看不到)。

## 4. 插件 manifest:替换 `YOUR_BACKEND_HOST`

打包插件前**必须**改 `extension/manifest.json` 的 `host_permissions`:

```jsonc
"host_permissions": ["https://YOUR_BACKEND_HOST/*"]
//                     ^^^^^^^^^^^^^^^^^^^^^ 替换为真实后端域名,如 https://api.example.com
```

- 这是 content-script 跨域 fetch / EventSource 连后端的权限。`activeTab` 不够(不覆盖跨域请求)。
- scoped 到自有域名(不是 `<all_urls>`),过 Chrome 商店审核时可说明用途。
- 改完域名 → 重新「加载已解压的扩展程序」,或重新打包 `.crx`。
- **本地开发**:`host_permissions` 用 `http://localhost:8000/*` 即可。

## 5. 稳定 URL 托管(跨版本 re-anchor 的前提)

生成的 HTML 挂在**不含版本号**的稳定 URL 上:

```
✓ https://example.com/d/spec          ← 稳定,版本切换 URL 不变
✗ https://example.com/d/spec/v2       ← 带版本号,跨版本锚定会断
```

后端取 current 版本渲染,URL 始终是 `/d/<document_id>`。这样 AI 重写出的新版本挂回同一 URL 后,旧批注基于 text-quote selector(exact + prefix + suffix)才能自动 re-anchor 到新内容。

## 6. 手动集成验收矩阵

> 自动化回归见 `uv run pytest -v`(69 项全绿)。**以下矩阵需人工在两台机器/两个浏览器跑**,跑过再发布。

环境:两个浏览器(或一机双浏览器 profile),都加载插件、填**同一 team token**、不同名字,打开**同一稳定 URL**。

- [ ] **实时同步**:A 划词批注 → B/C 秒级出高亮 + 卡片。
- [ ] **多级回复**:B 在 A 的卡片 hover → 回复 → A 卡片下出现子卡;再多级回复,线索树结构正确。
- [ ] **作者删除 + 级联**:A hover 自己卡片 → 删除(级联提示 N 条子批注);B 看不到删除按钮、强行 `DELETE` API 返回 403。
- [ ] **presence**:A 进入 → B 侧栏出现 A;A 关标签 → 60s 内 B 看到 A 移除。
- [ ] **一键复制全量**:含多级回复时,「复制所有评论」产物含**全部层级**(DFS 不漏)。
- [ ] **编辑/查看 toggle**:点「切换编辑模式」开 contenteditable;再点回「切换查看模式」恢复只读。
- [ ] **AI 重写同 URL re-anchor**:把回灌 prompt 喂 AI 改出新版本,挂回同一稳定 URL → 旧批注自动 re-anchor 到新内容位置。
- [ ] **mode=local 零回归**:side panel 切回 local 模式(或断开后端),行为与 v0.3 完全一致(本地增删改、不连后端)。

附加(若时间允许):

- [ ] **断线重连**:断网恢复后 EventSource 自动重连,`GET` 全量对账与最新一致。
- [ ] **team 隔离**:两浏览器填**不同 team token** → 互相看不到对方批注。

## 7. 常见坑

| 现象 | 原因 | 处理 |
|---|---|---|
| SSE 连上但收不到事件 | Nginx 没关 `proxy_buffering` | 见 §2 |
| 第 7 个标签连不上 | 走了 HTTP/1.1,6 连接额度耗光 | 确保 `listen ... http2` |
| 全部接口 401 | `HG_TEAMS` 没配 / token 拼错 / JSON 非法 | 检查 env,日志会打 team 解析失败 |
| 插件连不上后端 | manifest `host_permissions` 还是占位符 / 域名拼错 | §4 替换 `YOUR_BACKEND_HOST` |
| 严格 CSP 站点不实时 | 页面 CSP `connect-src` 拦了 SSE | 退化为轮询(数据不丢,只是不实时) |
| 仅作者删除被绕过 | ~~team-token 不是强身份~~ | v0.5 已修复(author=飞书 open_id,硬身份) |

## 8. v0.5 补充:飞书 OAuth + session(替换 team-token)

### 8.1 后端 env

```bash
HG_LARK_APP_ID=cli_xxx HG_LARK_APP_SECRET=sec_xxx \
HG_AUTH_ALLOW_DEV=0 \
HG_DEFAULT_TEAM=default \
HG_SESSION_TTL=604800 \
uv run uvicorn server.app:app --port 8000
```

- `HG_LARK_APP_ID/SECRET`:飞书自建应用凭据,必填。
- `HG_AUTH_ALLOW_DEV`:**生产保持 `0`**;仅本地/测试设 `1` 开 `/auth/dev-login`。
- `HG_TEAMS` 已移除,不再生效。

### 8.2 manifest:`identity` + `host_permissions`

```jsonc
"permissions": ["activeTab", "sidePanel", "storage", "identity"],
"host_permissions": ["http://localhost/*", "http://127.0.0.1/*", "https://*/*"],
```

- `identity`:`chrome.identity.launchWebAuthFlow`(飞书登录)所需。
- `host_permissions`:扩展跨域连后端所需。生产可把 `https://*/*` 收窄到 `https://<你的后端域名>/*`。

### 8.3 飞书后台:重定向 URI

扩展登录时调 `chrome.identity.getRedirectURL()`,返回形如 `https://<扩展ID>.chromiumapp.org/`。**把该串加入飞书自建应用的「安全设置 → 重定向 URL」白名单**(开发机与正式扩展 ID 不同,分别配置)。

### 8.4 登录流程(扩展侧)

侧边栏填后端地址 → 点「飞书登录」→ `launchWebAuthFlow` 打开飞书授权页 → 回调带 `code` → 后端 `/auth/lark/callback` 用 code 换飞书用户信息(`open_id`/`name`/`tenant_key`)→ 建 session → 返回 session token,扩展存 `chrome.storage.sync`,后续请求带 `Authorization: Bearer <session_token>`。

### 8.5 v0.5 常见坑

| 现象 | 原因 | 处理 |
|---|---|---|
| 登录后扩展仍 401 | session 过期 | 重新登录;查 `HG_SESSION_TTL` |
| 飞书授权页打不开 | 重定向 URI 没加白名单 / 与 `getRedirectURL()` 不一致 | §8.3 |
| `/auth/lark/callback` 502 | `HG_LARK_APP_ID/SECRET` 错 / code 失效 / 飞书端点版本不符 | 查后端日志;确认飞书 authen v1 端点可用,否则改 `server/lark.py` |
| `dev-login` 在生产可用 | `HG_AUTH_ALLOW_DEV=1` 没关 | 生产设 `0` |
